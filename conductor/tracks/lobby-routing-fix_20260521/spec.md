# Specification: Lobby Join/Create Routing Fix

**Track ID:** lobby-routing-fix_20260521
**Type:** Bug
**Created:** 2026-05-21
**Status:** Draft
**GitHub Issue:** #407

## Summary

Update the game lobby so players can explicitly join as either side (USA or CSA), and ensure
that creating a new game joins the creator as CSA and immediately navigates to the game view.
Currently, the lobby neither navigates after join/create nor exposes side choice.

## Context

lob-online is in M5 (turn structure, orders, game map UI). The lobby (`LobbyView.vue` +
`useLobbyStore`) was delivered in M4. The server assigns sides implicitly: creator → 'union'
(USA), joiner → 'confederate' (CSA). Issue #407 requires flipping the creator's side to CSA,
surfacing explicit side-choice buttons in the UI, and routing to `/games/:id` after a
successful action. Navigation uses Vue Router (`/games/:id` route already exists in the router).

## User Story

As a player, I want to choose which side I join and be taken directly into the game view
after joining or creating, so that I don't have to manually navigate after the lobby action.

## Problem Description

1. No post-action navigation — lobby stays on screen after join/create; player must manually
   navigate to the game.
2. Side assignment is implicit and hardcoded; creator is always 'union', joiner always 'confederate'.
3. The issue requires creator = CSA and a UI that exposes both side choices.

## Acceptance Criteria

- [ ] From the lobby, clicking "Join as USA" joins the game as 'union' and routes to `/games/:id`.
- [ ] From the lobby, clicking "Join as CSA" joins the game as 'confederate' and routes to `/games/:id`.
- [ ] Creating a new game assigns the creator as 'confederate' (CSA) and navigates to `/games/:id`.
- [ ] The join endpoint rejects a request for a side that is already occupied (409).
- [ ] The lobby UI shows two join buttons per game row ("Join as USA", "Join as CSA") instead of one.
- [ ] The lobby UI does not leave the player on the lobby page after a successful join or create.
- [ ] All existing tests pass; new tests cover the changed behavior.

## Dependencies

- `client/src/stores/lobby.js` — store with `createGame` / `joinGame` actions
- `client/src/views/LobbyView.vue` — lobby UI
- `server/src/routes/games.js` — POST /api/v1/games and POST /api/v1/games/:id/join
- Vue Router `/games/:id` route — already registered in `client/src/router/index.js`

## Out of Scope

- Side-availability indicators fetched from the API (both buttons are shown; server rejects
  invalid choices at submit time)
- Allowing the creator to choose their side (creator is always CSA per this issue)
- Any changes to the SQLite schema or `gameSqlite.js`

## Technical Notes

- The store needs `useRouter()` injected to navigate after create/join. Since Pinia stores
  run inside a Vue app context, `useRouter()` works in action closures as long as the store
  is used inside a component setup.
- `POST /api/v1/games/:id/join` must accept `{ side: 'union' | 'confederate' }` in the request
  body. Since the creator always takes 'confederate', the route can reject 'confederate' joins
  with 409 ("Side already taken") without adding a DB column.
- Side label convention: server uses 'union'/'confederate'; UI displays "USA"/"CSA".
