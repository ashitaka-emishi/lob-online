# Specification: Technical Debt Tracking System

**Track ID:** tech-debt-tracking_20260318
**Type:** Feature
**Created:** 2026-03-18
**Status:** Draft

## Summary

Build a lightweight technical debt tracking system that enforces review resolution standards (fix in place or defer with a scored issue), scores deferred items on a 1–5 logarithmic scale, and produces a per-PR technical debt report that is stored in the project and aggregated into a project-wide debt report.

## Context

lob-online is in Phase 1 scaffold with active code review cycles already happening via `/team-review` and PR-based workflows. As the codebase grows toward game logic, auth, and multiplayer phases, accumulated deferred review findings will become increasingly risky. A structured way to measure and communicate that risk is needed before the codebase grows further.

## User Story

As a lead developer or architect, I want to monitor the health of the codebase, because the amount of technical debt is an indicator of codebase health and long-term maintainability risk.

## Acceptance Criteria

- [ ] A technical debt report is produced after each PR review resolution cycle.
- [ ] The report is stored in `docs/tech-debt/reports/` with a filename that links it to the PR (e.g., `pr-{number}_{date}.md`).
- [ ] A project-wide aggregated debt report (`docs/tech-debt/report.md`) is updated after each per-PR report is produced.
- [ ] The aggregated report contains: an executive summary, a debt-over-time chart (text/markdown), a risk assessment section, and an ordered list of open debt items (most impactful to least) each with their debt score.
- [ ] A skill (`/tech-debt-report`) exists that orchestrates the resolution standard enforcement and report generation workflow, hooking in naturally after review resolution.
- [ ] Deferred items are required to have: a linked issue, a debt score (1–5), and a debt assessment summary before the workflow completes.

## Dependencies

None.

## Out of Scope

- Deciding which findings from a code review to fix vs. defer (that remains a human judgment call).
- Storing or querying debt data in anything more complex than a directory of markdown files in the project structure.
- Automated complexity analysis, static analysis tooling, or any debt measurement beyond: numerical score (1–5), written assessment, and risk description.
- Future plugin extensibility is a design consideration but no plugin infrastructure will be built in v0.

## Technical Notes

- Reports live in `docs/tech-debt/reports/` (per-PR) and `docs/tech-debt/report.md` (aggregated).
- Filename convention for per-PR reports: `pr-{number}_{YYYY-MM-DD}.md`.
- The debt score uses a 1–5 logarithmic scale: 1 = trivial (style/naming), 2 = minor (duplicated logic), 3 = moderate (missing abstraction, workaround), 4 = significant (architectural compromise, risky coupling), 5 = critical (security risk, correctness hazard, blocks future work).
- A new skill (`/tech-debt-report`) will be the primary entry point; it should be invocable after `/team-review` resolution and referenced in the SDLC workflow docs.
- The skill should guide the user through: (1) classifying each deferred finding with a score + assessment, (2) generating the per-PR report, (3) updating the aggregated report.
