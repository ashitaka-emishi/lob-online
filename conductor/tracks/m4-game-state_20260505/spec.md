# Specification: M4 — Game State Model + Initializer + Setup Phase

**Track ID:** m4-game-state_20260505
**Type:** Feature
**Created:** 2026-05-05
**Status:** Draft

## Summary

Deliver the in-memory game state model, a game state initializer that reads `scenario.json`
setup data, minimal "pick a side" session management, SQLite game index, file persistence,
and a basic lobby UI — everything needed to create, join, and inspect an initialized game
before the turn loop is built in M5.

## Context

M3 is complete: the rules engine, all game tables, and both testing tools are delivered. M4
bridges the gap between the pure-function engine and a playable game by establishing the game
state data model, the initialization logic (reading real setup positions from `scenario.json`),
and the server/client scaffolding that M5 will use to implement the turn loop.

## User Story

As a developer, I want to create and join a game with initial unit positions so that I can
begin testing the turn loop in M5.

## Acceptance Criteria

- [ ] `GET /games/:id` returns initial unit positions from `scenario.json` setup data
- [ ] Union units are constrained to their setup zones; CSA units at fixed SM §3.x positions
- [ ] Game state schema covers all fields needed by M5: unit positions, morale states, orders,
      ammo, facing, turn number, active phase, initiative state
- [ ] Game state persists across server restarts (local file store)
- [ ] Both players can create/join a game and pick a side without OAuth

## Dependencies

- `scenario.json` setup block (zone-constraint and fixed-hex formats) — exists
- M3 engine modules (`engine/hex.js`, `engine/scenario.js`) — complete
- Existing Zod schema infrastructure (`server/src/schemas/`) — complete
- `better-sqlite3` — new dependency to add
- `express-session` — new dependency for minimal session

## Out of Scope

All M5 deliverables: turn phase state machine, `dispatch(state, action)`, order acceptance,
movement enforcement, game map UI with units rendered on map, Socket.io real-time updates.

## Technical Notes

- Game state initializer at `server/src/engine/init.js`. Two setup formats:
  zone-constraint `{ setupZone: "within5Of", referenceHex }` (Union) and fixed-hex
  `{ unitId, hex }` (CSA). SM Errata corrections applied at init time. Reinforcements
  pre-queued from `scenario.json reinforcements` by arrival turn.
- SQLite `games` table only (no users table). Schema: `id`, `side_a_token`, `side_b_token`,
  `status`, `created_at`, `state_path`.
- File persistence: `data/games/{id}/state.json` — simple JSON read/write, no history log
  yet (M8).
- Session: `express-session` with in-memory store, `sideToken` cookie identifying which side
  the browser controls. Sufficient for local dev; replaced by real auth in M8.
- Zod schema for `GameState` lives in `server/src/schemas/gameState.schema.js`.
