# Specification: North Calibration — gridSpec northOffset + Cardinal Edge Labels

**Track ID:** north-calibration_20260317
**Type:** Feature
**Created:** 2026-03-17
**GitHub Issue:** #80
**Status:** Draft

## Summary

Add a 12-position `northOffset` field to `gridSpec` (covering all 6 hex edges and 6 vertices at 30° increments), wire a visual calibration picker into `CalibrationControls`, derive cardinal edge labels from the offset via a new `getEdgeLabels()` utility, and propagate those labels through `WedgeEditor`, `HexEditPanel`, and `HexMapOverlay`.

## Context

The map editor digitizes `docs/reference/sm-map.jpg` into `map.json` terrain data. South Mountain's geographic north aligns with the right vertex of each flat-top hex (northOffset = 3), but the schema has no way to declare this. Without it, any future directional game mechanic (reinforcement entry sides, wind, line-of-sight) cannot correctly translate hex-edge indices to compass directions.

## User Story

As a map editor operator, I want to set geographic north relative to the hex grid so that edge/direction labels throughout the editor display correct cardinal directions for the South Mountain map.

## Technical Notes

- `getEdgeLabels(northOffset)` — the base array for northOffset=0 (top edge = N for flat-top) is `['N','NE','SE','S','SW','NW']`. The 12 positions map as: 0=top-edge, 1=top-right-vertex, 2=right-edge, 3=right-vertex, 4=bottom-right-edge, 5=bottom-right-vertex, 6=bottom-edge, 7=bottom-left-vertex, 8=left-edge, 9=left-vertex, 10=top-left-edge, 11=top-left-vertex (flat-top hex). Rotating by northOffset steps of 30° means: shift the 6-element edge array by `Math.floor(northOffset / 2)` positions (wrapping), with direction reversal when the offset is odd (vertex position).
- `northOffset` only needs to be read by client components; the server schema change is solely for data validation.
