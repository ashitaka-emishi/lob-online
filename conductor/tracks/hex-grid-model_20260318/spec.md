# Specification: Hex Grid Model — Fix EVEN_Q Column Offset Convention

**Track ID:** hex-grid-model_20260318
**Type:** Bug
**Created:** 2026-03-18
**Status:** Draft

## Summary

The South Mountain hex grid uses the wrong column-offset parity convention throughout the
coordinate model. All code assumes ODD_Q (odd columns shifted, even columns at baseline) but
the physical SM map requires EVEN_Q (even columns shifted up relative to odd columns). This
causes hex labels, LOS calculations, and the visual overlay to be mismatched from the map.

## Context

The map editor digitises `docs/reference/sm-map.jpg` into `map.json`. The hex grid overlay
in `HexMapOverlay.vue` reads `gridSpec.evenColUp` from that file and passes it as `offset` to
the honeycomb-grid library. However, all coordinate-math helpers — `adjacentHexId`,
`getCellAndNeighbors`, `colRowToCube`, `cubeToColRow` — hardcode the ODD_Q formula regardless
of the `evenColUp` flag, and `map.json` currently stores `evenColUp: false`.

## Problem Description

**South Mountain map properties:**

- 36 rows numbered 0–35 in the model; game-coordinate origin is 01.01 at the bottom-left.
- Columns 1–63, all playable on left/right edges.
- Even-numbered columns are offset **up** relative to odd columns:
  - Top edge: even columns show only the bottom half of the hex (partially cropped).
  - Bottom edge: even columns show only the top half of the hex (row 0 partially visible).
  - Bottom edge: odd-column row-0 hexes are off-map (neither visible nor playable).
- The hex to the **NE** of 01.01 is **02.01** (same row); the hex to the **SE** is **02.00**
  (one row below, partially off-map for even columns).

**Bug symptom:** turning on the hex col.row label overlay shows that labels are mismatched
with their visual hex positions.

**Root cause:** `evenColUp` is `false` and coordinate math uses `(hcCol - (hcCol & 1))`,
which is the ODD_Q formula. It should use `(hcCol + (hcCol & 1))` (EVEN_Q).

## Technical Notes

The fix is a formula substitution throughout coordinate math:

- ODD_Q (current): `Math.floor((hcCol - (hcCol & 1)) / 2)`
- EVEN_Q (correct): `Math.floor((hcCol + (hcCol & 1)) / 2)`

Both the offset→cube and cube→offset directions must be updated consistently. The honeycomb-grid
library call in `HexMapOverlay.vue` already respects `evenColUp` dynamically, so only the
`map.json` data and the pure-math helpers need changing.
