# Specification: Auth Guard + sideToken Validation on Join Endpoint

**Track ID:** auth-join-guard_20260506
**Type:** Bug
**Created:** 2026-05-06
**Status:** Draft

## Summary

Two related auth defects on `POST /api/v1/games/:id/join` (SEC-H1 + ARCH-H2 from PR #339 team review, deferred from pre-M5 debt sprint as issue #340): the endpoint accepts any caller with no session check, and the store layer does not validate that the token it writes is a non-empty UUID string.

## Context

M4 shipped a minimal session system: the game creator gets a session (union side + sideToken), and the join endpoint assigns a second sideToken to the confederate player. There is no OAuth yet (deferred to M8). The session is cookie-based via `express-session`. `requireSide` middleware guards `GET /games/:id` by matching the caller's session `gameId` to the route param — but the join endpoint has no equivalent guard.

## Problem Description

**SEC-H1 — Missing sideToken validation in store layer:**  
`gameSqlite.joinGame(id, sideBToken)` runs an UPDATE with whatever value is passed as `sideBToken`. An empty string, null, or non-UUID value could be written to `side_b_token`. The route currently generates `sideBToken = randomUUID()` so this is a defensive gap, but the store layer should validate its own contract.

**ARCH-H2 — No session conflict guard on join route:**  
Any caller who knows a game UUID can POST to `/:id/join` and claim the confederate seat, including the union player from the same browser session. If the union player POSTs to join their own game, a new sideToken is generated, the DB row is updated (since status is `'open'`), and the caller's session is overwritten to confederate — corrupting the game. No OAuth is needed to close this: a session-match check (if the caller is already in this game, reject 409) closes the worst-case exploit.

## Acceptance Criteria

- [ ] `gameSqlite.joinGame` throws `TypeError` (or equivalent) if `sideBToken` is not a non-empty string matching UUID format
- [ ] `POST /:id/join` returns 409 if the caller's session already contains the same `gameId` as the route param
- [ ] Valid join by a session-less caller (new player) continues to work and returns 200 with confederate side
- [ ] All new paths covered by unit tests
- [ ] `npm run test` passes with no regressions

## Dependencies

- `server/src/store/gameSqlite.js` — `createStore` / `joinGame`
- `server/src/routes/games.js` — join route handler
- `server/src/auth/session.js` — `getPlayerSession`
- `server/src/routes/games.test.js` — existing route tests

## Out of Scope

- OAuth or user-account authentication (M8)
- Preventing a completely unauthenticated new browser from joining (requires M8 OAuth)
- Lobby invitation codes or access tokens
- Any changes to `GET /games/:id`, `POST /games`, or `requireSide`

## Technical Notes

- Session check in the route: call `getPlayerSession(req)` — if `player.gameId === id`, return 409 `{ error: 'Already in this game' }`. A caller with no session or a different-game session is allowed through.
- UUID regex already defined in `games.js` (`UUID_RE`) — reuse it for store-layer validation rather than introducing a new dependency.
- The store `joinGame` should throw a named error class so the route can map it to a clean HTTP status — but since this is defensive (the route already passes a valid UUID), a plain `TypeError` with a descriptive message is sufficient.
