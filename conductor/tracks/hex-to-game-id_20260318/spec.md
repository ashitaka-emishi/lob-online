## Summary

Extract the repeated `gameCol / gameRow / id` coordinate formula from three locations in
`HexMapOverlay.vue` into a shared `hexToGameId(hex, gridRows)` helper in `hexGeometry.js`,
and add unit tests covering both the helper and the click handler paths that previously had
no coverage.

## Context

`HexMapOverlay.vue` converts honeycomb-grid `{col, row}` offsets to the game's `"CC.RR"` hex
ID format in three independent places:

| Location            | Lines   |
| ------------------- | ------- |
| `gridData` computed | 170–172 |
| `onSvgClick`        | 288–291 |
| `onSvgContextMenu`  | 357–360 |

All three blocks are identical:

```js
const gameCol = hex.col + 1;
const gameRow = gridRows - hex.row;
const id = `${String(gameCol).padStart(2, '0')}.${String(gameRow).padStart(2, '0')}`;
```

This triplication was the **direct structural cause** of the off-by-one bug fixed in PR #117:
a single erroneous term silently propagated to all three sites. The existing regression test
(added in PR #117) covers only the `gridData` label-rendering path. The two click handlers
emit hex IDs with no test coverage — a re-introduced parity bug there would go undetected.

## Technical Notes

Signature: `hexToGameId(hex, gridRows)` where `hex` is any object with `.col` and `.row`
(honeycomb-grid offset coordinates) and `gridRows` is the total number of rows in the grid.
Returns the `"CC.RR"` string. Keep it a pure function with no side effects.

The `gridRows` value used in `onSvgClick` and `onSvgContextMenu` must match the same
`calibration.rows > 0 ? calibration.rows : 35` guard already present at `gridData` line 145.
After refactor, all three sites read from the same local variable or call the same helper —
no re-deriving of `gridRows` inline.
