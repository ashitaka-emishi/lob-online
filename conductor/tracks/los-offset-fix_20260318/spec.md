# Specification: LOS Display Off-by-One — Hex Column Offset

**Track ID:** los-offset-fix_20260318
**Type:** Bug
**Created:** 2026-03-18
**Status:** Draft
**GitHub Issue:** #114

## Summary

The LOS highlight in the map editor lands one game-row off from the selected hex on odd-numbered game columns (1, 3, 5, 7…). The `gameRow` formula in `HexMapOverlay.vue` applies the column-offset correction to the wrong parity, causing displayed hex IDs to be one row lower than the actual hex that was clicked or entered.

## Context

The map editor renders a flat-top, ODD_Q offset hex grid via honeycomb-grid (`offset: -1`, `evenColUp: false`). When converting from honeycomb's internal `hex.col`/`hex.row` indices to the game's `CC.RR` identifier, the formula must compensate for the vertical stagger of alternating columns. The current formula applies the correction to even game columns but the bug evidence shows odd game columns are the affected parity.

In addition, two pieces of pre-existing uncommitted work will be bundled into this track:

- **Settings flag**: `CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1` added to `.claude/settings.json`
- **Map data**: ~100 hexes digitized for the north-west sector of the South Mountain map (cols 01–21, rows 01–10), including terrain types and elevation levels, committed to `data/scenarios/south-mountain/map.json`

## Problem Description

**Steps to reproduce:**

1. Open the map editor (`/tools/map-editor`)
2. Open the LOS Test panel
3. Pick a hex in an odd-numbered column (e.g. 7, 9, 11) by clicking on the map
4. Observe that the hex ID shown is one row lower than the hex that was clicked

## Root Cause Hypothesis

The formula `gridRows - hex.row - (gameCol % 2 === 0 ? 1 : 0)` subtracts 1 for even game columns (= odd `hex.col`, the shifted columns under `offset: -1`). With `offset: -1` in honeycomb-grid, odd `hex.col` values are shifted UP (smaller `hex.row`), so the `gridRows - hex.row` term over-shoots by 1 for that parity, and the subtraction corrects it. The observed bug on ODD game columns (= even `hex.col`, unshifted) suggests either the parity check is inverted, or the `evenColUp` → `offset` mapping applies a different direction than assumed. The fix requires verifying the exact parity with a failing test before modifying the formula.

- All three occurrences of the formula must be fixed identically.

- After the fix, a round-trip test (`colRow → cube → colRow`) for both parities is a reliable regression guard.
