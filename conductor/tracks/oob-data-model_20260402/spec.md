# Specification: OOB Data Model — Leader Ratings, Supply Units, and Succession Variants

**Track ID:** oob-data-model_20260402
**Type:** Feature
**Created:** 2026-04-02
**Status:** Draft

## Summary

Three data-model gaps in the OOB/leaders data files and their Zod schemas, all uncovered
during OOB editor implementation. Addresses GitHub issues #233, #234, and #235.

## Context

The OOB editor revealed that the current data model has three structural problems that must
be fixed before the editor UI issues (##228–232) can be addressed cleanly:

1. `initiativeRating` on leaders is the wrong concept — LoB leaders have two ratings
   (Command Value and Morale Value) printed on their counters, not a single initiative number.
2. 1 Corps and 9 Corps have no supply unit entries in `oob.json`, even though the army-level
   supply train exists and the editor tree needs to display corps-level supply.
3. Promoted leaders (e.g., Walker Promoted) and scenario-specific command variants (Burnside
   as Wing / 9 Corps / AotP) have different stats than their base leader records and must not
   pollute `leaders.json`. A new `succession.json` file will hold them.

## Acceptance Criteria

- [ ] All leader records in `leaders.json` have `commandValue` and `moraleValue` (integer | null);

## Technical Notes

### succession.json shape

```jsonc
{
  "union": [
    {
      "id": "walker-promoted",
      "name": "Col Joseph Walker (Promoted)",
      "baseLeaderId": "walker",
      "commandLevel": "brigade",
      "commandsId": null,
      "commandValue": 0,
      "moraleValue": 1,
      "counterRef": { "front": null, "frontConfidence": null },
    },
  ],
  "confederate": [],
}
```

### commandValue / moraleValue

- Both integer | null. Null = not yet sourced from counter scan.
- Where a leader has a single printed value both fields carry the same number.
- Field order in schema and JSON: `commandValue` before `moraleValue`.

### Supply unit shape (same as hq)

```jsonc
"supply": {
  "id": "1c-supply",
  "name": "1 Corps Supply",
  "counterRef": { "front": null, "frontConfidence": null }
}
```

### SuccessionList resolution

`successionIds` arrays remain plain ID strings. The resolver checks `leaders` first,
then `succession`; the first match wins.
