## Technical Notes

### #237 fix

`processUSACavDiv` currently inlines Pleasonton variant logic (lines 162–172) that
duplicates what `withLeader` already does, and attaches Farnsworth leader without a variant
lookup. Replace both with `withLeader` calls:

```js
const processedFcav = withLeader(
  { ...(fcavBde ?? { id: 'fcav' }), name: 'F/Cav', batteries: cavArty?.batteries ?? [] },
  leadersMap,
  variantsMap
);
// Pleasonton:
return withLeader(
  { id: cd.id, name: 'Cavalry Division', _hq: hqNode, brigades: [processedFcav] },
  leadersMap,
  variantsMap
);
```

### #245 factory

Factory signature:

```js
export function createEditorRoute({ schema, filePath, filePrefix, backupDir, maxBackups })
```

Returns an Express Router with GET `/data` and PUT `/data` handlers. Each existing route
file constructs its constants and calls `createEditorRoute(...)`, then `export default`s
the returned router. Rate limiter stays in each caller file (different services may want
different limits).

### #247 tests

Model after `useMapPersistence.test.js`. Mock `fetch` and `localStorage`. Key scenarios:

- L1 path: server returns valid succession → `succession.value` populated
- L1 path: succession endpoint non-OK → `succession.value` stays null (graceful degradation)
- L2 path: localStorage has valid succession but missing OOB keys → falls through to L3
- L2 path: localStorage has OOB+leaders but no succession key → loads without succession
- `_executePush` with null succession → only 2 fetches, succession endpoint not called
- `_executePush` success → `localStorage.removeItem` called for succession key
- `pullFromServer` failure → `syncError` non-null, `isSyncing` false
- `pullFromServer` succession endpoint non-OK → loads oob+leaders only, no crash
