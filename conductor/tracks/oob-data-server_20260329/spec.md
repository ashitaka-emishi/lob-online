# Specification: OOB Data Model Completion and Server Routes

**Track ID:** oob-data-server_20260329
**Type:** Feature
**Created:** 2026-03-29
**Status:** Draft

## Summary

Extend `oob.json` and its Zod schema to add `successionIds` and `counterRef` to command-level
nodes (brigade, division, corps), then create Express routes for `/api/oob/data`,
`/api/leaders/data`, and `/api/counters/list` + upload — the server-side foundation the OOB
editor reads and writes against.

## Context

The OOB editor (PR #208) reads `oob.json` and `leaders.json` via client-side fallbacks today.
The store's `loadData()` already calls `/api/oob/data` and `/api/leaders/data` as its first
choice — those routes just don't exist yet. Succession and counterRef on command nodes are
required for the succession list component (#195) and leader detail panel (#194), the two
remaining OOB editor UI features.

Note: Issue #190 (leaders counterRef) is already complete — all 48 leader records have
`counterRef` in schema and data from PR #208. This track closes #189 and #191 only.

## User Story

As an OOB editor user, I want brigade/division/corps nodes to carry `successionIds` and
`counterRef` so that I can assign counter images and succession order at every level of
command, and have those edits round-trip through the server reliably.

## Acceptance Criteria

- [ ] `Brigade`, `Division`, `Corps`, `CavalryDivision`, and `IndependentBrigade` Zod schemas
      include `counterRef: CounterRef.optional()` and
      `successionIds: z.array(z.string()).optional().default([])`
- [ ] Every brigade, division, corps, and cavalry division node in `oob.json` has
      `"successionIds": []` and `"counterRef": null`
- [ ] `GET /api/oob/data` returns `oob.json`; `PUT /api/oob/data` validates, backs up, writes
- [ ] `GET /api/leaders/data` returns `leaders.json`; `PUT /api/leaders/data` validates, backs up, writes
- [ ] `GET /api/counters/list` returns an array of filenames from `client/public/counters/`
- [ ] `POST /api/counters/upload` saves a file to `client/public/counters/` (multipart/form-data)
- [ ] All six new route endpoints have test coverage
- [ ] `npm run test` passes with ≥70% coverage

## Dependencies

- `OOBSchema` and `LeadersSchema` in `server/src/schemas/` (already exist — extending only)
- `mapEditor.js` route pattern (rate limiter, backup rotation, `_savedAt` timestamp)
- `multer` (for file upload in counters route) — not yet installed; add to `server/package.json`

## Out of Scope

- UI components for succession list (#195) or leader detail panel (#194)
- Auto-detection of counter images
- Updating `countersManifest.js` (handled separately by auto-detect script)
- Issue #190 (leaders counterRef) — already complete

## Technical Notes

- Follow `mapEditor.js` exactly: 60 req/min rate limiter, max 20 backups, `_savedAt` timestamp on write
- `counters.js` upload endpoint: validate extension (`.jpg`, `.png` only), reject files >500KB
- `oob.json` data patch: ~45 command nodes need `"successionIds": []` and `"counterRef": null`
- `successionIds` goes on Brigade, Division, Corps, CavalryDivision, IndependentBrigade only
  (not on regiment/battery leaf nodes, which already have `counterRef`)
