## Context

The map editor dev tool works with hex arrays of up to 2000+ entries. Several hot paths in `MapEditorView.vue` and `HexEditPanel.vue` use repeated O(n) `.find()`/`.findIndex()` scans and contain undocumented intentional bypasses of the normal `onHexUpdate` mutation path. These were surfaced in the PR #91 team review and deferred to this track.

## Problem Description

1. **Medium — Shared hex index (P1, P2):** `selectedHex`, `onHexClick`, `onEdgeClick`, `onHexUpdate`, and `onDeriveWedges` all do O(n) `.find()`/`.findIndex()` scans on every interaction. `onDeriveWedges` builds a local `Map` per call and discards it.
2. **Low — Double-write in autoDerive (L1):** `autoDerive` calls `emitUpdate()` (writes the source hex via `hex-update`) and also includes the source hex in the `derive-wedges` payload, causing the same hex to be written twice per click.
3. **Low — onDeriveWedges bypass undocumented (L2):** `onDeriveWedges` inlines the mutation logic that normally flows through `onHexUpdate`, with no comment explaining why.
4. **Low — computedEngineExport recalculates on every edit (L4):** The deep-walk computed is only consumed by the export overlay but runs reactively on every hex change.
5. **Low — Slope tie-break undocumented (L7):** `elevationDerive.js:41` picks the first minimum-wedge index (N before NE, etc.) via `indexOf` with no comment marking this as an intentional game-rule decision.

## Technical Notes
