# Specification: Refactor Map Elevation Model

**Track ID:** elevation-model-refactor_20260317
**Type:** Refactor
**Created:** 2026-03-17
**Status:** Draft

## Summary

Refactor the map elevation model to use zero-based integer level indices (instead of
absolute feet), add `baseElevation` and `elevationLevels` to `elevationSystem`, convert
`wedgeElevations` to integer offsets relative to hex center, replace `WedgeEditor` text
inputs with click interactions, and add an "Auto-derive edges & slope" button.

## Context

Phase 1 scaffold is complete. The map editor dev tool digitizes the South Mountain map into
`map.json`. The current model stores per-hex elevation as absolute feet; this refactor
normalizes it to integer contour levels consistent with LOB rules (LOB_RULES 4.0,
SM_RULES 1.1–1.2).

## Problem Description

- `map.json` stores `elevation` in feet (e.g., 550) rather than integer level index (e.g., 1)
- `elevationSystem` lacks `baseElevation` and `elevationLevels` fields
- `wedgeElevations` values are not yet semantically defined as integer offsets from center
- `WedgeEditor.vue` uses text inputs instead of click interactions
- No auto-derive tooling exists to compute hexside edge types and slope from wedge values

## Acceptance Criteria

- [ ] `elevationSystem` in `map.json` and `MapSchema` includes `baseElevation: 500` and
      `elevationLevels: 22`
- [ ] Per-hex `elevation` is a non-negative integer (`z.number().int().min(0)`)
- [ ] `wedgeElevations` values are integers (`z.number().int()`), positive = hexside higher
      than center
- [ ] `map.json` migrated: hex elevations converted `(feet − 500) / 50`, metadata added,
      `_note` added
- [ ] `WedgeEditor.vue` uses left-click (−1) / right-click (+1), suppresses context menu,
      displays current integer value on each triangle

- [ ] All new Vitest tests passing (schema validation, auto-derive logic, neighbor propagation)

## Dependencies

None

## Out of Scope

- LoS engine (elevation/wedge values do not affect LoS per SM_RULES 1.2, LOB_RULES 1.7c)
- "elevation" thin contour edge type (manual only, not auto-derived)
- Non-elevation edge features (road, stream, stoneWall) — not overwritten by auto-derive

## Technical Notes

- Base elevation: 500 ft; contour interval: 50 ft; 22 distinct levels
- Delta rules: 0 → remove slope edge, 1 → `slope`, 2 → `extremeSlope`, ≥3 → `verticalSlope`
- Neighbor lookup via `adjacentHexId` in `client/src/utils/hexGeometry.js`
- Opposite hexside: `(i + 3) % 6`
- Rules: SM_RULES 1.1, 1.2; LOB_RULES 1.7c, 4.0; rules-lawyer consulted 2026-03-17
