# Specification: Extract ElevationSystemControls Component

**Created:** 2026-03-18

## Summary

Extract the elevation system UI (baseElevation + elevationLevels inputs) out of
`CalibrationControls.vue` into a dedicated `ElevationSystemControls.vue` component, so that
each component owns exactly one data domain.

## Context

`CalibrationControls` currently owns two unrelated data domains — `gridSpec` (grid geometry
calibration) and `elevationSystem` (terrain elevation parameters) — with different persistence
semantics and lock behavior. As additional `elevationSystem` fields gain UI surface (e.g.
`contourInterval`, `unit`), the component will accumulate further misplaced responsibility.
This refactor restores single-responsibility before that growth begins.

## Acceptance Criteria

- [ ] A new `ElevationSystemControls.vue` component exists with its own props (`elevationSystem`,
      `locked`) and emits (`elevation-system-change`).
- [ ] `ElevationSystemControls` has a dedicated `ElevationSystemControls.test.js` with
      `data-testid`-based selectors (not index-based) covering: value rendering, change emission,
      and disabled state when `locked` or `elevationSystem` is absent.
- [ ] `CalibrationControls.vue` delegates to `<ElevationSystemControls>` as a child component;
      it no longer contains elevation input markup directly.
- [ ] `CalibrationControls.vue` still passes `elevationSystem` and `locked` down as props and
      re-emits `elevation-system-change` upward — `MapEditorView` is unchanged.
- [ ] All existing `CalibrationControls.test.js` elevation-section tests continue to pass
      (they exercise CalibrationControls end-to-end, which now delegates internally).
- [ ] No visible behavior change for the user. CI gates (lint, format, tests) pass.

## Dependencies

- `CalibrationControls.vue` and its existing test file
- `MapEditorView.vue` (caller — must remain unchanged after refactor)

## Out of Scope

- Replacing index-based input selectors in `CalibrationControls.test.js` with `data-testid`
  (tracked in #101 — separate concern)
- Replacing mutable shared fixtures in test files (#102)
- Any new elevation fields (`contourInterval`, `unit`)
- Moving `ElevationSystemControls` into its own accordion panel in `MapEditorView`

## Technical Notes

`CalibrationControls` renders inside a calibration accordion panel in `MapEditorView`. To keep
the MapEditorView change surface at zero, `CalibrationControls` should render
`ElevationSystemControls` as a child, passing `elevationSystem` + `locked` props and listening
for `elevation-system-change` to re-emit. The elevation section `<div class="elevation-system-label">`
and its two `<label>` inputs move wholesale into the new component.
