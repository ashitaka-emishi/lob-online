## Summary

Move `slopeArrowLine` and `slopeArrowLabel` per-cell computation out of template-layer
functions and into the `gridData` computed property in `HexMapOverlay.vue`, eliminating
redundant recalculation and aligning slope arrow data with the existing data-preparation layer.

## Context

`HexMapOverlay.vue` assembles per-cell display data (`fill`, `fillOpacity`, `isVP`, etc.)
inside the `gridData` computed property. Slope arrow geometry is the exception: two
template-layer functions (`slopeArrowLine(cell)` and `slopeArrowLabel(cell)`) are called
reactively inside a `v-for` over up to 2240 cells. `slopeArrowLine` is invoked up to 3Ă— per
cell per render (one `v-if` guard and two attribute bindings on line 437â€“447), and
`slopeArrowLabel` once more â€” a small but unnecessary per-render cost and an architectural
inconsistency flagged by both the performance and architecture reviewers on PR #86.

## Technical Notes

Inside the `gridData` forEach loop (after `cx`/`cy` are established):

```js
const SLOPE_DIRS = ['N', 'NE', 'SE', 'S', 'SW', 'NW'];
const northOffset = props.calibration.northOffset ?? 0;

// inside forEach:
const slopeDir = cell.slope !== null && cell.slope !== undefined
  ? SLOPE_DIRS[cell.slope] : null;
const slopeArrowLine = slopeDir
  ? { x1: cx, y1: cy, ...edgeMidpoint(corners, slopeDir) } : null;
const slopeArrowLabel = slopeDir
  ? (getEdgeLabels(northOffset)[cell.slope] ?? null) : null;
cells.push({ ..., slopeArrowLine, slopeArrowLabel });
```

Note: `edgeMidpoint` returns `{ x, y }` â€” spread gives `x2`/`y2` via destructuring alias, or

`x2`/`y2` from the old function's return; keep the same key names for a minimal diff).

The standalone `slopeArrowLine()` and `slopeArrowLabel()` functions can then be deleted.
