# Specification: Game Map Viewport Fix тАФ Scrolling, Grid Config, Counter Size

**Track ID:** game-view-viewport_20260521
**Type:** Bug
**Created:** 2026-05-21
**Status:** Draft
**GitHub Issue:** #406

## Summary

The M5 game map route (`/games/:id`) allows document-level scrolling instead of confining all
scrolling to the map viewport. The hex grid renders with stale localStorage calibration instead
of the scenario's saved `gridSpec`. Unit counters are not sized to the expected 80% of their hex.

## Context

The `GameView` component was delivered in M5 (PR #400) with a CSS skeleton that constrains the
layout to `100vh`. However, the `<body>` in `App.vue` lacks `overflow: hidden`, allowing the
document to scroll when map content overflows. The `useCalibration()` composable reads from
`localStorage` (shared with the Map Editor tool), but `GameView` never applies the `gridSpec`
from the scenario's map data тАФ so the hex overlay is positioned using whatever the map editor
last saved, not the canonical grid configuration for the scenario. Counter sizing is a
consequence of the same miscalibration.

The `gridSpec` already lives in `map.json` (the canonical scenario data file) and is served
today only via the dev-tool endpoint (`/api/tools/map-test/data`, gated by
`MAP_EDITOR_ENABLED=true`). That path is not appropriate for production use. The scenario's
map configuration should be exposed through the games API so `GameView` can load it from a
stable, auth-appropriate route.

## Technical Notes

**New API endpoint:** `GET /api/v1/games/:id/map-config` returns `{ gridSpec }` loaded from
`map.json` at startup. No `requireSide` guard тАФ `gridSpec` is static scenario data, not
player-specific state. Must be declared before the `/:id` catch-all in the games router to
avoid being swallowed by `router.param`.

`GameView` should remove its dependency on `useCalibration()` (localStorage) and on the
dev-tool fetch. Instead, it reads `gameStore.gridSpec` (populated by `loadGame`) and computes
calibration locally using exported helpers from `useCalibration.js`.

The `min-width: 0` CSS rule is required on `.map-area` because flex children default to
`min-width: auto`, which prevents them from shrinking below content size. Without this fix,
`overflow: auto` on `.map-area` has no effect тАФ the flex item expands instead of scrolling.
