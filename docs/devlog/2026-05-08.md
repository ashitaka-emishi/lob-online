# 2026-05-08

## 02:15 — Plan Wrap: M5 Schema Prerequisites (PR #359)

PR #359 introduced the `UnitOrderState` Zod schema, upgraded `UnitStateSchema.orders` from a raw
nullable string to a structured object, and added the `isDetached` boolean field. The track
(`m5-schema-prereqs_20260507`) originated as a stash on the design branch — the schema work was
drafted during the M5 design phase but deliberately not committed there so that implementation
evidence would stay off the design PR. Applying the stash cleanly to the implementation branch was
straightforward; no conflicts arose because no intervening schema edits had landed on master.

The most significant design decision in the schema was the five-way cross-field refinement on
`UnitOrderState`. A simpler design would have used a discriminated union (one Zod shape per
status), but the project is plain JavaScript with no TypeScript — the discriminated union pattern
buys less here without static type narrowing at call sites. The refinement approach keeps a single
flat object shape while still rejecting invalid combinations at runtime. The tradeoff is verbosity:
five `.refine()` calls instead of one `z.discriminatedUnion()`. The team review caught that the
initial implementation was missing three of the five required refinements, which were added in
place before merge.

The `isDetached` field is a forward-looking prerequisite for the SM brigade detachment rules. At
this stage it defaults to `false` for all units and is never set to `true` by the initializer —
the scenario data model has no way to express that a brigade starts the game already detached, and
`processSetupSide()` has no path to handle it. That gap is filed as issue #361. Similarly, the
`initGameState()` function reads `orderType` from reinforcement group entries in `scenario.json`
but silently ignores it (#360). Both of these are score-4 deferred findings that must be resolved
before the M5 order-delivery pipeline can be wired up correctly.

The team review also surfaced a semantic ambiguity in the schema: `orders: null` (non-order-holding
unit, inherits from parent division) and `orders: { type: null, status: 'none', deliveryTurnDue:
null }` (order-holding unit, no current order issued) are both valid but mean different things. No
code path currently produces the second form — it was introduced as a forward-looking state for
order-holding divisions that have exhausted their current order — but future query code will need
to guard against both. This was deferred as issue #364. The 'none' status may ultimately be
consolidated away if the design settles on `orders: null` as the universal no-order representation
for divisions as well as non-order-holders.

Thirteen findings were deferred across the review cycle (issues #360–#372), adding +29 to the
gross debt register and bringing net open debt to 46 — the highest since PR #236. The two
highest-priority items (#360 and #361) directly block correct M5 gameplay and should be addressed
as a pre-cursor track before order-pipeline implementation begins. The remaining eleven items are
documentation, schema ergonomics, and performance flags that can be batched into a future debt
sprint.
