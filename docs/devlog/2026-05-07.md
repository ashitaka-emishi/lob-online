# 2026-05-07

## 18:12 — PR: M5 Schema Prerequisites — UnitOrderState + isDetached

This PR delivers the game-state schema prerequisites that M5's turn-structure and orders work
depends on. Before this change, `UnitStateSchema.orders` held a raw nullable string (`'attack'`
or `'move'`). That was sufficient for M4's initializer, but the M5 order pipeline needs to
distinguish between three states: no order issued (none), order in transit awaiting delivery
(delay), and order accepted and active (accepted). Forcing that distinction into a single enum
field would have required string smuggling — encoding status into the value — so the right move
was to introduce a proper object schema.

`UnitOrderState` is a Zod object with three fields: `type` (reusing the existing `OrderType`
enum), `status` (`'none' | 'delay' | 'accepted'`), and `deliveryTurnDue` (a positive integer
or null). Two cross-field refinements enforce the invariants: delay status requires
`deliveryTurnDue` to be set, and none status requires `type` to be null. These are the two
combinations that, if allowed through, would create silent bugs in the M5 orders phase engine —
a delay with no scheduled delivery turn would cause orders to be lost, and a none-status unit
carrying a non-null type would confuse initiative roll logic.

The other schema addition is `isDetached: z.boolean()` on `UnitStateSchema`. This is needed
for SM detachment rules, where a brigade can be operationally separated from its parent
division and must hold orders independently. At init, all units receive `isDetached: false`.
The field will be flipped to `true` by a future dispatch path when a brigade is detached. The
null-orders convention is also clarified: `orders: null` on a unit means it is a non-order-
holding brigade that inherits its effective order from its parent division at query time, not
that it has no order. This distinction is important for the orders-resolution logic that will
be implemented in M5.

The implementation reused a stash (`stash@{0}` on `design/m5-turn-structure-orders-game-map-ui`)
that had been authored and parked during the M5 design session. The stash covered all four
primary files cleanly. One additional fix was needed: the `gameFile.test.js` round-trip fixture
was missing `isDetached`, which caused the only test failure on first run. That was a one-line
addition. All 2022 tests pass; `quality:strict` is clean.

What was deferred: the `'stopped'` status variant mentioned in the M5 design doc (§5) is not
included here. That status is only needed by the fluke-stoppage and attack-recovery logic, which
are implemented later in M5. Adding it now would expand the schema's validation surface without
a consumer to test it against. It will be added as part of the specific M5 task that implements
those rules.
