# 2026-03-31

## 07:25 — PR: OOB Editor Final UI — Succession List and Leader Detail Panel (#194, #195)

This PR closes out the OOB editor's final two UI gaps: a succession-list editor wired into
brigade, division, and corps panels, and a fully-editable leader detail panel with counter
image upload support. Both features were driven by issues #194 and #195, which identified
that the command hierarchy editor was incomplete — units could be viewed but their succession
chains could not be managed, and leader nodes had no editable fields at all.

The `SuccessionList` component is a self-contained Vue widget that renders an ordered list of
leader IDs with up/down reordering and inline removal, plus a type-ahead search field that
filters from the full leader pool (filtered to the correct side). The component calls
`store.updateSuccession(unitPath, newIds)` to persist changes, keeping all mutation logic in
the Pinia store. Suggestions exclude already-assigned leaders to prevent duplicates. The army
field in `leaders.json` is a plain string rather than an array, so the component guards against
non-array inputs when building the candidate pool.

The `CounterImageWidget` was extended with a `mode='leader'` prop. In leader mode, two
additional promoted counter slots (front and back) appear below the standard slots, each with
its own file-browse button and upload path. Uploads POST to `/api/tools/counters/upload` and
call `store.updateCounterRef` with the `promotedFront` or `promotedBack` key. The active face
is tracked via a component ref (`activePromoFace`) set at browse time, before the file dialog
opens, so the upload handler knows which slot to update.

Leader nodes in `OobDetailPanel` were previously gated behind an `isEditable` check that
excluded them, showing only a "not editable" notice. The `isLeader` computed property was added
and folded into `isEditable`, unlocking a dedicated leader field section with inputs for name,
rank, commandLevel (select), commandsId, initiativeRating, and specialRules (textarea), plus
the counter widget in leader mode. The `nodePath` for leader nodes carries the `leaders.`
prefix (e.g. `leaders.union.corps.0`), so `store.updateField` routes correctly to
`leaders.value` rather than `oob.value` without any changes to the component.

Deferred: `selectNode` in `useOobStore` does not yet search `leaders.value` when the OOB path
lookup returns null, so clicking a leader node in the tree will not populate `nodePath`
correctly at runtime. That wiring is left for the next session once the full tree navigation
is revisited. All acceptance criteria for the UI rendering and store interaction paths are
covered by the test suite (1126 tests passing).

## 13:55 — PR: Technical Debt Cleanup — All Open Items

This PR clears all 13 open technical debt items accumulated since the start of the OOB
editor work, bringing the net debt score from 19 to 0. The work was organized into four
phases, each independently verifiable and grouped by risk: test-only additions first, then
server hardening, schema strengthening, and finally the highest-risk client architecture
refactor. The goal was to enter Phase 2 (rules engine, auth, multiplayer) on a clean
foundation.

Phase 1 added three missing test cases covering existing guard paths with no production
changes: a test confirming `commit()` in `CounterImageWidget` is a no-op when `nodePath` is
null (#213), a test confirming arrow-key cycling is suppressed when focus is inside an INPUT,
SELECT, or TEXTAREA element (#212), and two tests confirming `pullFromServer` in
`useOobStore` correctly sets `syncError` and resets `isSyncing` on both network failure and
non-ok HTTP response (#214). The `#212` test required dispatching the keyboard event from the
input element itself (rather than `window`) so that `e.target.tagName` would be set correctly
— dispatching from `window` leaves `e.target` as `window`, bypassing the guard.

Phase 2 hardened three server routes. The multer middleware in `counters.js` was refactored
from an inline callback to route-level middleware with a separate error-handler middleware
(`router.use((err, _req, res, _next) => {...})`), which is the documented Express pattern for
multer errors (#222). `oobEditor.js` and `leadersEditor.js` were migrated from synchronous
`readFileSync`/`writeFileSync` to `fs/promises` async equivalents, making all route handlers
`async` and updating test mocks from sync `vi.mock('fs')` to `vi.mock('fs/promises')` with
`mockResolvedValue`/`mockRejectedValueOnce` (#220). `useOobStore.js` gained `'toString'` and
`'valueOf'` in its `FORBIDDEN_PATH_KEYS` set, closing a gap in the prototype-pollution guard
that the original three entries (`__proto__`, `constructor`, `prototype`) left open (#216).

Phase 3 strengthened the two Zod schemas. `_savedAt: z.number().optional()` was added to
both `OOBSchema` and `LeadersSchema` (the server stamps this on every write; it is absent in
the committed JSON files, so `.optional()` was required), then `.strict()` was applied to the
top-level objects (#221). The plan originally said `z.string().optional()` but `Date.now()`
returns a number, so `z.number()` was used. Nested schemas (unit, brigade, division, etc.)
were left non-strict to avoid breakage from `_errata`/`_note` fields scattered through the
data. Both real data files pass the new `.strict()` parse.

Phase 4 was the highest-risk phase. The main item was extracting a `useOobPersistence`
composable from `useOobStore.js`, mirroring the `useMapPersistence` pattern (#207). The
composable accepts `{ oob, leaders, dirty }` refs and owns all sync state (`isSyncing`,
`syncError`, `showPushConfirm`, `showPullConfirm`) and all persistence functions (`loadData`,
`_executePush`, `pullFromServer`, `requestPush/confirmPush/cancelPush`,
`requestPull/confirmPull/cancelPull`, `_saveToStorage`, `_loadFromStorage`, `_scheduleSave`).
The store retains domain mutations and exposes `persistence.scheduleSave()` from `updateField`.
The store's public API is unchanged — all 35 existing store tests pass without modification.
The remaining items were lighter: `OobTreeNode.vue` was converted from the dual
`<script>`+`<script setup>` pattern to a single `<script setup>` with `defineOptions({ name:
'OobTreeNode' })` (#215); the Vue Router was updated to lazy-load `MapEditorView` and
`ScenarioEditorView` for bundle-size consistency with the already-lazy OOB editor route
(#206); the bundled JSON fallbacks in `useOobPersistence.js` were converted to dynamic
`import()` inside the L3 fallback branch so they are not included in the initial chunk (#205);
and decision comments were added to `OobTreeNode.vue` and `oobTreeTransform.js` documenting
that the per-node watcher count and linear artillery scan are acceptable at South Mountain
scale (#204, #201).
