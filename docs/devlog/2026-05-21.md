## 0254 — PR: Engine & Utility Performance Sprint (#324 #295 #294 #201)

This PR closes four score-1/2 tech debt items that were accumulating in the hex engine and OOB
utility — all pure algorithmic optimizations with no behavioral changes. The motivation was to
enter M6 and UI design with as clean a debt register as possible rather than carrying forward
known hot-path inefficiencies.

Phase 1 tackled the three tightly coupled server-side items in `hex.js` and `movement.js`.
`parseHexId` and `formatHexId` are called on virtually every engine operation, so adding
module-level `Map` caches (`_parseCache`, `_formatCache`) eliminates redundant string parsing
on repeated calls — the cache key for `formatHexId` is `"${col},${row}"`. Dijkstra early
termination was added as an optional fifth parameter (`targetHex = null`): the first time a hex
is popped from the min-heap it has its optimal cost, so breaking immediately on target pop is
provably correct and avoids exploring the rest of the reachable frontier for point-to-point
queries. `movementPath()` passes `endHexId`; `movementRange()` passes nothing (full exploration
unchanged). The `hexEntryCost` → `hexEntryCostBreakdown` passthrough for `noEffectSet` lets
callers pre-build the Set once across a batch of cost evaluations rather than allocating a fresh
`new Set()` on every call.

Phase 2 addressed the OOB utility item in `distributeCorpsArtillery`. The function matches
artillery groups to divisions and brigades by key pattern. Legacy division matches and all brigade
matches are exact-key lookups — wrapping `artyEntries` in a `Map` up front converts these from
O(n) `find()` scans to O(1) `Map.get()`. The `endsWith` division-match pattern (new-style keys
that end with `-{div.id}`) cannot use a Map for suffix matching, so it remains a linear scan —
this is documented in the comment above the function. All 20 existing `oobTreeTransform.test.js`
tests passed without modification; the transform output is identical.

The main non-obvious constraint: the `noEffectSet` passthrough required threading the parameter
through both `hexEntryCost` and `hexEntryCostBreakdown` in `movement.js`. When null (the default),
the existing `new Set()` fallback inside `hexEntryCostBreakdown` fires as before — callers that
don't pre-build a Set see no change in behavior. A minor lint issue surfaced during `quality:strict`:
the Dijkstra early-termination test's `costFn` callback had unused `from`/`to`/`dir` parameters
that needed `_` prefixing per the project's ESLint config.

Deferred to the separate `vue-oob-debt` track: #204 (OobTreeNode watcher efficiency) and #205
(useOobStore lazy-loading). Those items touch Vue component reactivity and Pinia store chunk
strategy — higher-risk than these pure-JS optimizations and warranting their own focused sprint.

## 1557 — PR: Debt Register Housekeeping + Conductor State Cleanup (#346 #349 #350)

A short housekeeping PR clearing out loose ends before M5 UI work begins. Three tech-debt
issues (#346, #349, #350) had been filed during the May auth/join review but never scored or
registered in the debt report. Two stale conductor track statuses and one orphaned GitHub issue
also needed correction. The goal was to enter the M5 game map UI milestone with an accurate
register, correct project state, and no unresolved triaging questions.

The only production code change is a two-line comment added to the `POST /:id/join` handler
in `server/src/routes/games.js`. Issue #349 raised the question of whether a player already
in game A is allowed to silently switch to game B by joining it — the existing guard only
blocks same-game re-join (`session.gameId === id`), leaving cross-game switching unrestricted.
The decision was to document the policy explicitly rather than enforce a broader block. At this
stage, with no persistent user accounts, "your games" is a per-session concept only; the
tradeoff between ergonomics and strictness favors the looser policy until M8 OAuth work
introduces real identity. The comment makes the intent clear to future readers and closes #349.

Issues #346 (test consolidation across table-test panels, editor routes, and compass utils)
and #350 (rate limiting on games routes) were registered in `docs/tech-debt/report.md` as
score-2 items — each genuinely deferred but previously invisible to the register. Adding them
raised the net open score from 12 to 16, which crosses the "Elevated" threshold by the
register's scale. The prose in the Risk Assessment was updated to flag this as a threshold
artifact rather than new architectural risk: the items are both well-understood, non-blocking,
and have clear natural resolution paths (#346 in a test-hygiene sprint, #350 alongside M8

The issue audit found one stale open issue: #347 ("add agentic quality rails to Conductor

## 1828 — PR: M5 Game Map UI — Unit Layer, Stats Panel, Game Store

This PR delivers the first playable game view for lob-online: a live hex map with unit counter
images, a sidebar stats panel, and a Pinia store that loads game state from the server. Four
components were built TDD across four phases: `useGameStore`, `UnitCounterLayer`, `UnitStatsPanel`,
and the full `GameView` integration. The previous `GameView.vue` was a placeholder stub; it is
now a complete production layout.

The key architectural decision was to decouple OOB enrichment from the Pinia store. The store
holds the raw server-schema `GameState` (unit IDs, hex positions, morale, orders) without any
OOB knowledge. `GameView.vue` fetches OOB data separately from the dev-tool endpoint and performs
the enrichment in a computed — flattening the OOB hierarchy via a recursive `_collectOobUnits`
walker into a `Map<unitId, {name, side, strengthPoints, counterFile}>`. `selectedDisplayUnit`
combines raw game state with OOB metadata just before passing to `UnitStatsPanel`. This keeps the
store portable (it can be used by future views without pulling in OOB loading concerns) and makes
the enrichment easy to swap out when production-stable public routes replace the dev-tool endpoints
at M8.

`UnitCounterLayer` renders SVG `<image>` elements at hex centers derived from `HexMapOverlay`'s
`cellById` map. Units at the same hex are x-offset by 4px per stacking index so overlapping
counters remain distinguishable. Only units with a `counterFile` in the OOB are rendered — supply
wagons and corps HQ nodes without counter assignments are intentionally invisible. Stacking order
follows insertion order; no sorting is applied at this milestone. The `units` prop added to
`HexMapOverlay` defaults to `[]` so all existing tool views remain unaffected.

The `GameView.test.js` suite required one non-obvious router fix: `createMemoryHistory(base)`
sets the URL _base_, not the initial location — so `route.params.id` was `undefined` on first
render when the base was set to `/games/game-1`. The fix was to call `await router.push('/games/game-1')`
before mounting, which resolves the navigation fully before the component's `onMounted` fires.
Deferred to M8: replacing the two dev-tool fetch calls (`/api/tools/map-test/data` and
`/api/tools/oob-editor/data`) with production-stable public routes; the comment `// M8:` marks
the exact lines in `GameView.vue`.
