# 2026-03-15

## 11:05 — Map Editor Full Implementation (Phases 1–5)

Completed the full five-phase map editor implementation plan that transforms the scaffold into a
production-ready terrain digitization tool. The work covered schema extensions, new UI components,
a major overlay refactor, interaction modes with localStorage autosave, and a rich hex-detail
panel with slope, wedge, edge, and feature editing.

**Phase 1 & 2 — Schema and CalibrationControls.** Added `rotation` (±15° float) and `locked`
(boolean) to `GridSpecSchema`. The rotation field feeds directly into an SVG `rotate()` transform
on the grid group, letting operators fine-tune map alignment without moving the anchor point. The
lock flag disables all numeric calibration inputs so a finalized calibration cannot be accidentally
bumped. The Lock button itself stays enabled so the operator can always unlock.

**Phase 3 — HexMapOverlay layer system.** The overlay previously hid unknown-terrain hexes and had
no layer controls; it now renders every hex unconditionally and gates visibility on a `layers` prop
object (`grid`, `terrain`, `elevation`, `wedges`, `edges`, `slopeArrows`). Pure geometry helpers
were extracted to `client/src/utils/hexGeometry.js` — `edgeMidpoint`, `edgeLine20_80`,
`wedgePolygonPoints`, and `adjacentHexId` — all independently tested with 23 unit tests. The
adjacentHexId function uses cube-coordinate arithmetic consistent with the existing `los.js`
helpers, keeping the coordinate system uniform across the client. A key tradeoff: edges are drawn
as 20%–80% segments rather than full side-to-side lines, which leaves visual breathing room at
corners where two features of different types meet.

**Phase 4 — EditorToolbar and MapEditorView interaction modes.** Four editor modes (select, paint,
elevation, edge) are managed in `MapEditorView` and reflected in the toolbar. Paint mode applies
terrain on mouseenter so dragging across hexes paints continuously. Elevation mode increments by
50 ft per click (shift-click decrements), matching the South Mountain contour interval. Edge mode
detects the nearest edge midpoint within 8px on click and mirrors the feature to the adjacent hex
automatically, preventing half-edge data inconsistencies. localStorage autosave writes on every
mutation under the key `lob-map-editor-mapdata-v1`; a draft banner compares `_savedAt` timestamps
to offer restore without overwriting a newer server save. The engine-export function strips all
`_`-prefixed private fields before offering a blob download, so internal metadata never leaks into
the game data files.

**Phase 5 — HexEditPanel, WedgeEditor, EdgeEditPanel.** The hex detail panel now exposes every
field the schema supports: terrain, elevation, slope direction (mini hex picker with seven
buttons), hex-level features, per-direction edge features, and sub-hex wedge elevations. The
`WedgeEditor` is a standalone SVG component with six clickable triangles; clicking a wedge opens
an inline number input at that wedge's centroid, and confirming emits the full updated array. The
`EdgeEditPanel` shows all six directions simultaneously with chips for existing features, a type
dropdown, and add/remove controls — emitting `{hexId, dir, features}` so the parent can update
both the hex and its mirror in a single handler. All components were mocked in
`MapEditorView.test.js` using `vi.mock()` to prevent honeycomb-grid's SVG geometry from hanging
jsdom during CI.

The final suite is 190 tests across 14 files, all green, with lint and Prettier checks clean.

## 18:46 — PR #7 Coverage Gaps and Bug Fixes (LOS Panel Reactivity, localStorage Ordering, Dead Edge Logic)

Addressed all findings from the PR #7 code review: one ERROR-level reactivity bug that silently
disabled the LOS hex-pick workflow, two WARNING-level logic bugs, and several coverage gaps in the
new LOS test panel code. Total suite now stands at 201 tests.

**B-2 (ERROR) — Vue 3 ref unwrapping in template inline functions.** The LOS event handlers
originally used bare assignments (`losSelectingHex = side`) inside inline arrow functions, which
rebind the local variable rather than mutating the ref — the classic Vue 3 trap. The attempted
fix of appending `.value` made things worse: Vue's template compiler auto-unwraps top-level refs
in template expressions, so `losSelectingHex.value` compiled to `unref(losSelectingHex).value`,
which is `null.value` — a TypeError. The correct fix is to extract named handler functions in
`<script setup>` (`onLosPickStart`, `onLosPickCancel`, `onLosSetHexA`, `onLosSetHexB`,
`onLosResult`) and reference them in the template. Inside a `<script setup>` function, refs are
plain ref objects and `.value` works normally with no compiler interference.

**B-3 — localStorage written before null-data guard.** The `save()` function persisted
calibration to localStorage unconditionally at the top, before checking whether `mapData.value`
existed. A failed PUT would leave stale calibration on disk even though the map save failed. The
fix moves the `localStorage.setItem` call into (a) the early-return branch for the
calibration-only case, and (b) the `res.ok` success branch after the PUT succeeds.

**B-1 — Dead edge-height logic on the target hex.** `edgeHeightBonus` accumulated
`losHeightBonus` from edge features for every step including the target, but the height-blocking
check only fires for `role === 'intermediate'`. The target's edge bonus affected `effectiveHeight`
displayed in step detail but could never block — misleading and inconsistent. Guarded with
`role !== 'target'` and exported `dirFromDelta` so it can be directly tested. The Q-1 cleanup
extracted the copy-pasted `DIR_DELTAS` array from two test cases into a single module-level
constant. New C-7/C-8 tests cover the `dq > dr && dq > ds` branch of `cubeRound` and the
null-return path of `dirFromDelta`. C-5/C-6 tests in `LosTestPanel.test.js` cover the `watch()`
prop-sync callbacks and the `onPickB` cancel branch. Five new MapEditorView tests verify the LOS
event handlers update the correct refs via the parent's prop bindings.
