## 01:49 — PR: Minor Debt Bundle — Issues #111 #112 #154 #162 #165 #166 #170

This PR closes seven accumulated technical debt items across the map editor component tree, grouped
into three phases by dependency and risk. The work was entirely structural — no game logic touched —
and leaves the test suite at 835 tests passing with 83.78% coverage.

Phase 1 extracted duplicate CSS from `CalibrationControls` and `ElevationSystemControls` into a
shared `client/src/assets/editor-form.css` stylesheet, imported non-scoped via Vue's
`<style src="...">` syntax. This pattern was chosen over a scoped approach because the rules
(`label`, `input[type='number']`) are framework-level form control styles shared across multiple
independent components and don't benefit from scoping.

Phase 2 made four structural changes. `ElevationSystemControls` was hoisted out of
`CalibrationControls` into `MapEditorView` as a sibling, eliminating an unnecessary prop
pass-through layer that was one of the original motivations for the component extraction. A
`CSS.supports` guard was added to `ToolChooser` to warn on invalid color values; this required a
`typeof CSS !== 'undefined'` check because jsdom (the Vitest test environment) doesn't implement
`CSS.supports`. The biggest structural change was the new `EdgeToolPanelShell` component, which
absorbs the duplicated mode-toggle chrome and shared CSS from `RoadToolPanel`, `StreamWallToolPanel`,
and `ContourToolPanel`. The shell uses Vue 3's `::v-slotted()` pseudo-selector to apply scoped CSS
to elements rendered inside its named and default slots, keeping all shared styles centralized
without leaking to the global scope.

Phase 3 removed all `console.assert` calls from `utils/compass.js` (these were hot-path assertions
that had no effect in production and were already covered by Vitest tests) and added an integration
test for the `HexMapOverlay` `onSvgMouseMove` rAF callback. The rAF test stubs
`requestAnimationFrame` to execute synchronously, computes hex center coordinates by parsing the
SVG group transform and polygon corner points from the rendered DOM, and asserts the full shape of
the resulting `hoverInfo` object. This required adding `hoverInfo` to `defineExpose` in the
component so Vue Test Utils could access the internal `<script setup>` ref.

One item was deferred from this bundle: `perf(compass): remove console.assert from hot path (#165)`
was confirmed complete — the assertions were removed and existing tests provide coverage. Net open
debt items decreased from 12 to 5 as a result of this bundle.

## 07:28 — PR: Map Editor Tool Closeout — Issues #137 #138 #139 #140 #141 #142

This PR closes the six open "Create usable map editor" milestone issues. Three were genuine
implementation gaps that required real work; three were already fully implemented by earlier PRs
and needed only a closure comment. The delivered features are: a fixed-value elevation slider,
terrain color swatches plus a building button in the terrain palette, and through-hex road
rendering in `HexMapOverlay`.

The elevation slider (#137) replaces the old click-to-increment/decrement interaction with a
`<input type="range">` that targets a fixed elevation value. The key architectural decision was
ownership: `targetElevation` is a `ref` owned by `MapEditorView` (not the panel), emitted
upward via `target-elevation-change` and threaded into `useHexInteraction` as a parameter.
This keeps the interaction composable stateless with respect to tool settings and matches the
existing pattern for `paintTerrain`. The old `adjustHexElevation(±1)` calls on click and
mouseenter were replaced by a `paintHexElevation()` helper that clamps and applies the slider's
fixed target; right-click clears to 0 via a separate `clearHexElevation()` helper. The
`adjustHexElevation` function is retained for API stability (it's used by the accordion's
raise-all/lower-all bulk operations).

Terrain swatches (#138) were straightforward: a `<span class="terrain-swatch">` is injected
inside each palette button with an inline `background-color` pulled from `TERRAIN_COLORS`. The
building button is appended unconditionally after the terrain list — it is not a terrain type but
triggers a `hexFeature` mutation instead of a `terrain` mutation. Rather than adding a separate
event type, building paint is handled inside `applyPaint()` in `useHexInteraction` by checking
`paintTerrain === 'building'` and writing `hexFeature: { type: 'building' }` while leaving the
hex's terrain field untouched. Right-click in building mode clears `hexFeature` to null. This
kept the paint mode UI uniform with no special-casing in `MapEditorView`.

Through-hex road rendering (#139) adds a `throughHexSegments` computed to `HexMapOverlay`
alongside the existing `cellsForEdges`. When `overlayConfig.edgeLine.style === 'through-hex'`,
the new computed uses `edgeToCenter()` (already present in `hexGeometry.js`) to build
centre→midpoint line segments for each canonical edge that has a matching road feature. The
segments render in a new `<g class="layer-through-hex-lines">` group; the existing along-edge
layer is unchanged and still active when `style` is absent. `RoadToolPanel` switches to
`style: 'through-hex'` immediately so roads connect through hex centres rather than running
along the edge. Issues #140 (stream/wall), #141 (contour), and #142 (ford/bridge) were closed
with comments citing PR #158 (edge tools bundle), which delivered the full implementations.

## 03:40 — PR: MapEditorView Extraction Debt Bundle (#125 #126)

This PR closes the two highest-priority structural debt items in the map editor: extracting
`useCalibration` and `useMapExport` out of the 1193-line `MapEditorView.vue`, and reducing the
`useMapPersistence` return surface from 23 flat values to a grouped `{ state, dialog, actions }`
shape. All three items were score-3 in the debt register and were flagged as compounding risks
before Phase 2 game logic is layered on top of the editor codebase.

The calibration extraction (`useCalibration.js`) encountered a dependency-ordering problem during
implementation. The original plan called for `useCalibration(mapData, unsaved)` — passing both refs
from `useMapPersistence` — but `useMapPersistence` itself needs the `calibration` ref at init time
(to write to localStorage on save), creating a circular dependency. The resolution was to split
responsibilities cleanly: `useCalibration` owns calibration state and localStorage persistence
only; the two-line `mapData.gridSpec` sync and `unsaved.value = true` remain in `MapEditorView` as
explicit cross-composable wiring. This is the right boundary — calibration logic vs. integration
glue. The `onCalibrationLoaded` callback (used by persistence to notify the caller when server data
arrives with a gridSpec) is returned from `useCalibration` and threaded directly into
`useMapPersistence`, so the view never touches the callback body.

The export extraction (`useMapExport.js`) was straightforward. The key design constraint preserved
from the original code is that `getEngineExport` is deliberately NOT a Vue computed property — the
`stripPrivateFields` deep-walk over ~2000 hex objects is expensive, and it only needs to run when
the export overlay opens or when copy/download is triggered, not on every reactive change. The
`exportSnapshot` ref is populated by a `watch(showExportOverlay, ...)` watcher, giving the template
a stable cached value without recomputing on unrelated state changes. `hasMapData` (the boolean
guard that disables the export button) moved into the composable alongside the rest of the export
logic.

The `useMapPersistence` grouping (`state`, `dialog`, `actions`) reduces the flat 23-name
destructure to three semantic buckets. `state` holds core data refs the view reads directly;
`dialog` holds the pull/push confirmation overlay refs and their cancel handlers; `actions` holds
all lifecycle functions. The MapEditorView update also removed a previously unused `serverSavedAt`
destructure that had been carried forward from an earlier refactor without ever being referenced in
the view. The test suite was fully updated to use the grouped API, with four new shape-assertion
tests verifying the structure at the composable boundary. Net result: 27 new tests, 863 total,
85.49% coverage.

## 11:20 — PR: Tech Debt Closeout — All Open Items (#151 #161 #169 #175 #176)

This PR closes all five outstanding technical debt items accumulated across the map editor
build, retiring the full open score of 9 before the final map editor polish phase begins. The
work was purely structural — no game logic, no user-visible behaviour changes — and leaves the
test suite at 905 tests passing across 47 test files with lint and format clean.

Phase 1 addressed the highest-risk item (#151): the monolithic `gridData` computed in
`HexMapOverlay` was rebuilding ~2,240 cell objects (polygon points, corners, wedge polygons)
on every calibration nudge, including `strokeWidth` and `northOffset` which have nothing to do
with hex geometry. The fix replaces the single computed with a `gridGeometry` shallowRef backed
by a `watch` on only the six geometry-affecting calibration fields (`cols`, `rows`, `hexWidth`,
`hexHeight`, `orientation`, `evenColUp`). Vue's `watch` does per-element primitive comparison on
the source array, so a `northOffset` nudge — which emits a new calibration object reference —
does not fire the geometry watcher. `gridData` remains a computed that reads `gridGeometry.value`
and does only fast arithmetic (tx/ty) plus a `geoCells.map()` for data enrichment. The expensive
`defineHex` + `new Grid` + polygon computation path is now skipped for all non-geometry changes.

Phase 2 extracted two composables. `useEdgePanelWiring` (#161) encapsulates the identical
`selectedType` ref and four event handler wrappers that were repeated verbatim for Road, Stream,
and Contour tool panels in `MapEditorView`. Each panel now gets one `useEdgePanelWiring(defaultType,
deps)` call; the template binds named methods (`road.onEdgePaint`, `road.onEdgeClear`, etc.)
instead of inline expressions three times over. `useEdgeLineLayer` (#169) extracts
`cellsForEdges` and `throughHexSegments` out of `HexMapOverlay` into a composable that takes
explicit `cells` and `overlayConfig` reactive inputs. The key benefit is that LOS, selection, and
other overlay state mutations no longer invalidate the edge line data, since those changes live in
unrelated `overlayConfig` keys that the composable does not read.

Phase 3 closed the two low-risk defensive items. `stripPrivateFields` (#175) gained a `_depth = 0`
recursion guard: nodes beyond depth 10 return `null` instead of recursing indefinitely. No real
caller passes malformed input — all paths go through Zod-validated `mapData` with at most ~5
nesting levels — but the guard prevents runaway recursion in any hypothetical future caller.
`onCalibrationChange` (#176) now applies the same `safeNumeric`/`safeBoolean`/`safeOrientation`
validation used by the localStorage read path, so invalid values written by a buggy caller are
clamped to defaults rather than stored as-is. Optional extra fields (`rotation`, `locked`) are
preserved unchanged, matching the intent of the original API.

The PR follows strict TDD throughout: failing tests were written before each implementation step.
The `gridGeometry` stability tests use Vue's `defineExpose` to assert reference identity, which
was the cleanest way to verify the watcher's selectivity without mocking the honeycomb-grid
module. All five GitHub issues are closed with fix-summary comments.
