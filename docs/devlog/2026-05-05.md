## 16:53 — PR: M4 — Game State Model, Initializer, Server Layer, Lobby UI

This PR delivers M4 in full: a Zod-validated in-memory game state model, a game state initializer that reads `scenario.json` setup data, a file-based and SQLite-backed server layer, and a minimal lobby UI for create/join without OAuth.

The data model work (Phase 1) produced two Zod schemas: `UnitStateSchema` and `GameStateSchema`. `UnitState` captures per-unit position, facing, morale state, wrecked flag, orders, ammo, and board presence. `GameState` wraps all units in a `z.record()` keyed by `unitId`, plus `reinforcementQueue`, `sides` (player tokens), turn/phase/initiative state, and a `status` enum (`setup`/`active`/`complete`). The schema acts as a runtime type boundary — all game state exits `initGameState()` through `GameStateSchema.parse()`, so malformed state is caught early.

`initGameState()` (Phase 2) handles two distinct setup entry formats from `scenario.json`. Zone-constraint entries (`{ setupZone: "within5Of", referenceHex, units: [...] }`) place all listed units at the reference hex for M4; the M5 setup-phase UI will let the union player reposition within the zone. Fixed-hex entries (`{ unitId, hex }` or `{ units: [{ unitId, hex }] }`) place CSA units at exact positions from the scenario. Reinforcements are pre-queued from `scenario.json reinforcements` into `reinforcementQueue` with their arrival turn computed from the scenario's `firstTurn` time string and `minutesPerTurn`. This design separates init-time work (build the queue once) from turn-resolution work (check queue on each turn transition).

The server layer (Phase 3) uses two separate stores: `gameFile.js` for JSON state persistence (`data/games/{id}/state.json` with atomic tmp-then-rename writes) and `gameSqlite.js` for the game index (`games` table with id, side tokens, status, created_at). Keeping them separate was intentional — `gameFile.js` is a drop-in swap point for DigitalOcean Spaces in M8, while SQLite stays for the games index even in production. The games router wires in `express-session` (in-memory store, secret from env) so that `POST /games` and `POST /games/:id/join` can record `sideToken` and `side` in the session — the minimal "pick a side" mechanism that stands in for OAuth until M8. The `_scenario` lazy cache inside `games.js` avoids reloading `scenario.json` on every request while remaining easy to reset in tests via module-level reset.

Phase 4 delivers the lobby UI. `useLobbyStore` (Pinia) wraps the three API calls (`fetchGames`, `createGame`, `joinGame`) with loading and error state. `LobbyView.vue` is deliberately minimal — a game list table with status and join button, a New Game button, and a side-assignment banner after create/join. Functional but not styled. `GameView.vue` is a placeholder stub for M5. All client code is tested with VTU and mocked stores; the session-management detail is invisible to the client tests since the store just reads `side` from the JSON response.
