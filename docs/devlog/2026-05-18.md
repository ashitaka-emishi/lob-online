# 2026-05-18

## 10:35 — PR: M5 Engine Debt Bundle 2 — Schema Hardening & Code Quality

This PR is the second M5 debt sprint, closing nine deferred findings from PRs #359, #375, and
#386. The items fall into three natural groups: UnitStateSchema refinements (Phase 1), engine
constants and loop hardening (Phase 2), and documentation corrections (Phase 3). One item (#369
— reserving `'stopped'` status) turned out to already be implemented; it was closed immediately
without a code change.

The most consequential Phase 1 change is the `isDetached ↔ orders !== null` cross-field
refinement on `UnitStateSchema`. The constraint is one-directional: a detached brigade must have
its own `UnitOrderState`, but a non-detached unit with non-null orders is valid (divisions hold
their own orders without being detached). This is distinct from the biconditional
`activityPhase ↔ phase === 'activity'` added in PR #386 — that was a full iff because both
directions were always meaningful; here only one direction matters. The `.default(false)` on
`isDetached` was straightforward, and adding `path` metadata to the five `UnitOrderState`
`.refine()` calls required no behavior change but surfaces field-level errors instead of
object-root errors.

Phase 2 extracted all phase and step string literals into `server/src/engine/phases.js`. This
is a required cleanup before M6 adds morale and combat phases — with only three phases it is
manageable to grep for bare strings, but after M6 expansion that window closes. The
`drainAutoSteps` envelope guard merits explanation: the original issue (#389) proposed a full
`GameStateSchema.safeParse()` call per iteration, but that would validate the entire game state
(including all units) on every auto-step, which is expensive in the hot path. Instead, a
lightweight one-directional check was used: if `activityPhase` is non-null outside the activity
phase, or `ordersPhase` is non-null outside the command phase, throw immediately. The reverse
direction (envelope is null when expected) is already caught by the terminal `safeParse()` call
at the end of `dispatch()`. The intermediate state `{ phase: 'command', ordersPhase: null }` is
legitimately produced by `handleEndPhase` at the `attackRecovery` step, so the full biconditional
would fire as a false positive on every END_PHASE dispatch.

Phase 3 corrected a citation error introduced in Phase 1: the `isDetached` comment had been
updated to "SM §7.3" as a placeholder, but the domain-expert agent confirmed this section covers
Random Events, not detachment. The correct references are SM §2.3 (Union detachments) and SM
§3.3 (Confederate detachments), sourced from the SM scenario booklet page 2. This citation
correction was applied to both `gameState.schema.js` and `init.js`.

Explicitly deferred: #340 (sideToken auth guard — auth/session surface, own Checkpointed track),
#361 (scenario-start detached brigades — requires scenario data model changes), #360
(reinforcement orderType in initGameState — own track), and #363 (migration path for orders
schema — needs migration design). The remaining score-2 stubs (#379, #381, #382, #383) are
blocked on M6 morale/combat data and remain correctly deferred.

## 22:26 — PR: Pre-UI Debt Sweep — initGameState Correctness + Schema Quality

This PR closes six deferred debt items (scores 4+4+3+2+2+2 = 17 points) before starting M5 game-map UI work, dropping the open register from 21 items / 43 points to 14 items / 24 points. Additionally, a stale entry for #340 (already closed in PR #339) was corrected. The items were chosen as the highest-impact actionable group that could be fixed together without conflicting change surfaces — all six touch the same server-side cluster: `init.js`, `gameState.schema.js`, `gameFile.js`, `routes/games.js`, and `scenario.json`.

The two score-4 items (#361 and #360) were correctness bugs in `initGameState()`. The Garland Brigade group in `scenario.json` was marked DETACHED in comments but had no machine-readable `isDetached` field, so all units started with `isDetached: false`. Similarly, every reinforcement group carries an `orderType` field in `scenario.json` that `processReinforcementGroup()` silently discarded, leaving all reinforcements with `orders: null`. Both fixes follow the same pattern: read the field from the scenario data and forward it through `defaultUnit()`. The key design decision was to keep `isDetached` as a group-level annotation (propagated to each unit) rather than per-unit, matching how the real scenario groups work.

The schema migration fix (#363) settled on `z.literal(1)` without a default, combined with an explicit pre-check in `loadGame()`. Using `.default(1)` would have silently accepted legacy saves by supplying the default, defeating the purpose. The pre-check throws a descriptive error with both the stored and expected version, making the diagnosis actionable. `STATE_SCHEMA_VERSION = 1` lives in `constants/schemaVersion.js`, a neutral layer that both `init.js` and `gameFile.js` can import without inverting the dependency direction — the same pattern used for `constants/phases.js`. All three test fixture sets (`BASE_GAME_STATE`, `BASE_VERSIONED_STATE`, `SETUP_STATE`) were updated to include `schemaVersion: 1`.

The scenario cache fix (#337) was wired through `editorRouteFactory.js` rather than adding a custom PUT handler to `scenarioEditor.js`. The factory gained an optional `afterSave` callback that fires after a successful atomic rename; `scenarioEditor.js` passes `clearScenarioCache` as that callback. This keeps the factory generic and testable in isolation — the `afterSave` tests in `editorRouteFactory.test.js` verify the callback is called on success and skipped on validation failure, while `scenarioEditor.test.js` mocks `games.js` to verify the wiring end-to-end.

The two code-quality items were straightforward but important for M6 readability. `isOrderHolder(unit)` (#364) centralises the `orders !== null` check and its JSDoc explicitly documents the semantic distinction between `null` (non-order-holding unit) and `{ status: 'none' }` (order-holder with no active order) — a subtle difference that query code in M6 will need to get right. The `defaultUnit()` refactor (#371) converts six positional arguments to a named options object, eliminating the transposition risk that would grow as M6 adds fields. No new debt was introduced; all findings were fixed in-place.
