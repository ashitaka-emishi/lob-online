## 01:13 — PR: M5 Engine Debt Bundle — Issues #377 #378 #380 #384

This PR closes four actionable findings deferred from PR #375's team-review of the M5 phase engine. All four are in the action engine's dispatch layer and the `GameStateSchema` — the two surfaces that gate correctness before the game map UI is built on top of them. The remaining five deferred findings (#379 #381 #382 #383 #385) require M6 morale/combat data or are indefinitely deferred fuzz-test work; they were intentionally excluded from this sprint.

The first two items (#384, #377) are pure dispatch hardening. The `UNKNOWN_ACTION` error message already included `action.type` in the PR #375 implementation, so the only work was adding a test via `vi.mock` to cover that otherwise-unreachable path (the `!handler` guard is dead code when `getValidActions` and `HANDLERS` stay in sync, but it's worth testing to lock in the message format). For `#377`, rather than relying on `getValidActions` returning `[]` for the wrong player — which achieves the same rejection but produces a generic "not a valid action" message — an explicit guard was added before the `getValidActions` call. It emits "It is union's turn, not confederate's" so the error is unambiguous at the API boundary. The guard skips the check when `state.activePlayer === null` (setup phase, no player designated yet).

The schema items (#378, #380) add cross-field Zod biconditionals to `GameStateSchema`. The core insight is that `activityPhase` and `ordersPhase` are phase-envelope objects — they're supposed to exist only while their respective phases are active. Without the refinements, a state with `phase: 'rally'` and a non-null `activityPhase` would pass validation. With them, `GameStateSchema.safeParse()` (called at the end of every `dispatch()`) catches such inconsistencies before they reach callers. Both refinements are bidirectional (iff, not just "must be null outside"): `activityPhase !== null ↔ phase === 'activity'` and `ordersPhase !== null ↔ phase === 'command'`. The biconditional is safe because `drainAutoSteps` always sets both fields atomically when transitioning phases — intermediate states inside the drain loop are never schema-validated.

The implementation order was chosen deliberately: fix the dispatch layer first (low risk, no schema changes), get a checkpoint review, then add the schema refinements. This kept the riskier persistence-surface changes isolated in their own commits with clear test coverage. No new debt was introduced; all four issues will be closed on merge.
