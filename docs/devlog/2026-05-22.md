## 02:20 — PR: Game Map Viewport Fix — Scrolling, gridSpec, Counter Sizing

This PR resolves three related issues on the M5 game map route (`/games/:id`) that were filed together as #406. The page was scrolling at the document level rather than confining scroll to the map viewport, the hex grid calibration was reading stale values from localStorage instead of the scenario's canonical `gridSpec`, and unit counters were not sized correctly relative to the hex geometry.

The document overflow fix was a straightforward CSS change: `overflow: hidden` on both `html` and `body`, plus `min-width: 0` on the `.map-area` flex child in `GameView.vue`. The `min-width: 0` is the non-obvious one — without it, a flex child defaults to `min-width: auto` (sized to its content), which prevents it from shrinking enough to trigger the horizontal scrollbar. The flex body centering that was previously on `body` was moved to `LobbyView`'s own root element, since the game view's full-height layout is incompatible with flex-centered body.

The gridSpec fix introduced a new `GET /api/v1/games/:id/map-config` endpoint in the games router. It loads `map.json` at server startup (mirroring the pattern in `mapTest.js`) and serves `{ gridSpec, hexes }` with no `requireSide` guard — this is static scenario data, identical for all players. The endpoint is placed before the `GET /:id` catch-all to avoid Express swallowing the sub-path. In `useGameStore`, `loadGame()` now fetches both the game state and map config in parallel with `Promise.all`, treating the map-config fetch as non-fatal (a rejection leaves `gridSpec` and `hexes` null). `GameView` replaced `useCalibration()` with a simple computed that merges `DEFAULT_CALIBRATION` with `gameStore.gridSpec`, and the dev-tool fetch of `/api/tools/map-test/data` was removed entirely.

Counter sizing required understanding honeycomb-grid's coordinate system. The calibration fields `hexWidth` and `hexHeight` are passed as `xRadius` and `yRadius` to `defineHex` — they are circumradii, not inradii. For the SM map both are approximately 40.5–40.7 SVG units. For a regular flat-top hex the circumradius equals the side length, so `calibration.hexWidth` is the hex side length. Counters are now sized at `hexSideLength × sizeRatio` (default 1.1), making the prop name and math explicit and giving a counter of roughly 44.6 SVG units on a hex with ~81 SVG unit diameter.

A pre-existing Prettier violation in `countersManifest.js` and uncommitted map-editor changes in `map.json` were encountered during the stash/pop cycle and fixed before the quality gates ran. All 2260 tests pass and `quality:strict` exits 0.

## 06:32 — PR: Post-M5 debt bundle (Phase 1 + Phase 3 + Phase 4.1)

This PR closes 8 of 17 actionable debt items from the post-M5 backlog, reducing the open score from 51 toward the target of ≤ 18. The work is organized around four deliverables: the OOB production data path, session and constant quality improvements, test duplication reduction, and a cosmetic accessibility fix.

The most significant change is the OOB data path (#431, #401, #404, #432). The old code fetched from `/api/tools/oob-editor/data`, a MAP_EDITOR_ENABLED-gated dev route, which meant the game view silently received no data in production. A new `/api/v1/oob` route was added outside the feature flag, backed by a standalone `server/src/engine/oob.js` module that loads and validates against OOBSchema. On the client, the inline OOB fetch, walk function, and enriched-unit computed were extracted into a `useOobData` composable, which GameView now imports. This also resolved the stale aria-label bug (#432) — `UnitCounterLayer` now receives a human-readable `name` field from the composable rather than falling back to the unit ID.

Phase 3 addressed two code-quality items. The `session.regenerate()` promise wrapper (#411) appeared three times inline in `games.js`; it was extracted to a named `regenerateSession()` helper. The `SIDES` constant (#412) — `{ UNION: 'union', CONFEDERATE: 'confederate' }` — was defined in canonical modules at `server/src/util/sides.js` and `client/src/utils/sides.js` and threaded through `games.js` and `useOobData.js`, eliminating all bare string literals. Test consolidation (#346) removed 58 duplicate test cases spread across five per-route editor test files, the compass utility test, the TableTestView test, and the router test. The pattern in all cases was the same: generic factory behavior (backup creation, `_savedAt`, backup trimming, mkdir) was tested once in `editorRouteFactory.test.js` and re-tested verbatim in each concrete route's file. The per-route files were trimmed to schema-specific coverage only.

Two fixes were backported from the still-open PR #420 (`feat/406-game-view-viewport-fix`): the document-level `overflow: hidden` + flex-centering removal that caused the map view to not scroll, and the `hexInradius` → `hexSideLength` prop rename (with `sizeRatio` default 0.8 → 1.1) in `UnitCounterLayer` that caused counters to render too small. These were manual cherry-picks since PR #420 is not yet merged. The `LobbyView` error color (#413) was changed from `color: red` to `color: #c0392b` with a left border added as a non-color visual cue.

Phase 2 items (#421–#427) and Phase 4 items #428–#430 depend on code that only exists on PR #420 — specifically the `useCalibration` composable restructuring and the `/scenarios/:scenarioId/map-config` route. These 9 remaining items are blocked until #420 merges and will be picked up in a follow-on commit or a separate track.

## 21:11 — PR: Post-M5 debt bundle (Phase 2 + Phase 4.2–4.4)

This PR completes the seven map-config/store contract items from the Phase 2 backlog (#421–#427), plus three remaining Phase 4 cosmetics (#428–#430). Together with the earlier PR #433 (Phase 1 + Phase 3 + Phase 4.1), this closes all 17 items in the post-M5 debt bundle and brings the debt score from 51 to ≤ 18.

The largest change is the map-config route architecture (#421). The old `GET /api/v1/games/:id/map-config` endpoint lived in `games.js` and was game-scoped, but the data it served was scenario-static — the same `gridSpec` and `hexes` for any game of the same scenario. It was moved to a new `server/src/routes/scenarios.js` router at `GET /api/v1/scenarios/:scenarioId/map-config`. On the client, `useGameStore.loadGame()` was changed from a parallel `Promise.all` to a sequential fetch: the game-state response must arrive first so that `state.scenarioId` is available when constructing the map-config URL. The map-config fetch is still non-fatal — a rejection sets `mapConfigError` without blocking the game-state load.

Testability of the 503 startup-error path (#423) was a design constraint. Module-level initialization code (the `try { _mapData = loadMap() } catch` block) runs once when the module is first imported, which means a mock that throws needs to be in place _before_ the import. The solution uses `vi.resetModules()` to discard the module cache, then re-imports `scenarios.js` with a `loadMap` mock that throws — forcing re-evaluation in the same test. This pattern makes the 503 branch fully exercisable without restructuring the production code.

The `sanitizeCalibration` extraction (#424) moved the function and `DEFAULT_CALIBRATION` from `useCalibration.js` (a Map-Editor-specific composable) to `client/src/utils/calibration.js` (app-wide pure utilities). `useCalibration.js` now re-exports both from the utility module, so the Map Editor retains its import paths unchanged. The store boundary was the right place to call `sanitizeCalibration` on the incoming `gridSpec` (#425): the store is the single ingest point for external JSON, so a NaN or invalid orientation in the API response gets normalized once, before any consumer sees the value. A comment documents the field-name contract between `DEFAULT_CALIBRATION` and `gridSpec` (#426), explaining that the merge in `GameView` works because both sides use the same keys — a rename on either side produces a default fallback rather than a silent wrong value.

The `loadMap()` double-call warning (#427) was fixed by adding module-level caching to `map.js` itself: a `Map<path, frozenData>` cache means the second call from `mapTest.js` returns the already-frozen result immediately. The `STUB_GRID_SPEC` fixture extraction (#429) created `client/src/test/fixtures.js` as a shared source for both `useGameStore.test.js` and `GameView.test.js` — the SM-scale fixture (`64×35`) and a mini fixture (`4×3`) live there. The `Cache-Control: public, max-age=3600` header (#430) was a one-liner on the new `scenarios.js` route, which is the correct home for it since the old game-scoped endpoint would have been incorrect (game data can change; scenario data cannot). A `.prettierignore` entry for `countersManifest.js` was also added — the Vite plugin regenerates this file as compact single-line JSON on every build, which breaks `format:check` in the `quality:strict` sequence.
