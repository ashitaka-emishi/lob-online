## 13:14 — Implement #80: northOffset + cardinal edge labels (north-calibration track)

This track adds `northOffset` to the `gridSpec` schema, implements a `getEdgeLabels` utility, replaces
the plain number input in `CalibrationControls` with a visual SVG picker, and propagates geographic
direction labels through `WedgeEditor`, `HexEditPanel`, and `HexMapOverlay`. It was implemented using
the new conductor workflow as the first end-to-end test of that infrastructure on this project.

The core design challenge was encoding geographic direction on a flat-top hex. A hex has 6 edges and
6 vertices — 12 distinct 30° positions — but only 8 compass points are in common use. The solution
is a 12-element lookup array `COMPASS_12 = ['N','NE','NE','E','SE','SE','S','SW','SW','W','NW','NW']`
where each vertex position shares its label with one of the adjacent edges. The formula
`COMPASS_12[((i*2 - northOffset) % 12 + 12) % 12]` maps each of the 6 geometric edge indices `i` to a
geographic label given the northOffset. This produces the intuitive results: `northOffset=0` (top edge
= geographic N) → `['N','NE','SE','S','SW','NW']`; `northOffset=3` (right vertex = geographic N, the
South Mountain default) → `['W','NW','NE','E','SE','SW']`.

The visual picker in `CalibrationControls` uses flat-top hex SVG geometry with circumradius R=28.
Even-index positions (0,2,4,6,8,10) are edge midpoints at distance `Rin = R*√3/2` from center;
odd-index positions (1,3,5,7,9,11) are vertices at distance R. Clicking any of the 12 circles emits
`calibration-change` with the updated northOffset. The selected circle is highlighted yellow. This is
purely a data-entry convenience; the SVG is not interactive in the main game view.

Label propagation was the most mechanical phase: `WedgeEditor` receives `northOffset` as a prop and
replaces its static `WEDGE_DIRS` array with `getEdgeLabels(northOffset)`; `HexEditPanel` drives its
slope picker buttons from `edgeLabels` computed from the same utility (clicking button at position `i`
stores slope index `i` directly — the geometric and geographic orderings are aligned); `HexMapOverlay`
adds a text label beside each slope arrow by calling `getEdgeLabels(calibration.northOffset)[cell.slope]`.
One design decision in `HexEditPanel`: the "Show/Hide Wedge Editor" toggle was removed entirely — the
`WedgeEditor` now renders automatically whenever `wedgeElevations` data exists, and an "Add Wedge Elevations"
button initializes the array from null. This simplifies state management and removes a UX friction point.

16 new tests were added across 6 files (schema, hexGeometry, CalibrationControls, WedgeEditor, HexEditPanel,
HexMapOverlay), bringing the suite to 354 tests at 76.77% coverage. The track also served as the inaugural
use of `conductor/` scaffolding on this project — the conductor directory was created from CLAUDE.md context
during this session as the brownfield initialization step.

## 15:10 — PR: Move slope arrow geometry into gridData computed (#87)

This PR is a focused single-file refactor of `HexMapOverlay.vue` that moves slope arrow
geometry computation out of the template layer and into the `gridData` computed property.
The motivation was a code-review finding on PR #86: the two template-layer functions
`slopeArrowLine(cell)` and `slopeArrowLabel(cell)` were being called up to four times per
cell per render (three `slopeArrowLine` calls — one `v-if` guard and two attribute bindings
— and one `slopeArrowLabel` call) inside a `v-for` over up to 2240 cells, and this
inconsistency with the established `gridData` data-preparation pattern was independently
flagged by both the performance and architecture reviewers.

The fix is straightforward: inside the `gridData` forEach loop, after `cx`/`cy` are
established, we compute an intermediate `slopeMid = edgeMidpoint(corners, slopeDir)` and
from it derive `slopeArrowLine = { x1, y1, x2, y2 }` and `slopeArrowLabel` (the geographic
compass label for that slope index at the current `northOffset`). Both are pushed into the
cell object and then read directly in the template as `cell.slopeArrowLine` /
`cell.slopeArrowLabel`. The `northOffset` derivation is hoisted before the loop so it is
not re-evaluated per cell.

The template simplification is notable: the redundant
`v-if="cell.slope !== null && cell.slope !== undefined"` wrapper around the slope arrows
block is replaced by the single `v-if="cell.slopeArrowLine"` guard, which is already `null`
for cells with no slope. The two standalone helper functions are deleted entirely. The net
diff is 16 insertions and 24 deletions.

All 354 existing tests pass without modification — the test suite asserts on rendered SVG
output, not on the internal helper functions, so the refactor is verified end-to-end at the
right level. Nothing is deferred; this track is complete in a single commit.

## 16:38 — Implement #81: Refactor map elevation model — integer levels, integer wedges, auto-derive

This track refactors the map elevation model across four phases: schema enforcement, data migration, WedgeEditor UX, and a new auto-derive tool. The underlying motivation is to normalize the elevation encoding before the LoS engine is built — the engine needs integer contour levels, not absolute feet, and needs wedge offsets to be semantically defined as relative integers.

Phase 1 tightened `MapSchema` with `z.number().int().min(0).max(21)` for per-hex `elevation` (zero-based level index, max 21 for 22 SM levels) and `z.number().int()` on each `wedgeElevations` tuple element. Two new required fields — `baseElevation` and `elevationLevels` — were added to the `elevationSystem` object. The decision to hard-code `.max(21)` in the schema deserves a note: the issue test required rejecting `elevation: 550` (an absolute feet value), but the acceptance criteria spec only said `.int().min(0)`. Since 550 is a valid integer ≥ 0, a max constraint was necessary. With SM having 22 levels (0–21), `.max(21)` is the correct bound. 21 new schema tests were added.

Phase 2 migrated `data/scenarios/south-mountain/map.json`. In practice, the existing `elevation` values (1 and 2) were already integer level indices — they had never been stored as feet. So the "migration" was entirely adding the two new metadata fields and updating the `_note` to document the encoding formula. The `_note` is worth preserving: `feet = (elevation + wedgeElevations[i]) * contourInterval + baseElevation` is the inverse that the future LoS engine will need.

Phase 3 replaced the `WedgeEditor` text-input edit form with direct click interactions: left-click decrements by 1, right-click (`@contextmenu.prevent`) increments by 1. The existing SVG text labels already displayed the current value on each wedge triangle, so no new display work was needed. Three existing tests that checked for `.wedge-input-row` and `.wedge-input` elements were rewritten to test the new click-emit behavior; one new test was added to verify value labels render correctly.

Phase 4 introduced `client/src/utils/elevationDerive.js` — a pure function `deriveEdgesAndSlope(hex)` that derives slope index (index of minimum wedge value) and hexside edge types from `abs(wedgeElevations[i])`: delta 0 → remove slope edge, 1 → `slope`, 2 → `extremeSlope`, ≥3 → `verticalSlope`. Non-elevation features (road, stream, stoneWall) are preserved via explicit filtering against an `ELEVATION_EDGE_TYPES` Set. The function was kept pure (deep-clones edges, does not mutate input) specifically to make it unit-testable in isolation. The "Auto-derive edges & slope" button in `HexEditPanel` calls the helper for the current hex, then iterates all 6 sides, looks up each neighbor via `adjacentHexId`, sets the neighbor's opposite wedge `(i+3)%6`, and re-derives the neighbor's edges — emitting `hex-update` for each. The `hexes` and `gridSpec` props were added to `HexEditPanel` and wired up in `MapEditorView`. 16 new tests cover the helper. Total suite: 395 tests.

## 16:00 — PR: Throttle HexMapOverlay onSvgMouseMove with rAF + Spatial Index

This PR addresses a pre-existing performance issue in the map editor flagged during the team-review
of PR #86. The `onSvgMouseMove` handler was performing ~13,440 distance calculations on every raw
mousemove event by iterating all grid cells across all 6 edge directions with no throttling. At 60 Hz
cursor movement over the full South Mountain map this saturated a CPU core with pure distance math —
a real problem for multi-hour digitization sessions.

Two complementary fixes were applied. First, a `requestAnimationFrame` throttle collapses all
mousemove events within a single frame into one handler invocation, capping the calculation rate at
the display refresh frequency. Second, a spatial index reduces the per-frame edge search from
~13,440 checks to at most 42: `grid.pointToHex` identifies the hex under the cursor, and a new
`getCellAndNeighbors` helper returns only that cell and its six immediate neighbors to check.

The key design decision was to implement `getCellAndNeighbors` using hand-rolled ODD_Q offset math
rather than the honeycomb-grid `grid.neighborOf` API. During implementation `neighborOf` threw on
boundary hexes due to internal `q` property access on undefined neighbors, so the pure-math
approach was adopted instead — it's simpler, dependency-free, and consistent with the existing
`adjacentHexId` function in `hexGeometry.js`. Both `findNearestEdge` and `getCellAndNeighbors` were
exported from `hexGeometry.js` so they could be unit-tested directly, which also let us remove the
duplicated inline edge-nearest loop from `onSvgClick` and replace it with the shared helper.

The `gridData` computed was extended with `col`, `row`, and a `cellByColRow` Map so that
`getCellAndNeighbors` can do O(1) neighbor lookups without rebuilding any index at call-time. The
13 new unit tests (in hexGeometry.test.js) cover both helpers thoroughly including boundary and
multi-cell cases. No behavior change to edge click or edge hover is expected; the visible change is
smoother cursor response during heavy digitization sessions.

## 16:41 — PR: Refactor map elevation model (#81)

This PR normalizes the elevation encoding in the map data model ahead of the LoS engine.
Previously, `elevation` was an untyped `z.number()` that could hold absolute feet values,
`wedgeElevations` elements were unvalidated floats, and `elevationSystem` lacked the
`baseElevation` and `elevationLevels` fields needed to interpret those values. The refactor
tightens all three, migrates the existing `map.json` data, replaces the WedgeEditor text input
with direct click interactions, and adds an "Auto-derive edges & slope" button that computes
hexside edge types from wedge values.

The schema decision that warrants explanation is `.max(21)` on `elevation`. The acceptance
criteria spec only says `.int().min(0)`, but the required test "rejects `elevation: 550`" forces a
max constraint — 550 is a valid integer ≥ 0. With SM having 22 contour levels (0–21), `.max(21)`
is the correct bound and makes the schema reject any value stored in absolute feet. The data
migration turned out to be a no-op: the existing `map.json` elevation values (1, 2) were already
integer level indices, never stored as feet. The real work was adding `baseElevation: 500`,
`elevationLevels: 22`, and the encoding formula `_note`.

The `elevationDerive.js` pure function was kept pure (deep-clones edges, no input mutation)
deliberately — it lets the Vitest tests run in plain Node without a component mount, and it means
the auto-derive computation can later be called server-side if needed. The delta-to-edge-type
mapping follows SM_RULES 1.1 exactly. Non-elevation edge features are preserved by filtering
against an `ELEVATION_EDGE_TYPES` Set before rebuilding each edge's feature list.

Neighbor propagation in `HexEditPanel.autoDerive()` required passing `hexes` and `gridSpec` as new
props. The parent (`MapEditorView`) already handles multiple `hex-update` emissions from edge-toggle
mirroring, so the pattern is established. Neighbor hexes that don't yet exist in `mapData.hexes`
are handled gracefully with `{ hex: neighborId, terrain: 'unknown' }` as a fallback — consistent
with other places in `MapEditorView` that synthesize stub hexes for new IDs. 37 new tests bring
the total to 395.
