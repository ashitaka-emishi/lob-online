## 07:18 — PR: Integrate wshobson/agents plugins and best-practice Claude patterns

This PR modernizes the Claude ecosystem's configuration layer by adopting the patterns established
in `shanraisshan/claude-code-best-practice` and documenting the `wshobson/agents` plugin
marketplace integration. The motivation was to reduce the amount of custom infrastructure by
building on standardized tooling rather than maintaining everything from scratch.

Most of the groundwork turned out to already be in place from a prior session: `settings.json`
existed with shared permissions, and all five agents already carried `model: sonnet`. The actual
delta was smaller than anticipated — the main work was fixing a non-portable absolute path in the
PostToolUse hook, cleaning up `settings.local.json` to contain only truly machine-specific entries,
and splitting the verbose Coding Standards and SDLC sections of `CLAUDE.md` into separate
`.claude/rules/` files. This keeps CLAUDE.md focused and under 200 lines while still giving Claude
Code access to the full detail via the `@` import mechanism.

The plugin coexistence note in CLAUDE.md is the key design decision: rather than replacing the
existing lob-specific skills (`/issue-implement`, `/pr-review`) with conductor/agent-teams
equivalents, both systems coexist. The table in CLAUDE.md makes the boundary explicit — native
skills are the primary workflow; plugin commands are available as supplementary tools. This was a
deliberate choice to preserve the investment in the battle-tested SDLC pipeline while still making
the wshobson/agents capabilities available. The actual plugin installation (`/plugin marketplace add
wshobson/agents` + `/plugin install conductor` + `/plugin install agent-teams`) is a one-time
Claude Code CLI operation documented in CLAUDE.md, not automated in code.

Three `settings.local.json` references in agent design docs and templates were also corrected to
point to `settings.json` — the shared file where `Skill()` and `Agent()` permissions now live.
The `tutorial-new-agent.md` ecosystem doc had five such stale references that were updated.

Deferred: removing the custom Node.js orchestrator (`server/src/orchestrator/`) and migrating the
6-HCP issue-implement workflow to conductor tracks. That is a larger undertaking requiring a
dedicated design pass and is tracked as a follow-up issue.

## 13:14 — Implement #80: northOffset + cardinal edge labels (north-calibration track)

This track adds `northOffset` to the `gridSpec` schema, implements a `getEdgeLabels` utility, replaces
the plain number input in `CalibrationControls` with a visual SVG picker, and propagates geographic
direction labels through `WedgeEditor`, `HexEditPanel`, and `HexMapOverlay`. It was implemented using
the new conductor workflow as the first end-to-end test of that infrastructure on this project.

The core design challenge was encoding geographic direction on a flat-top hex. A hex has 6 edges and
6 vertices — 12 distinct 30° positions — but only 8 compass points are in common use. The solution
is a 12-element lookup array `COMPASS_12 = ['N','NE','NE','E','SE','SE','S','SW','SW','W','NW','NW']`
where each vertex position shares its label with one of the adjacent edges. The formula
`COMPASS_12[((i*2 - northOffset) % 12 + 12) % 12]` maps each of the 6 geometric edge indices `i` to a
geographic label given the northOffset. This produces the intuitive results: `northOffset=0` (top edge
= geographic N) → `['N','NE','SE','S','SW','NW']`; `northOffset=3` (right vertex = geographic N, the
South Mountain default) → `['W','NW','NE','E','SE','SW']`.

The visual picker in `CalibrationControls` uses flat-top hex SVG geometry with circumradius R=28.
Even-index positions (0,2,4,6,8,10) are edge midpoints at distance `Rin = R*√3/2` from center;
odd-index positions (1,3,5,7,9,11) are vertices at distance R. Clicking any of the 12 circles emits
`calibration-change` with the updated northOffset. The selected circle is highlighted yellow. This is
purely a data-entry convenience; the SVG is not interactive in the main game view.

Label propagation was the most mechanical phase: `WedgeEditor` receives `northOffset` as a prop and
replaces its static `WEDGE_DIRS` array with `getEdgeLabels(northOffset)`; `HexEditPanel` drives its
slope picker buttons from `edgeLabels` computed from the same utility (clicking button at position `i`
stores slope index `i` directly — the geometric and geographic orderings are aligned); `HexMapOverlay`
adds a text label beside each slope arrow by calling `getEdgeLabels(calibration.northOffset)[cell.slope]`.
One design decision in `HexEditPanel`: the "Show/Hide Wedge Editor" toggle was removed entirely — the
`WedgeEditor` now renders automatically whenever `wedgeElevations` data exists, and an "Add Wedge Elevations"
button initializes the array from null. This simplifies state management and removes a UX friction point.

16 new tests were added across 6 files (schema, hexGeometry, CalibrationControls, WedgeEditor, HexEditPanel,
HexMapOverlay), bringing the suite to 354 tests at 76.77% coverage. The track also served as the inaugural
use of `conductor/` scaffolding on this project — the conductor directory was created from CLAUDE.md context
during this session as the brownfield initialization step.

## 14:21 — PR: Add northOffset to gridSpec with cardinal edge labels (#80)

This PR delivers the north calibration feature scoped in issue #80: a 12-position `northOffset`
integer in the `gridSpec` schema that encodes which compass position geometric north points toward
on the physical map sheet, plus a full propagation of geographic direction labels through every
component that displays or accepts directional input.

The key abstraction is `getEdgeLabels(northOffset)` in `hexGeometry.js`, which maps the 6 geometric
hex edges to 8-compass geographic labels using a 12-element lookup array and the formula
`COMPASS_12[((i*2 - northOffset) % 12 + 12) % 12]`. This handles the mismatch between the 6 hex
edges and 12 discrete 30° positions gracefully: vertex positions share a label with one of their
adjacent edges (e.g., the right vertex is 'E' for both northOffset=3 positions that straddle it).
South Mountain uses `northOffset=3` (right vertex = geographic N), which produces the canonical SM
labels W/NW/NE/E/SE/SW for the six edges.

The `CalibrationControls` visual picker replaces the plain number input with a flat-top SVG hex
diagram showing 12 clickable circles — 6 at edge midpoints (distance Rin from center) and 6 at
vertices (distance R). The selected position is highlighted yellow. This is purely a map-editor
convenience; it has no rendering impact in the game view.

The `WedgeEditor` toggle ("Show/Hide Wedge Editor") was removed as part of this PR. The editor
now renders automatically whenever `wedgeElevations` data exists, and an "Add Wedge Elevations"
button initializes the array from null. This was a natural simplification: the toggle existed to
save vertical space, but wedge elevations are rarely present in practice, and when they are, the
editor should always be visible. Removing the toggle eliminates a state variable and a class of
tests that were testing UI toggle behavior rather than data behavior.

This PR also initializes the `conductor/` scaffolding directory — the first use of the
`/conductor:new-track` + `/conductor:implement` workflow on this project. The conductor artifacts
(product.md, tech-stack.md, workflow.md, tracks.md, and the north-calibration track docs) are
committed here as part of the brownfield onboarding. 16 new tests bring the suite to 354 passing
at 76.77% coverage.
