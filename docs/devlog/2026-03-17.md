# 2026-03-17

## 07:18 — PR: Integrate wshobson/agents plugins and best-practice Claude patterns

This PR modernizes the Claude ecosystem's configuration layer by adopting the patterns established
in `shanraisshan/claude-code-best-practice` and documenting the `wshobson/agents` plugin
marketplace integration. The motivation was to reduce the amount of custom infrastructure by
building on standardized tooling rather than maintaining everything from scratch.

Most of the groundwork turned out to already be in place from a prior session: `settings.json`
existed with shared permissions, and all five agents already carried `model: sonnet`. The actual
delta was smaller than anticipated — the main work was fixing a non-portable absolute path in the
PostToolUse hook, cleaning up `settings.local.json` to contain only truly machine-specific entries,
and splitting the verbose Coding Standards and SDLC sections of `CLAUDE.md` into separate
`.claude/rules/` files. This keeps CLAUDE.md focused and under 200 lines while still giving Claude
Code access to the full detail via the `@` import mechanism.

The plugin coexistence note in CLAUDE.md is the key design decision: rather than replacing the
existing lob-specific skills (`/issue-implement`, `/pr-review`) with conductor/agent-teams
equivalents, both systems coexist. The table in CLAUDE.md makes the boundary explicit — native
skills are the primary workflow; plugin commands are available as supplementary tools. This was a
deliberate choice to preserve the investment in the battle-tested SDLC pipeline while still making
the wshobson/agents capabilities available. The actual plugin installation (`/plugin marketplace add
wshobson/agents` + `/plugin install conductor` + `/plugin install agent-teams`) is a one-time
Claude Code CLI operation documented in CLAUDE.md, not automated in code.

Three `settings.local.json` references in agent design docs and templates were also corrected to
point to `settings.json` — the shared file where `Skill()` and `Agent()` permissions now live.
The `tutorial-new-agent.md` ecosystem doc had five such stale references that were updated.

Deferred: removing the custom Node.js orchestrator (`server/src/orchestrator/`) and migrating the
6-HCP issue-implement workflow to conductor tracks. That is a larger undertaking requiring a
dedicated design pass and is tracked as a follow-up issue.

## 10:19 — PR: Migrate completely to wshobson/agents as primary workflow layer

This PR completes the work deferred from the previous session: fully retiring the hand-rolled
SDLC infrastructure and designating the wshobson/agents plugin layer as the single source of
workflow orchestration. The Node.js orchestrator (`server/src/orchestrator/`), all four
workflow JSON definitions (`docs/workflows/`), and three custom agents (code-review,
issue-intake, project-manager) with their nine command skill files are all deleted. The result
is a net reduction of nearly five thousand lines with a leaner, more maintainable ecosystem.

The key design decision was choosing which assets to delete versus which to retain. Agents and
skills that have a direct plugin equivalent (issue-intake → `/conductor:new-track`,
issue-implement → `/conductor:implement`, pr-review → `/team-review`) were removed. Agents and
skills with no plugin equivalent — `devops`, `domain-expert`, and the PR/issue lifecycle skills
(`/pr-create`, `/pr-merge`, `/issue-close`) — were kept. This boundary is now the stable
extension surface: conductor and agent-teams handle SDLC sequencing; lob-specific extensions
handle domain knowledge and devops automation.

A migration guide was created at `docs/migration-wshobson-agents.md` documenting the old-to-new
command mapping. This is the reference for anyone who has muscle memory for the old `/issue-implement`
workflow. The `rules-lawyer` agent was also renamed to `domain-expert` throughout — the old name
was a Claude Code agent type name (`rules-lawyer`) from a prior convention; the new name better
describes the agent's role and matches how it is referred to in prose everywhere.

The six ecosystem reference files in `docs/claude-ecosystem/` were rebuilt to match the
post-migration surface: two agents, eleven skills, conductor/agent-teams as the orchestration
layer. Four narrative files (blog, tutorial-new-agent, tutorial-modify-agent, network-diagram)
were deleted as they described the old hand-rolled system in detail and had no equivalent value
for the plugin-based setup. `docs/ecosystem-design.md` was rebuilt from scratch to remove all
references to the deleted five-agent structure and old skill set.

Deferred: no game logic changes on this branch. Issue #81 (wedge elevation neighbor mirroring)
remains open and is the next implementation target.
