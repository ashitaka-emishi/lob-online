## 13:43 — DevOps agent and skills

Added a `devops` Claude Code agent and four companion skills (`/build`, `/start`, `/stop`,
`/test`) to automate the core development operations for lob-online.

The design started with a requirements session that produced `docs/devops-agent-design.md` — a
spec covering the four operations, directory layout for logs and test-result snapshots, and a
full implementation checklist. Writing the design document first rather than diving straight into
the skill files was deliberate: it forced a concrete decision about sequencing rules (build →
start → test), port numbers, log file paths, and flake-tracking mechanics before any files were
created. The design doc now serves as durable reference for future maintainers.

Each operation maps cleanly to an existing `npm run` script: `format`, `lint`, `build`, and
`test` at the root; `dev -w client` and `node server/src/server.js` for the runtime processes.
No new `package.json` entries were needed. The `start` skill deliberately uses `node` directly
(not `--watch`) so the background process is stable for automated test runs; hot-reload is
unsuitable when tests need a predictable server state.

The `test` skill runs Vitest twice — once with `--reporter=verbose` for human-readable log
output, and once with `--reporter=json` for machine-readable snapshots. The JSON snapshots are
retained in `test-results/` (gitignored) so that a subsequent run can diff pass/fail states and
flag flaky tests. Server log correlation is wired to grep `logs/server/server.log` for ERROR
lines when failures are present. Two open implementation details were noted in the design doc:
configuring Vitest to emit both reporters in a single pass (avoids the double-run), and deciding
on a log rotation strategy once `logs/` grows large.

The `test-results/` directory was added to `.gitignore`; the `logs/` pattern already covered
server and client log files. Prettier reformatted the two new agent files and the design doc
before checks passed.

## 14:26 — Project manager agent design document

Created `docs/project-manager-agent-design.md`, a specification for a `project-manager` Claude
Code agent that manages the lob-online SDLC using GitHub as the source of truth.

The project already has a `rules-lawyer` agent for game-mechanics arbitration and a `devops`
agent for the build/run/test cycle, but no mechanism for tracking planned work as GitHub issues
or for coordinating long-term scope against the HLD phases. The design doc fills that gap by
defining the full pipeline from raw idea to a filed, milestone-assigned GitHub issue.

The central design decision was making the `rules-lawyer` consultation a hard gate rather than an
optional step. Any issue touching game mechanics, movement, LOS, combat, morale, orders,
artillery, or data models must be routed through `rules-lawyer` before filing. This prevents
acceptance criteria from being drafted without grounding in the actual rules, which would
otherwise require a second pass to correct after implementation began. The trigger list is spelled
out explicitly in both the Overview section and the `issue-intake` skill steps so neither the
agent nor a human reviewer needs to make a judgment call about what counts as "game logic."

The issue template was designed around a single question: can an AI coding agent implement this
ticket without asking follow-up questions? That required eight named fields — title, description,
acceptance criteria, files to create/modify, tests required, rules/data dependencies, depends-on,
and milestone — rather than free-form prose. The "Files to create/modify" and "Tests required"
fields in particular push the author to do the architectural thinking upfront, so the coding
agent receives explicit paths rather than having to infer them.

Milestones were mapped directly to the three HLD phases (`v1.0 — MVP`, `v2.0 — Enhanced`,
`v3.0 — Extended`) so that GitHub milestone completion state can be compared against the HLD
Implementation Status callout as a consistency check. The design notes that `gh milestone`
doesn't exist as a CLI subcommand and all milestone operations must go through `gh api`, which is
a non-obvious constraint worth capturing. The `settings.local.json` additions required for full
agent functionality were enumerated explicitly rather than left implicit.

template, the PR template, the `issue-intake` skill, the `project-manager` agent definition, the

The two GitHub template files (`ISSUE_TEMPLATE/feature.md` and `pull_request_template.md`) are

The `issue-intake` skill is structured as a numbered procedure rather than a free-form prompt so
that the agent cannot skip the rules-lawyer gate by accident. Step 3 names the trigger conditions
verbatim from the design document: game mechanics, movement, LOS, combat, morale, orders,
artillery, and data models. Repeating the list in the skill file means the agent doesn't need to
read the design doc to know when to consult `rules-lawyer`.

The `project-manager` agent definition keeps the long-term planning procedure and the `gh` command
examples inline, so the agent has executable guidance without reading any external spec. The
milestone creation commands use the `{owner}/{repo}` pattern documented in the design, with the
actual repo slug noted in the key-files section. `Skill(issue-intake)` was added to
`settings.local.json` alongside the five `gh` command patterns so the agent can invoke the skill
without a permissions prompt.

The three GitHub milestones (`v1.0 — MVP`, `v2.0 — Enhanced`, `v3.0 — Extended`) were created via
`gh api POST /repos/ashitaka-emishi/lob-online/milestones` and are now live at milestones 1–3 on
the repository. They are the first structured project-management artifacts in the GitHub repo and
make it possible to assign issues to phases going forward.

## 15:06 — Code review agent design and skills

Created `docs/code-review-agent-design.md` and implemented the four supporting files that turn
the design into an operational agent: `.claude/commands/review.md`, `.claude/commands/assess.md`,
`.claude/agents/code-review.md`, and an update to `CLAUDE.md`.

The agent completes the tooling triad alongside `devops` (build/run/test) and `project-manager`
(SDLC/issue management). Its domain is the quality gate: checking that code reaching a PR meets
the project's formatting, lint, coverage, and cleanliness standards. The two-mode design —
`/review` for narrow PR diffs and `/assess` for broad codebase sweeps — was deliberate. PR review
needs to be fast and focused so it doesn't become a friction point in the merge workflow; a full
`/assess` run is expected to be slower and is suited for periodic audits or pre-refactor planning
rather than every commit.

A key constraint is the "build first, test before analysing" rule that both skills enforce. There
is no value in running static analysis on code that fails to compile or has broken tests — the
findings would be noise against a moving target. Both skills abort immediately if build or tests
fail, which also ensures the coverage report used by `/assess` reflects a clean baseline. Coverage
gaps are treated as findings (not blockers) so that a project with known under-covered areas can
still receive a useful quality report.

Log output goes to `logs/review/` (under the existing gitignored `logs/` root) with per-run
timestamps. The `/review` skill posts findings as a PR comment via `gh pr comment`, closing the
loop without requiring the reviewer to open a separate log file. The `/assess` skill writes a
summary to `docs/assess-YYYY-MM-DD.md` rather than committing it automatically, giving the
developer a chance to review before deciding whether the findings document belongs in the repo.

Prettier reformatted `code-review.md` and `assess.md` during the build check — the list-heavy
markdown in those files triggered a few blank-line insertions. No source code was touched;
all changes are documentation and agent configuration files.
