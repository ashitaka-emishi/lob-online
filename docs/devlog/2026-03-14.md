# Devlog — 2026-03-14

## 13:06 — Rename business-analyst agent to rules-lawyer

Renamed `.claude/agents/business-analyst.md` to `rules-lawyer` and extended its role from
passive assistant to active arbiter. The description was updated to reflect the new mandate:
when source documents disagree or a rule is ambiguous, the agent renders an authoritative
ruling by consulting all relevant sources in priority order, quoting conflicting passages, and
explaining its reasoning. A four-level priority hierarchy (SM_ERRATA → LOB_GAME_UPDATES →
SM_RULES → LOB_RULES/LOB_CHARTS) was added to the agent definition to make conflict resolution
deterministic rather than ad hoc.

## 13:43 — DevOps agent and skills

Added a `devops` Claude Code agent and four companion skills (`/build`, `/start`, `/stop`,
`/test`) to automate the core development operations for lob-online.

The design started with a requirements session that produced a spec covering the four operations,
directory layout for logs and test-result snapshots, and a full implementation checklist. Writing
the design document first rather than diving straight into the skill files was deliberate: it
forced concrete decisions about sequencing rules (build → start → test), port numbers, log file
paths, and flake-tracking mechanics before any files were created.

Each operation maps cleanly to an existing `npm run` script: `format`, `lint`, `build`, and
`test` at the root; `dev -w client` and `node server/src/server.js` for the runtime processes.
No new `package.json` entries were needed. The `start` skill deliberately uses `node` directly
(not `--watch`) so the background process is stable for automated test runs; hot-reload is
unsuitable when tests need a predictable server state.

The `test` skill runs Vitest twice — once with `--reporter=verbose` for human-readable log
output, and once with `--reporter=json` for machine-readable snapshots. The JSON snapshots are
retained in `test-results/` (gitignored) so that a subsequent run can diff pass/fail states and
flag flaky tests. Server log correlation is wired to grep `logs/server/server.log` for ERROR
lines when failures are present. Two open implementation details were noted in the design doc:
configuring Vitest to emit both reporters in a single pass, and deciding on a log rotation
strategy once `logs/` grows large.

## 14:26 — Project manager agent design and implementation

Designed and implemented the `project-manager` agent: the GitHub issue template, PR template,
`/issue-intake` skill, agent definition, `settings.local.json` permission additions, and the
three GitHub milestones.

The project already had a `rules-lawyer` agent for game-mechanics arbitration and a `devops`
agent for the build/run/test cycle, but no mechanism for tracking planned work as GitHub issues
or for coordinating long-term scope against the HLD phases. The agent fills that gap by defining
the full pipeline from raw idea to a filed, milestone-assigned GitHub issue.

The central design decision was making `rules-lawyer` consultation a hard gate rather than an
optional step. Any issue touching game mechanics, movement, LOS, combat, morale, orders,
artillery, or data models must be routed through `rules-lawyer` before filing. The trigger list
is spelled out explicitly in both the agent definition and the `issue-intake` skill steps so
neither the agent nor a human reviewer needs to make a judgment call about what counts as
"game logic."

The issue template was designed around a single question: can an AI coding agent implement this
ticket without asking follow-up questions? That required eight named fields — title, description,
acceptance criteria, files to create/modify, tests required, rules/data dependencies, depends-on,
and milestone. The "Files to create/modify" and "Tests required" fields push the author to do
architectural thinking upfront, so the coding agent receives explicit paths rather than having to
infer them.

The `issue-intake` skill is structured as a numbered procedure so the agent cannot skip the
rules-lawyer gate by accident. The `project-manager` agent definition keeps the long-term
planning procedure and `gh` command examples inline so the agent has executable guidance without
reading an external spec. Milestones were mapped to the three HLD phases (`v1.0 — MVP`,
`v2.0 — Enhanced`, `v3.0 — Extended`) and created via `gh api` — `gh milestone` does not exist
as a CLI subcommand, which is a non-obvious constraint worth capturing in the design. The three
milestones are now live at milestones 1–3 on the repository.

## 15:06 — Code review agent design and skills

Created the `code-review` agent and implemented two skills: `/review` for narrow PR diff
inspection and `/assess` for broad full-codebase examination.

The agent completes the tooling triad alongside `devops` (build/run/test) and `project-manager`
(SDLC/issue management). The two-mode design was deliberate: PR review needs to be fast and
focused so it doesn't become a friction point in the merge workflow; a full `/assess` run is
suited for periodic audits or pre-refactor planning rather than every commit.

A key constraint is the "build first, test before analysing" rule both skills enforce. There is
no value in running static analysis on code that fails to compile or has broken tests — the
findings would be noise against a moving target. Both skills abort immediately if build or tests
fail, ensuring the coverage report used by `/assess` reflects a clean baseline. Coverage gaps are
treated as findings (not blockers) so that a project with known under-covered areas can still
receive a useful quality report.

Log output goes to `logs/review/` (under the existing gitignored `logs/` root) with per-run
timestamps. The `/review` skill posts findings as a PR comment via `gh pr comment`, closing the
loop without requiring the reviewer to open a separate log file. The `/assess` skill writes a
summary to `docs/assess-YYYY-MM-DD.md` rather than committing it automatically, giving the
developer a chance to review before deciding whether the findings document belongs in the repo.

rule rather than just the list. The agent definition itself required no changes; both new
documents were purely additive.

## 18:18 — Agent documentation refactor: consolidated docs/agents/ structure

Refactored the four agent companion documents from a flat scatter of eight files in `docs/` into
a structured `docs/agents/<name>/` hierarchy, and introduced three new skills to manage the
structure going forward.

The motivation was drift accumulating across agents: the devops and code-review prompt files were
raw bullet notes with no standard outline, the project-manager prompt file had a reversed filename
(`project-manager-design-agent-prompt.md`), the agent definition files were missing uniform
"What the Agent Does NOT Do" and "Key Files" sections, and design documents mixed numbered and
unnumbered headings with no consistent pattern. As more agents are added this would only compound.
The refactor establishes `docs/agents/PROMPT_TEMPLATE.md` and `docs/agents/DESIGN_TEMPLATE.md` as
the canonical structures for new agents, and `docs/agents/README.md` as the entry point for
contributors working with the agent layer.

The most important architectural decision was making `## 4. Agent Definition` in each `design.md`
the single source of truth for the corresponding `.claude/agents/<name>.md` file. That heading
is now a contract: the three new skills (`/standardize-agents`, `/regenerate-agents`,
`/sync-agents`) all locate content by searching for that exact string. Renaming it would break the
skills. The `DESIGN_TEMPLATE.md` carries a comment warning future editors not to rename it.

The three skills divide responsibilities cleanly: `/regenerate-agents` is write-only (rebuilds
agent files from `design.md` section 4 with no questions asked), `/sync-agents` is read-only (a
drift detector that never writes), and `/standardize-agents` is a heavier tool that normalizes
`prompt.md` files against the template and then cascades changes through `design.md` and the agent
file. The read-only `/sync-agents` was run immediately after creation and reported all four agents
in sync, confirming that the new `design.md` section 4 blocks and the existing agent file YAML
frontmatters were consistent.

The eight old `docs/` files were deleted outright rather than archived. The content is preserved
verbatim in the new `docs/agents/<name>/` files — with added structure — so no information was
lost. Four path references in `CLAUDE.md` and three Key Files paths in agent definition files were
updated to the new locations. The devops agent file received a new "What the Agent Does NOT Do"
and "Key Files" section that was missing from the original. Build, lint, and format checks all
passed with no source code changes; the refactor is entirely documentation and agent tooling.
