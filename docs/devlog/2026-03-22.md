# 2026-03-22

## 19:10 — Fix hex edge geographic labels (DIR_TO_CORNERS corner-order bug)

Fixed a root-cause bug where the hover tooltip in the map editor displayed geographic edge labels
that were off by one face (e.g., showing "NW" for the top flat edge when it should show "W" for
South Mountain's northOffset=3 calibration). The fix also corrected the `compassLabel` utility in
both `utils/compass.js` and `formulas/compass.js`, which had been using a 6-label lookup that
dropped "E" and "W" entirely for odd northOffset values.

The root cause was in `hexGeometry.js`: `DIR_TO_CORNERS` assumed honeycomb-grid's FLAT orientation
starts corners at the East vertex (0°), but the library actually starts at the NE vertex (−30° from
East). Every corner index in the mapping was off by one, so the segment the code called "N" was
actually the NW diagonal edge, and the segment called "NE" was actually the top flat N edge. Because
`findNearestEdge` and `edgeLine20_80` both used the same wrong table, the fuchsia snap-highlight
appeared in the visually correct place (consistently wrong in both read and write directions), which
masked the bug. The symptom only surfaced when geographic labels were applied on top: the geometric
"NE" direction mapped to geographic "NW" for northOffset=3 instead of the correct "W".

The fix corrected `DIR_TO_CORNERS` to `{N:[5,0], NE:[0,1], SE:[1,2], S:[2,3], SW:[3,4], NW:[4,5]}`
matching the actual honeycomb-grid source (`ut` function: corners NE→E→SE→SW→W→NW clockwise from
−30°). Two secondary fixes followed: `bottomCY` in `HexMapOverlay.vue` was updated to use corners
2 and 3 (both SE/SW, at the same y-position) rather than 1 and 2 (E and SE, which had different
y-coordinates and placed elevation labels about halfway up the lower half of the hex instead of at
the bottom edge). The wedge polygon rendering was also corrected with an index offset of `(wi+5)%6`
so face-index 0 (N direction) maps to wedge polygon 5 (the NW→NE triangle that faces the top flat
edge).

All test fixtures in `hexGeometry.test.js` were updated to match the actual corner order. Since
the six corner coordinate values are the same set regardless of starting point, the midpoint-based
`findNearestEdge` tests passed without value changes; only the `wedgePolygonPoints` string-equality
tests and the CORNERS fixture comment required updates. The `CalibrationControls` test was also
corrected to use the full 8-direction compass set (including "E" and "W") since odd northOffset
values legitimately produce those labels after the `compassLabel` fix. 812 tests pass.

## 19:11 — PR: Edge tools bundle + hex corner-order fix

This PR bundles two related bodies of work. The primary deliverable is the edge tool panel suite
for the map editor: `RoadToolPanel` (road, trail, pike with bridge sub-control and right-click
erase), `StreamWallToolPanel` (stream, stone wall with ford sub-control), and `ContourToolPanel`
(contour lines). These panels follow the shared composable architecture established in prior work —
`useEdgePaintTool` handles drag-trace painting, `useClickHexside` handles single-click
interaction, and each panel emits its overlay config so `MapEditorView` can merge it into the
unified `overlayConfig` prop passed to `HexMapOverlay`. The elevation formula module
(`formulas/elevation.js`) was also extracted here, providing `elevationTintPalette` and
`tintForLevel` as pure functions testable outside the component tree.

The secondary deliverable is a root-cause fix for the hex edge geographic label system. The
`DIR_TO_CORNERS` mapping in `hexGeometry.js` had corner indices that were off by one because
honeycomb-grid's FLAT orientation starts its corner array at the NE vertex (−30° from East), not
the East vertex (0°) as the code assumed. This caused the hover tooltip to show "NW" for the top
flat edge instead of "W" when northOffset=3 (South Mountain default). The bug was masked by
the fact that `findNearestEdge`, `edgeLine20_80`, and the fuchsia snap-highlight all used the same
wrong table, so they remained mutually consistent and the visual snap appeared correct — the error
only became visible when geographic labels were layered on top. The `compassLabel` utility in both
`utils/` and `formulas/` was also fixed to use the COMPASS_12 ring (same as `getEdgeLabels`)
rather than a 6-label array that silently dropped "E" and "W" for odd northOffset values.

Three secondary fixes rode along with the corner-order correction: `bottomCY` in
`HexMapOverlay.vue` now uses corners 2+3 (SE and SW, both at the bottom flat edge) rather than
corners 1+2 (E and SE, which averaged to mid-lower hex), so elevation labels render at the correct
vertical position. The wedge polygon rendering uses `(wi+5)%6` to align face-index 0 (N direction)
with wedge polygon 5 (the NW→NE triangle facing the top flat edge). Test fixtures in
`hexGeometry.test.js` and `CalibrationControls.test.js` were updated accordingly.

Explicitly deferred: the "8.04 SW draws 7.04 NE / 7.05 SW" original edge-placement complaint
should be revisited now that edge labels are geometrically correct — it may resolve itself or may
need a separate canonical-ownership investigation. Wedge elevation painting UI (per-wedge click
editing) was not part of this track and remains unimplemented; the wedge render fix is
forward-compatible when that editing is added.
