# 2026-03-22

## 19:10 — Fix hex edge geographic labels (DIR_TO_CORNERS corner-order bug)

Fixed a root-cause bug where the hover tooltip in the map editor displayed geographic edge labels
that were off by one face (e.g., showing "NW" for the top flat edge when it should show "W" for
South Mountain's northOffset=3 calibration). The fix also corrected the `compassLabel` utility in
both `utils/compass.js` and `formulas/compass.js`, which had been using a 6-label lookup that
dropped "E" and "W" entirely for odd northOffset values.

The root cause was in `hexGeometry.js`: `DIR_TO_CORNERS` assumed honeycomb-grid's FLAT orientation
starts corners at the East vertex (0°), but the library actually starts at the NE vertex (−30° from
East). Every corner index in the mapping was off by one, so the segment the code called "N" was
actually the NW diagonal edge, and the segment called "NE" was actually the top flat N edge. Because
`findNearestEdge` and `edgeLine20_80` both used the same wrong table, the fuchsia snap-highlight
appeared in the visually correct place (consistently wrong in both read and write directions), which
masked the bug. The symptom only surfaced when geographic labels were applied on top: the geometric
"NE" direction mapped to geographic "NW" for northOffset=3 instead of the correct "W".

The fix corrected `DIR_TO_CORNERS` to `{N:[5,0], NE:[0,1], SE:[1,2], S:[2,3], SW:[3,4], NW:[4,5]}`
matching the actual honeycomb-grid source (`ut` function: corners NE→E→SE→SW→W→NW clockwise from
−30°). Two secondary fixes followed: `bottomCY` in `HexMapOverlay.vue` was updated to use corners
2 and 3 (both SE/SW, at the same y-position) rather than 1 and 2 (E and SE, which had different
y-coordinates and placed elevation labels about halfway up the lower half of the hex instead of at
the bottom edge). The wedge polygon rendering was also corrected with an index offset of `(wi+5)%6`
so face-index 0 (N direction) maps to wedge polygon 5 (the NW→NE triangle that faces the top flat
edge).

All test fixtures in `hexGeometry.test.js` were updated to match the actual corner order. Since
the six corner coordinate values are the same set regardless of starting point, the midpoint-based
`findNearestEdge` tests passed without value changes; only the `wedgePolygonPoints` string-equality
tests and the CORNERS fixture comment required updates. The `CalibrationControls` test was also
corrected to use the full 8-direction compass set (including "E" and "W") since odd northOffset
values legitimately produce those labels after the `compassLabel` fix. 812 tests pass.

## 20:54 — PR: HexMapOverlay performance quick wins bundle

Four targeted performance fixes in `HexMapOverlay` and its rendering path, addressing tech-debt items #159, #160, #163, and #164. All were deferred from PR #158 and share a common theme: unnecessary per-frame work on hot rendering paths. The track was identified as the highest bang-for-buck debt cluster because all four items live in the same file, follow patterns already established in the codebase, and compound with map density.

The highest-impact fix is the `requestAnimationFrame` gate on the edge-hover path (#160). Previously, every raw `mousemove` event directly wrote to the `hoverInfo` reactive ref, triggering a Vue re-render on every pointer event. The fix adds a `_edgeHoverRafPending` flag (a plain module-level `let`, not a ref) that coalesces rapid mousemoves into at most one update per animation frame — the same pattern documented from PR #88 for the hex-hover path. This is the correct throttle mechanism for SVG interaction code because the browser already limits repaints to 60fps; without the gate, CPU time is spent computing and diffing renders that will never be painted. Vitest tests mock `requestAnimationFrame` globally via `vi.stubGlobal` to assert that two rapid mousemoves schedule exactly one rAF callback.

The `findNearestEdge` deduplication (#159) eliminated the second O(cells×6) edge traversal that was happening on every mousemove frame. Previously `onSvgMouseMove` called `findNearestEdge` twice — once with threshold=999 (always find nearest, for tooltip) and once with threshold=6 (snap threshold, for fuchsia highlight). To enable single-call deduplication without changing the function signature contract, `dist` was added to the return object. Callers that use `result.hexId` and `result.dir` are unaffected; the snap result is now derived by checking `nearest.dist <= 6` instead of a second full traversal. Existing `findNearestEdge` tests were updated from `toEqual({hexId, dir})` to `expect.objectContaining({hexId, dir})` to remain resilient to future additions to the return shape.

The two Vue-layer fixes (#163, #164) address per-render allocation. For #164, `wedgePolygonPoints` was moved from an inline template call (one call per wedge per render, allocating a 6-element string array each time) into the `gridData` computed, where the result is stored as `cell.wedgePoints`. The template now reads the stable pre-computed string. For #163, the inline `.filter()` in the EdgeLineLayer `v-for` was eliminated by adding a `cellsForEdges` computed that pre-filters edge features per cell per canonical face per group. This prevents per-render array creation and restores Vue's ability to diff the list against a stable array. Both `gridData` and `cellsForEdges` are reactive computeds that only invalidate when calibration or overlayConfig changes — not on every render.

## 19:11 — PR: Edge tools bundle + hex corner-order fix

This PR bundles two related bodies of work. The primary deliverable is the edge tool panel suite
for the map editor: `RoadToolPanel` (road, trail, pike with bridge sub-control and right-click
erase), `StreamWallToolPanel` (stream, stone wall with ford sub-control), and `ContourToolPanel`
(contour lines). These panels follow the shared composable architecture established in prior work —
`useEdgePaintTool` handles drag-trace painting, `useClickHexside` handles single-click
interaction, and each panel emits its overlay config so `MapEditorView` can merge it into the
unified `overlayConfig` prop passed to `HexMapOverlay`. The elevation formula module
(`formulas/elevation.js`) was also extracted here, providing `elevationTintPalette` and
`tintForLevel` as pure functions testable outside the component tree.

The secondary deliverable is a root-cause fix for the hex edge geographic label system. The
`DIR_TO_CORNERS` mapping in `hexGeometry.js` had corner indices that were off by one because
honeycomb-grid's FLAT orientation starts its corner array at the NE vertex (−30° from East), not
the East vertex (0°) as the code assumed. This caused the hover tooltip to show "NW" for the top
flat edge instead of "W" when northOffset=3 (South Mountain default). The bug was masked by
the fact that `findNearestEdge`, `edgeLine20_80`, and the fuchsia snap-highlight all used the same
wrong table, so they remained mutually consistent and the visual snap appeared correct — the error
only became visible when geographic labels were layered on top. The `compassLabel` utility in both
`utils/` and `formulas/` was also fixed to use the COMPASS_12 ring (same as `getEdgeLabels`)
rather than a 6-label array that silently dropped "E" and "W" for odd northOffset values.

Three secondary fixes rode along with the corner-order correction: `bottomCY` in
`HexMapOverlay.vue` now uses corners 2+3 (SE and SW, both at the bottom flat edge) rather than
corners 1+2 (E and SE, which averaged to mid-lower hex), so elevation labels render at the correct
vertical position. The wedge polygon rendering uses `(wi+5)%6` to align face-index 0 (N direction)
with wedge polygon 5 (the NW→NE triangle facing the top flat edge). Test fixtures in
`hexGeometry.test.js` and `CalibrationControls.test.js` were updated accordingly.

Explicitly deferred: the "8.04 SW draws 7.04 NE / 7.05 SW" original edge-placement complaint
should be revisited now that edge labels are geometrically correct — it may resolve itself or may
need a separate canonical-ownership investigation. Wedge elevation painting UI (per-wedge click
editing) was not part of this track and remains unimplemented; the wedge render fix is
forward-compatible when that editing is added.

## 21:54 — PR: Rendering architecture — overlayConfig ownership cleanup

This PR resolves three deferred architecture debt items (#152, #153, #167) that collectively
violated the tool-owns-its-overlays principle established in earlier PRs. The deliverable is a
clean separation between `HexMapOverlay`'s rendering API (what to show) and the editor's
interaction infrastructure (how to interact with it). The deferral condition for #153 — "wait
until tool panels exist to own it" — was met after PR #158 merged the edge tool panels, making
this the natural follow-on track.

The core Phase 2 change removes eight legacy flat props from `HexMapOverlay.defineProps`:
`vpHexIds`, `selectedHexId`, `calibrationMode`, `losHexA`, `losHexB`, `losPathHexes`,
`losBlockedHex`, and `seedHexIds`. All rendering state that those props carried now flows through
the unified `overlayConfig` object via five new keys: `selectedHex`, `calibration`, `los`,
`vpHighlight`, and `seedHighlight`. The ownership mechanism chosen is emits — four of six tool
panels already emitted `overlay-config` events, so extending the pattern was simpler than
introducing a new Pinia store. `MapEditorView` gains `_globalOverlayState()`, a function that
assembles the non-panel editor state (selected hex, LOS data, VP highlights, seed highlights,
calibration mode) and merges it into every overlayConfig branch so the 8 removed prop bindings
are replaced by a single `:overlay-config` attribute in the template.

Phase 1 established the design before writing any implementation code: reading the component
landscape to enumerate the 8 flat props, choosing emits as the ownership mechanism, creating
`client/src/utils/overlayConfig.js` as the canonical JSDoc type reference for the full
`OverlayConfig` shape (all 13 keys documented with types), and writing 8 intentionally-failing
test cases as the acceptance criteria for Phase 2. This TDD sequence made the implementation
straightforward — each failing test pointed to exactly one thing that needed to change in
`HexMapOverlay`.

Phase 3 moved the terrain-specific knowledge out of `MapEditorView` and into `TerrainToolPanel`.
`TerrainToolPanel` was the only panel that didn't yet emit `overlay-config`; it now imports
`TERRAIN_COLORS` from `feature-types.js`, defines its own `TERRAIN_ICON_MAP`, and emits a
`{ hexLabel, hexFill, hexIcon }` slice via `watch(ownOverlayConfig, emit, { immediate: true })`.
As a consequence, `MapEditorView` lost `TERRAIN_COLORS`, `hexFillFn`, `hexIconFn`,
`TERRAIN_ICON_MAP`, and the `layerFlags.terrain` / `layerFlags.elevation` flags — the base
overlay config is now strictly structural (grid labels, edge lines, wedge shading, slope arrows)
with no terrain-color or icon knowledge. The composable composition contract (#155) was
documented as a block comment at the top of `useEdgePaintTool.js`, covering how
`useEdgePaintTool`, `useClickHexside`, and `useHexPaintTool` compose, the event routing
contract, and the overlay config ownership contract.
