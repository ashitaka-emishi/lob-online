## 00:19 — Map editor server persistence: versioned backups, offline fallback, and push/pull sync UX

This session hardened the map editor's server persistence layer, which previously did a bare `writeFileSync` overwrite of `map.json` with no backup and no recovery path. The implementation covers three related concerns: server-side safety (versioned backups before every overwrite), client resilience (offline fallback to localStorage draft), and sync UX (explicit Push/Pull buttons with confirmation dialogs when data freshness is ambiguous).

On the server, every successful `PUT /api/tools/map-editor/data` now copies the current `map.json` to `data/scenarios/south-mountain/backups/map-YYYY-MM-DDTHH-mm-ss-mmmZ.json` before writing the new content. If the backup write fails, the handler returns 500 immediately without touching `map.json`—the "never silently half-save" rule. The backups directory is created automatically via `mkdirSync({ recursive: true })` and trimmed to the 20 most recent entries after each save. The server also injects an authoritative `_savedAt: Date.now()` timestamp into every saved document and returns it in the response body so the client can stay in sync. `MapSchema` was updated to include `_savedAt: z.number().optional()` so Zod's strip mode doesn't discard the field on round-trip.

On the client, `fetchMapData` now catches network errors and attempts a localStorage fallback before reporting a fetch error. If a scenario-namespaced draft exists (`lob-map-editor-mapdata-south-mountain-v2`), the editor loads it, sets `isOffline = true`, and shows a persistent orange "Server unreachable — working from local draft" banner. The Push to Server button is disabled and relabeled "Offline" in this state. The old unnested `lob-map-editor-mapdata-v1` key is migrated to the new namespaced v2 key on mount (one-time, then deleted), making the storage layout scenario-aware for eventual multi-scenario support. The calibration key is unchanged.

The sync UX adds two new affordances. "Save" was renamed to "Push to Server" (label only; `.save-btn` class retained to avoid cascading test selector churn). Before executing a PUT, the client compares the server's `_savedAt` (from the last successful GET) against the local draft's timestamp; if the server copy is newer, a `<ConfirmDialog>` asks "Server data is newer. Overwrite?" A new "Pull from Server" button fetches the current server copy; if there are unsaved local changes, a second confirmation asks "Discard local changes and load server data?" Both dialogs are implemented as a new reusable `client/src/components/ConfirmDialog.vue` SFC styled to match the editor's dark theme.

Test coverage spans both layers. The server tests were extended with `mkdirSync`, `readdirSync`, and `unlinkSync` in the `vi.mock('fs', ...)` factory, and `vi.clearAllMocks()` was replaced with `vi.resetAllMocks()` in the `beforeEach` — a necessary fix because `clearAllMocks` only clears call history while `resetAllMocks` also resets `mockReturnValue` state, which was leaking from the GET-test setup into the PUT tests and incorrectly triggering the backup branch. New client tests cover offline fallback (draft found / not found), the "Offline" button label, and the Pull confirmation dialog when the editor is dirty.

## 04:00 — PR: Cypress E2E suite for map editor offline mode and push/pull sync

This PR closes Issue #12, adding the Cypress E2E infrastructure that was explicitly deferred when the map editor's push/pull sync was merged in PR #13. Unit tests mock `fs` and `fetch`, which validates logic in isolation but cannot verify the full browser→server→filesystem path — that the `PUT` actually writes a backup file to disk, that the offline banner appears when `fetch` genuinely fails, or that the Vue component state machine transitions are correctly driven by real DOM interactions. Cypress covers those gaps.

The test suite has three files. `sync.cy.js` (4 tests) covers the two push paths (happy path and overwrite-confirmation when server data is newer) and the two pull paths (happy path and discard-confirmation when there are unsaved changes). `offline.cy.js` (2 tests) covers the offline-with-draft case (editor loads the localStorage draft and shows the banner) and offline-without-draft (fetch error is surfaced to the user). `backups.cy.js` (1 stretch test) is the only test that hits the real server: it records the backup file count before a push, executes a real PUT, and asserts the count increased — this is the test that mocked `fs` fundamentally cannot provide.

A non-obvious implementation detail: the offline-with-draft test cannot use `cy.seedMapDraft` (which calls `cy.window().then(...)` post-visit) because `fetchMapData` runs in `onMounted` — by the time `seedMapDraft` executes, the catch block may have already read localStorage and found it empty. The fix is `cy.visit(url, { onBeforeLoad })`, which seeds localStorage before any page scripts run. The `cy.seedMapDraft` command is retained for future tests where timing is not a concern. The pull discard-confirm test sets `unsaved = true` via Vue 3's dev-mode `__vueParentComponent.setupState` — a deliberate coupling to the Vite dev server that is acceptable because E2E tests run only against dev.

The GitHub Actions workflow is `workflow_dispatch`-only and is not a required CI gate. The decision against making it a required check reflects the cost model: E2E tests need a running server and real filesystem access, which adds significant setup time to every PR. The suite is most valuable as a regression tool run on demand or before releases, not as a commit-gate where its failures would block unrelated work.

## 04:47 — PR: Add scenario editor dev tool and extend scenario.json with lighting and rules fields

This PR delivers the scenario editor dev tool (Issue #15), a companion to the map editor for maintaining the non-map fields of `scenario.json`. While the map editor handles terrain geometry, the scenario editor covers the temporal and rules-layer data that governs how the game unfolds: when daylight transitions to twilight and night, what visibility cap applies after dark, which initiative system is in play, and a handful of scenario-level rule toggles drawn from the SM scenario rules.

Seven new fields were added to `scenario.json` and made optional in `ScenarioSchema` to ensure backward compatibility with documents that predate this editor. `lightingSchedule` is an array of `{ startTurn, condition }` entries sorted by turn number; the three South Mountain entries (day at turn 1, twilight at turn 28, night at turn 31) were populated from SM rules. The remaining fields — `nightVisibilityCap`, `flukeStoppageGracePeriodTurns`, `initiativeSystem`, `looseCannon`, `lossRecovery`, and `randomEventsEnabled`/`randomEventsTiming` — are scalar rule flags. All new types have their own Zod sub-schemas (`LightingEntry`, `LossRecovery`) with enum validation for `condition` and `initiativeSystem`.

The server route (`scenarioEditor.js`) is an exact structural copy of `mapEditor.js`: GET reads `scenario.json`, PUT validates with `ScenarioSchema`, backs up to `backups/scenario-<ts>.json` (max 20, auto-trimmed), injects `_savedAt`, and writes atomically. The route is registered under the same `MAP_EDITOR_ENABLED` guard in `server.js`, so both tools share a single environment flag. `ScenarioEditorView.vue` replicates the map editor's push/pull sync pattern — `_savedAt` staleness check before push, `ConfirmDialog` on overwrite and discard, offline banner on fetch failure — but the editing surface is three flat panels (Turn Structure, Lighting Schedule, Rules) rather than a canvas.

A non-obvious implementation detail in the test suite: `VALID_SCENARIO` is a module-level const shared across all tests. The Vue component receives the object returned by `json()`, which without deep cloning would be the same reference as `VALID_SCENARIO`. The `updateTurnStructure` handler then reassigns `scenarioData.value.turnStructure`, which — because `scenarioData.value` and `VALID_SCENARIO` share the same identity — mutates the module-level fixture. Later tests that used `VALID_SCENARIO` would see `firstTurn: '08:00'` instead of `'09:00'`, causing `gameDuration` to compute `12h` instead of `11h`. Similarly, `addLightingRow` mutated `VALID_SCENARIO.lightingSchedule`, leaving a pre-existing row in the fixture for the "does not add row when startTurn is 0" test. The fix was to change `mockFetch` to use `vi.fn().mockImplementation(...)` with `JSON.parse(JSON.stringify(data))` on every call, ensuring each test receives a fresh deep clone of the fixture.

The OOB editor (a third tool for editing unit statistics and brigade hierarchy) was deferred; it remains the last planned dev tool before the game logic phase. The `MAP_EDITOR_ENABLED` guard will extend to it as well.

## 05:38 — PR: Promote issue-intake to first-class agent with branch/PR workflow

This PR promotes `issue-intake` from a flat skill owned by the `project-manager` agent into a
standalone first-class agent with its own `design.md`, `prompt.md`, and `.claude/agents/`
definition. The motivation is that issue creation is itself a multi-step workflow that benefits
from the same branch/PR lifecycle discipline applied to code: every intake session should be
traceable, reviewable, and merged as a documented artifact rather than a one-off terminal
interaction.

The new agent orchestrates a 6-step workflow. Step 1 opens an `intake/{slug}` branch before
touching any file — a hard constraint that makes the scope of each session explicit and
reviewable. Steps 2a–2f cover the existing gather/classify/rules-gate/draft/milestone-check
loop, now extended with an explicit refinement iteration: the agent loops displaying the draft,
accepting feedback, and revising until the engineer signals ready. Step 3 is the first human
control point (HCP 1): `gh issue create` is never called without explicit approval. Step 4
commits the issue body as a `docs/intake/YYYY-MM-DD-{slug}.md` artifact — a versioned record
of what was filed and when. Step 5 runs `/pr-create` and waits for the engineer to say "merge"
(HCP 2). Step 6 runs `/pr-merge`, diagnosing and fixing any CI failures before retrying.

A key design constraint is that the `issue-intake` agent is explicitly code-free: it may only
commit to `docs/`, `.github/`, and `.claude/` paths. This keeps the intake branch lightweight
and prevents scope creep where issue drafting bleeds into implementation. The `docs/intake/`
directory is created as the canonical location for intake artifacts, giving future sessions a
consistent place to look for historical intake records.

The `project-manager` agent's §3 Skills section was updated to delegate to `issue-intake`
rather than own the skill directly. The distinction matters for routing: when an engineer
invokes `/issue-intake`, the `issue-intake` agent handles the full lifecycle, while
`project-manager` retains ownership of milestone planning, backlog auditing, and
`/issue-implement` orchestration. `docs/architecture.md` was updated with a new `issue-intake`
subgraph in the Mermaid agent-overview diagram, and the skill table now shows `issue-intake`
as the owning agent for `/issue-intake` with its calls to `/pr-create` and `/pr-merge` noted.

## 07:55 — PR: Orchestration agent with blocking-gate workflow engine

This PR delivers the orchestration subsystem proposed in issue #32 — a Node.js workflow engine that sequences agents and skills declaratively via `WorkflowDefinition` JSON, with blocking human gate checkpoints at each major transition. The runtime lives entirely in `server/src/orchestrator/` and is composed of three modules: `schemas.js` (Zod validation for all five domain types), `registry.js` (loads and resolves `.claude/agents/registry.json`), and `runtime.js` (step executor, JSONPath inputMap resolution, CLI readline gate handler, loop-back routing, and `WorkflowInstance` JSON persistence).

The central design decision was to use the Zod-parsed `defResult.data` (not the raw input object) throughout the runtime. This ensures that optional fields like `inputMap` get their Zod defaults (`{}`) applied before `Object.entries` is called on them — without this, any step definition missing an explicit `inputMap` key would crash at runtime. The JSONPath resolver is a bespoke subset supporting `$.stepId.field.subfield` notation, deliberately kept small rather than pulling in a full `jsonpath` library, since the v1 use cases only require simple dot-path traversal of prior step outputs.

Three workflow definitions were added under `docs/workflows/`: `sdlc-feature` (the primary new pipeline: draft-spec → gate → generate-code → review-code → gate → run-tests → gate → prepare-deploy → gate → deploy-exec), `issue-intake` (formalising the existing 6-step intake workflow as a version-controlled state machine), and `issue-implement` (the full ticket-to-merge pipeline with HCPs 1, 2, 2b, and 3). Each definition is paired with a `.states.md` containing a Mermaid `stateDiagram-v2` diagram and a gate checkpoint table describing loop-back risks.

The `.claude/agents/registry.json` file was created as the programmatic single source of truth for all 13 current agents and skills. Workflow definitions reference agents by `id` only — the runtime never spawns subprocesses or makes raw LLM calls, which keeps the dispatch interface injectable and testable. For tests, `vi.mock('./registry.js')` stubs `resolveAgent` to return a fixed manifest, decoupling runtime tests from the file system.

Three things were explicitly deferred to a future issue: an HTTP-based gate interface (the CLI readline handler is a swappable module by design), multi-agent parallelism, and workflow versioning/migration tooling. The `presentGate` function's readline path is not covered by the Vitest suite since testing interactive readline streams is non-trivial; coverage is 74% on `runtime.js`, clearing the 70% threshold on the other two source files at 100%.

## 23:33 — PR: Fix map editor UI bugs — hexsides, seed hex, northOffset, elevation, double-push

This session resolves twelve map editor UI defects accumulated since the last major editor pass. The work is entirely client-side with no server or data schema changes, touching `MapEditorView.vue`, `HexEditPanel.vue`, `HexMapOverlay.vue`, `EditorToolbar.vue`, `WedgeEditor.vue`, and `CalibrationControls.vue` plus their test suites.

Five bugs from the original ticket were straightforward: the Export button had been duplicated (it lived in both `EditorToolbar` and `MapEditorView`'s header — the toolbar copy was removed), the Hex Edit accordion header was dynamically showing the selected hex ID (changed to always read "Hex Edit"), clicking an uncharted hex showed no form (the `selectedHex` computed was returning `null` for hexes not yet in `hexes[]` — now returns a default `{ hex: id, terrain: 'unknown' }` object), hex ID labels were yellow and tiny (changed to dark blue `#00008b`, `0.78rem`, gated on `layers.grid` instead of `calibrationMode`), and even-column hex IDs had their row counter off by one (the formula `gridRows - hex.row` was adjusted to `gridRows - hex.row - 1` for even game columns, applied consistently in both `gridData` and `onSvgClick`).

The HCP 2a visual review surfaced six more issues. The legacy hexsides panel was removed from `HexEditPanel` — the six direction checkboxes had been superseded by the edge editor but never cleaned up. Seed hex was converted from a star-button (`★ Mark as Seed Hex`) to a checkbox grouped with VP Hex and Entry Hex, matching the UI convention for boolean hex properties. Seed hex state is managed in `MapEditorView` as a `Set<string>` ref (using set-replacement for Vue reactivity) and passed to `HexMapOverlay` as an array prop; seed hexes render with a purple `#cc44ee` border on the map. The double-push confirmation dialog bug — where clicking "Push to Server" a second time always triggered "Server data is newer" — was fixed by adding `unsaved.value &&` to the confirmation guard; the old condition was purely timestamp-based and triggered whenever `serverSavedAt > 0` and no local draft existed (always true after a successful push that clears the draft).

Two display improvements round out the session. A `northOffset` field (integer 0–5, default 0) was added to calibration. It controls how the `WedgeEditor`'s direction labels are rotated: `WEDGE_DIRS[(i + northOffset) % 6]` so that "N" appears at the correct visual position for maps where geographic north is not at the top of the screen (South Mountain has N to the right). The field is exposed in `CalibrationControls` and flows down through `HexEditPanel` to `WedgeEditor`. The elevation label was repositioned to the bottom-center of each hex using the midpoint of corners[1] and corners[2] (the SE and SW corners of a flat-top hex), increased in font size (8 → 11), recolored to dark green `#1a6b2a`, and given a white halo via `paint-order: stroke` for readability over varied terrain. Both changes avoid overlapping the hex ID grid label.
