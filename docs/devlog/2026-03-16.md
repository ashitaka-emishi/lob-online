## 00:19 — Map editor server persistence: versioned backups, offline fallback, and push/pull sync UX

This session hardened the map editor's server persistence layer, which previously did a bare `writeFileSync` overwrite of `map.json` with no backup and no recovery path. The implementation covers three related concerns: server-side safety (versioned backups before every overwrite), client resilience (offline fallback to localStorage draft), and sync UX (explicit Push/Pull buttons with confirmation dialogs when data freshness is ambiguous).

On the server, every successful `PUT /api/tools/map-editor/data` now copies the current `map.json` to `data/scenarios/south-mountain/backups/map-YYYY-MM-DDTHH-mm-ss-mmmZ.json` before writing the new content. If the backup write fails, the handler returns 500 immediately without touching `map.json`—the "never silently half-save" rule. The backups directory is created automatically via `mkdirSync({ recursive: true })` and trimmed to the 20 most recent entries after each save. The server also injects an authoritative `_savedAt: Date.now()` timestamp into every saved document and returns it in the response body so the client can stay in sync. `MapSchema` was updated to include `_savedAt: z.number().optional()` so Zod's strip mode doesn't discard the field on round-trip.

On the client, `fetchMapData` now catches network errors and attempts a localStorage fallback before reporting a fetch error. If a scenario-namespaced draft exists (`lob-map-editor-mapdata-south-mountain-v2`), the editor loads it, sets `isOffline = true`, and shows a persistent orange "Server unreachable — working from local draft" banner. The Push to Server button is disabled and relabeled "Offline" in this state. The old unnested `lob-map-editor-mapdata-v1` key is migrated to the new namespaced v2 key on mount (one-time, then deleted), making the storage layout scenario-aware for eventual multi-scenario support. The calibration key is unchanged.

The sync UX adds two new affordances. "Save" was renamed to "Push to Server" (label only; `.save-btn` class retained to avoid cascading test selector churn). Before executing a PUT, the client compares the server's `_savedAt` (from the last successful GET) against the local draft's timestamp; if the server copy is newer, a `<ConfirmDialog>` asks "Server data is newer. Overwrite?" A new "Pull from Server" button fetches the current server copy; if there are unsaved local changes, a second confirmation asks "Discard local changes and load server data?" Both dialogs are implemented as a new reusable `client/src/components/ConfirmDialog.vue` SFC styled to match the editor's dark theme.

Test coverage spans both layers. The server tests were extended with `mkdirSync`, `readdirSync`, and `unlinkSync` in the `vi.mock('fs', ...)` factory, and `vi.clearAllMocks()` was replaced with `vi.resetAllMocks()` in the `beforeEach` — a necessary fix because `clearAllMocks` only clears call history while `resetAllMocks` also resets `mockReturnValue` state, which was leaking from the GET-test setup into the PUT tests and incorrectly triggering the backup branch. New client tests cover offline fallback (draft found / not found), the "Offline" button label, and the Pull confirmation dialog when the editor is dirty.

## 00:35 — PR: feat(map-editor): versioned server backups, offline fallback, and push/pull sync UX

This PR packages the implementation session above into a reviewable unit. The single commit covers all three concern areas — server safety, client resilience, and sync UX — deliberately kept together because the `_savedAt` timestamp is the thread connecting all three: the server sets it authoritatively, the client uses it to detect staleness before a push, and the offline fallback preserves it in the localStorage draft so the comparison still works after reconnect.

The choice to keep the `.save-btn` CSS class on the renamed "Push to Server" button was intentional: renaming it would cascade through multiple test selectors with no functional gain. Similarly, the `ConfirmDialog` component was extracted into `client/src/components/` rather than inlined into the view because both push and pull confirmations share it — and it is small enough to be reused by future tools without over-engineering a prop API.

Deferred to Issue #12: Cypress E2E tests for the full browser→server→filesystem path. Unit tests mock `fs` and `fetch`, which is sufficient for the logic under test but cannot verify that an actual backup file appears on disk after a real save. That validation is inherently an E2E concern and is tracked separately.

## 02:29 — PR: Skill rename taxonomy, SKILL_TEMPLATE, architecture doc, and settings audit

This session completed two related plans: the skill naming taxonomy established in the previous session (renaming all skills to `<category>-<verb>`) and a follow-on infrastructure audit that fixed the remaining rough edges in the agent/skill layer. The two plans are committed together because the rename is only meaningful once the settings, README, and frontmatter are also consistent.

The naming rename touches 11 existing skills (e.g. `build` → `dev-build`, `review` → `pr-review`, `wrap-plan` → `plan-wrap`) and adds four new issue-workflow skills: `issue-start`, `issue-branch`, `issue-implement`, and `pr-merge`. The new skills complete the ticket-to-merge orchestration loop — `issue-implement` sequences all sub-skills with explicit human control points at plan approval (HCP 1), push (HCP 2), review triage (HCP 2b), and merge (HCP 3). This removes the previous gap where the workflow terminated at `pr-create` with no structured review or merge step.

Three legacy agent-maintenance skills (`agent-sync`, `agent-regenerate`, `agent-standardize`) were missing YAML frontmatter entirely, which prevented Claude Code from displaying them in the skill selector. Adding the frontmatter also made their `allowed-tools` declarations explicit — a constraint that previously lived only in prose. A minor factual error in `docs/agents/devops/prompt.md` was corrected: the `/dev-test` section previously said to invoke `/dev-start` if the server is not running, but the actual skill (correctly) stops and reports instead of auto-starting.

`docs/architecture.md` is a new document that answers the recurring question "should agents share skills?" with an explicit best-practice decision: skills are freely composable, analogous to shell scripts. The document includes three Mermaid diagrams — an agent overview showing skill ownership, a skill dependency graph, and an issue-to-merge sequence diagram with HCP annotations. The skill-sharing decision is worth documenting because the alternative (exclusive ownership) would require duplicating build/test logic in every skill that needs a quality gate, which is a DRY violation. `docs/agents/SKILL_TEMPLATE.md` was also created so future skills have a canonical starting point with authoring rules embedded as a removable comment block.

## 03:32 — PR: Add scenario editor and OOB editor stubs to HLD §10

This PR adds two planned dev-tool stubs to `docs/high-level-design.md` §10 (Data Preparation Tools) — a Scenario Editor and an OOB Editor — driven by a session where the issue-intake skill was being used to draft Issue #15 (scenario editor implementation). During that process it became clear the HLD only had an informal "unit data validation" bullet and no dedicated section for either tool, so the design doc needed updating before the implementation issue could be properly grounded.

The Scenario Editor stub documents the tool's purpose (editing `scenario.json` turn structure, lighting schedule, and rules settings), references the same `MAP_EDITOR_ENABLED` guard and push/pull sync pattern as the map editor, and records the planned routes. The OOB Editor stub is intentionally minimal — it names the data files it will target (`oob.json` and `leaders.json`) and the planned route, but makes no design commitments since the design hasn't happened yet. The instruction was explicit: "don't flush it out here, but add it to high level designs."

The §10 opening bullets were also renamed for precision: "Unit data validation" became "Order of battle editing" (with OOB Editor attribution), and "Scenario editing" became "Scenario configuration" (with Scenario Editor attribution). Each bullet now carries a status annotation (implemented / planned) so the section reads as a living inventory rather than a static wish list. The §3 route table and Implementation Status callout were updated to match.

The only deferred work is the actual design spec for the OOB editor — that will be a future session when the tool is ready to be built. The scenario editor design spec (`docs/scenario-editor-design.md`) is scoped inside Issue #15 itself as an acceptance criterion.
