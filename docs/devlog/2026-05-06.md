## 07:17 — PR: Pre-M5 Debt Sprint — Close 9 Score-≥-3 Findings from PR #328

This PR closes the nine deferred technical-debt findings (all scoring ≥ 3) that the `/team-review`
session surfaced on PR #328 before M5 work begins. The sprint was tracked as conductor track
`pre-m5-debt-sprint_20260505` and executed in five phases across two sessions.

**Phase 1 — Data Model Hardening** removed `'setup'` from the `phase` enum in `GameStateSchema`
(replacing it with `null` to represent the pre-game state) and introduced a `version` field for
optimistic concurrency control. `saveGame` now reads the stored version before writing, rejects a
mismatch with a `Version conflict` error, and increments before persisting — giving the server a
clean guard against lost-update races when multiple saves arrive for the same game. The version
starts at 0 in `initGameState` and reaches 1 after the first save.

**Phase 2 — Session Infrastructure** centralised all `req.session` reads and writes behind two
helpers: `setPlayerSession(req, gameId, side, token)` and `getPlayerSession(req)`. No route now
touches `req.session` properties directly, which makes the session shape a single-file contract.
The same phase swapped Express's default in-memory session store for
`better-sqlite3-session-store` backed by the game database, giving sessions 14-day persistence
across server restarts without a second database.

**Phase 3 — Authorization Gate** added the `requireSide` middleware, which reads
`getPlayerSession(req)` and returns 401 if the session is absent or if `session.gameId` doesn't
match the route's `:id` parameter. It is mounted only on `GET /api/v1/games/:id` — the one
endpoint that returns game state — leaving `POST /games` and `POST /games/:id/join` open so
unauthenticated players can create or join games.

**Phase 4 — Database Layer Cleanup** made three targeted changes. The `gameSqlite` module was
refactored to a `createStore(db)` factory that hoists all prepared statements at construction time,
enabling clean dependency injection in tests via an in-memory SQLite instance — eliminating the
`vi.resetModules()` hack. `joinGame` was rewritten as a single conditional `UPDATE ... WHERE id = ?
AND status = 'open'`, checking `result.changes === 0` to detect contention rather than a separate
`SELECT`; this closes the TOCTOU window where two concurrent join requests could both pass a
read-check and then both write. Finally, server startup code was wrapped in an exported
`startServer()` function so the module has no side-effects at import time, and a `SIGTERM` handler
was registered to close the database connection on graceful shutdown.

**Phase 5 — Storage Abstraction** created `server/src/store/index.js` as a thin re-export shim
that surfaces `loadGame`, `saveGame`, `createGame`, `joinGame`, `getGame`, and `listGames` from a
single stable path. All routes now import from `../store/index.js` and have no direct dependency
on `gameFile.js` or `gameSqlite.js`. The convenience delegates in `gameSqlite.js` remain as a
bridge for `server.js` infrastructure code (`initDb`, `getDb`) which is deliberately kept separate
from the route-facing API.

## 17:42 — PR: Auth Guard + sideToken Validation on Join Endpoint (#340)

This PR closes issue #340 (score-4 auth gap from the PR #339 team review) — the two security
findings deferred from the pre-M5 debt sprint because their scope exceeded the sprint budget.
The track was `auth-join-guard_20260506`, implemented in two phases following strict TDD.

**SEC-H1 — Store-layer token validation.** `gameSqlite.joinGame` now validates that `sideBToken`
is a non-empty UUID-format string before executing the `UPDATE`. The route has always passed
`randomUUID()` here, so this is a defensive layer rather than a fix for a reachable bug today —
but the store layer should enforce its own contract so callers can't accidentally write garbage
into `side_b_token` as the codebase grows. The UUID regex is defined at the top of `gameSqlite.js`
rather than imported from the route layer to keep the store self-contained. All existing store tests
were updated to pass valid UUIDs (they previously used short strings like `'tok-b'`), which had the
side effect of making those tests more realistic.

**ARCH-H2 — Session conflict guard on join route.** `POST /api/v1/games/:id/join` now calls
`getPlayerSession(req)` and returns 409 `{ error: 'Already in this game' }` if the caller's session
already holds the same `gameId`. This closes the worst-case exploit: without it, the union player
could POST to `/:id/join` from the same browser session, receiving a new confederate token and
overwriting their own session — effectively locking out both sides. The guard does not prevent a
completely unauthenticated newcomer from joining (that requires OAuth in M8); it only blocks the
same session from claiming both seats.

The implementation also surfaced and fixed a test isolation bug. `vi.clearAllMocks()` in
`beforeEach` clears call history but not `mockImplementation`. Tests 3 and 4 in the join-route
suite set `joinGame.mockImplementation(() => { throw GameNotOpenError })`, which was silently
bleeding into later tests — any test that called `joinGame` after test 4 without explicitly
resetting the mock would receive an unexpected 409. The fix is `joinGame.mockReset()` added to
`beforeEach`, which ensures each test starts with a clean no-op `joinGame`. This is a general
pattern worth applying whenever a test file uses `mockImplementation` and `clearAllMocks` together.
