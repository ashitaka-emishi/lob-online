## 01:46 — Map Editor Redesign: Accordion-Driven Tool Framework

Implemented the six-track map editor redesign plan, replacing the toolbar-based mode system with a property-centric accordion model. The core architectural shift is that tool activation is now driven by opening accordion panels in the right-hand panel, rather than clicking mode buttons in `EditorToolbar`. Each tool panel (Elevation, Terrain, Linear Feature, Wedge) activates its corresponding `editorMode` when opened and resets to `'select'` when closed or ESC is pressed. `EditorToolbar` was stripped down to layer toggles only.

Track 1 (calibration fixes) added cardinal direction labels (N, NE, SE, S, SW, NW) to the north-offset SVG picker that rotate as the `northOffset` value changes, so users can always see which position corresponds to "north." An Elevation Levels input was added directly to the calibration accordion panel (bound to `mapData.elevationSystem.elevationLevels`, persisted via the existing autosave path) rather than inside `CalibrationControls.vue` — this avoided breaking the existing test for "all numeric inputs disabled when locked," since elevation levels are independent of the grid lock. The calibration panel was also converted to the scrollable `accordion-hex` pattern.

Track 3 (elevation tool) replaced the old ±50-contour-interval click with ±1 per click and clamped values to `[0, elevationLevels − 1]`. Left-click selects a hex and increments; clicking the same hex again deselects without double-incrementing. Right-click (via a new `@contextmenu.prevent` handler in `HexMapOverlay`) decrements. The elevation layer is force-shown in `HexMapOverlay` whenever `editorMode === 'elevation'` regardless of the layer toggle. Bulk operations (clear all, raise all, lower all) were added to `ElevationToolPanel`. Track 4 added `TerrainToolPanel` with terrain icons (unicode symbols per terrain type) and a "Clear all terrain" bulk button; clicking a hex now applies terrain on click in addition to the existing drag-paint behavior on mouseenter.

Track 5 (linear feature draw tool) added drag-trace state to `HexMapOverlay`: mousedown starts a trace, mousemove collects the nearest edge at each rAF tick into a deduplicated set, mouseup emits `trace-complete` with the collected `{hexId, dir}` pairs. `MapEditorView` receives this event, stores the pending edges, shows a `ConfirmDialog`, and applies them on confirm. Traced edges are highlighted in cyan during the drag. Track 6 (wedge editor cleanup) removed the auto-derive button and WedgeEditor from `HexEditPanel` and moved them into a dedicated "Wedge Editor" accordion section in `MapEditorView`. The `elevationDerive.js` utility and `onDeriveWedges` were removed from the component tree (the utility itself is retained since it still has passing tests and may be useful in future game logic).

Three new component test files were added (`ElevationToolPanel.test.js`, `TerrainToolPanel.test.js`, `LinearFeaturePanel.test.js`) and affected existing tests updated — `EditorToolbar.test.js` rewritten for the simplified toolbar, `HexEditPanel.test.js` wedge/derive tests replaced with passthrough assertions. Final result: 415 tests passing, lint clean, coverage at 72.27% (above the 70% threshold).

## 01:55 — PR: Map Editor Accordion-Driven Tool Framework

This PR closes issue #94, delivering the full six-track map editor redesign. The motivating insight behind the redesign is that the old toolbar-button mode system required the user to mentally coordinate two separate UI areas (toolbar for mode, right panel for properties), whereas the accordion model makes the relationship explicit: the panel that shows a tool's controls is also the control that activates it. This eliminates a class of confusion where the mode was set but the wrong panel was open.

The most consequential architectural decision was routing `editorMode` through `togglePanel()` rather than as a separate piece of state. `TOOL_PANEL_MODES` maps panel names to modes, and closing any tool panel resets to `'select'`. This means there is only one place in `MapEditorView` where editorMode transitions happen, making the state machine easy to reason about. The ESC key handler closes the active tool panel via the same `togglePanel(null)` path, so keyboard and pointer interactions share one code path.

The drag-trace implementation in `HexMapOverlay` uses a `Set` keyed by `"hexId:dir"` for O(1) deduplication alongside an ordered array for emission — the same pattern used for the rAF-throttled spatial index added in PR #88. The trace is held in component state until `mouseup`, then emitted as `trace-complete`; `MapEditorView` stores the pending edges and shows a `ConfirmDialog` before writing to `mapData`. This two-phase commit pattern (collect → confirm → apply) avoids partial writes and lets users cancel a misdrawn path.

The elevation levels input was deliberately placed in `MapEditorView`'s calibration section rather than inside `CalibrationControls.vue`. `CalibrationControls` has a test asserting that all its number inputs are disabled when `calibration.locked === true`; elevation levels are independent of the grid lock and should remain editable after locking. Keeping the input in `MapEditorView` avoids entangling that concern with the lock logic and leaves the `CalibrationControls` test accurate. The `elevationDerive.js` utility was retained despite removing its UI surface — it still has passing tests and the derivation logic may be useful in future game rules code.

## 02:58 — Technical Debt Tracking System

Implemented a lightweight technical debt tracking system that hooks into the existing PR review workflow. The motivation was straightforward: with active code review cycles already happening via `/team-review` and the codebase about to grow into game logic and multiplayer phases, there was no structured way to measure or communicate the cost of deferred review findings. Without a register, debt accumulates invisibly until it becomes a crisis.

The design kept storage deliberately flat — markdown files in `docs/tech-debt/`, no database, no external tooling. The two key files are `docs/tech-debt/report.md` (the live aggregated register) and per-PR reports in `docs/tech-debt/reports/pr-{number}_{YYYY-MM-DD}.md`. The filename convention encodes the PR number for easy lookup. This was a conscious choice to keep the system auditable via git history and readable without any tooling.

The debt score scale (1–5 logarithmic) was the most significant design decision. A linear scale would compress low-end items together and make high-end items feel arbitrary. The logarithmic framing — where each step is roughly 2–3× the impact of the previous — matches how practitioners actually think about debt: a score-4 item really is qualitatively different from a score-2 item, not just incrementally worse. The scale guidance in `docs/tech-debt/README.md` was written to give concrete examples at each level so scoring stays consistent across reviewers over time.

The `/tech-debt-report` skill (`.claude/commands/tech-debt-report.md`) was structured as a 7-step linear workflow rather than a menu-driven tool. This was intentional: the skill needs to enforce the resolution standard (every finding is either fixed or deferred with a linked issue) before it generates any report. A menu would let users skip the enforcement step. The skill blocks on each deferred item until it has an issue number, score, and assessment — only then does it write the per-PR report and update the aggregated register.

Integration into the SDLC was kept minimal: a single row added to the workflow table in `CLAUDE.md` and `conductor/workflow.md`, a note added to the PR review sub-step in `.claude/rules/sdlc.md`, and a new "Technical Debt Tracking" subsection in §13 of `docs/high-level-design.md`. The goal was that `/tech-debt-report` feels like a natural sequel to `/team-review`, not a separate obligation to remember. No hooks or automation were added — the standard is enforced conversationally by the skill prompt itself.

## 03:44 — PR: feat(#92): Elevation System UI — baseElevation + elevationLevels

This PR closes issue #92, adding `baseElevation` and `elevationLevels` inputs to the map editor's calibration panel and replacing the hardcoded `.max(21)` elevation validation in `MapSchema` with a runtime check derived from the stored `elevationLevels` value. The motivation was straightforward: future scenarios will have different elevation ranges than South Mountain, and hard-coding 22 levels everywhere made it impossible to configure the map editor for a new scenario without editing JSON by hand.

The key design decision was where to add the two inputs. Issue #92 said "add to `CalibrationControls.vue`," but there was an existing `calibration-extra` section in `MapEditorView.vue` that already held an inline `elevationLevels` input from a previous implementation. Rather than maintain two parallel paths for the same data, the inline section was removed entirely and both fields moved into `CalibrationControls.vue` as a new `elevationSystem` prop. `CalibrationControls` now emits a distinct `elevation-system-change` event (rather than overloading `calibration-change`) because the two concerns belong to different top-level fields in `map.json` — `gridSpec` vs `elevationSystem`. Keeping the emits separate makes the data flow obvious and avoids a `calibration-change` handler that has to decide which fields belong to which section.

The elevation inputs are disabled both when the `elevationSystem` prop is absent (no map data loaded) and when `calibration.locked` is true. This matches the behavior of all other numeric inputs in the component and means the existing test — "all numeric inputs are disabled when calibration is locked" — still passes without modification. It also has an accidental product benefit: once a calibration is locked, the elevation configuration is frozen too, which is reasonable for a production-ready digitized map.

The schema fix in Phase 3 uses `z.superRefine` on `MapSchema` rather than parameterising `HexEntry` directly. The reason is that `HexEntry` is defined independently of `elevationSystem`, so the only place to cross-validate the two is at the `MapSchema` level. The `superRefine` defaults to `elevationLevels = 22` when `elevationSystem` is absent, preserving backward compatibility with any map data that doesn't have the field. SM `map.json` (22 levels, max elevation 21) continues to validate cleanly.

One deliberate non-decision: `wedgeElevations` still has its hardcoded `min(-21).max(21)` bounds in `HexEntry`. Those represent relative offsets from a hex's center elevation, not absolute levels, so they cannot be trivially derived from `elevationLevels` without knowing the hex's base elevation. That concern was explicitly called out of scope in the spec and left for a future track.

## 09:48 — PR: Hex Index Perf + Architecture Cleanup (Issue #93)

This PR addresses the deferred findings from the PR #91 team review that were filed as issue #93. The medium-priority finding (shared hex index) and two low-priority findings (computedEngineExport reactive cost, slope tie-break comment) were fully resolved. The remaining low-priority findings (double-write in autoDerive, onDeriveWedges bypass) turned out to be moot — those code paths were removed by the accordion-driven tool framework in PR #95, which moved wedge editing out of HexEditPanel into its own accordion section and eliminated the derive-wedges event entirely.

The central change is the `hexIndex` computed in `MapEditorView.vue`. Previously, ten separate interaction handlers — `selectedHex`, `adjustHexElevation`, `onHexClick`, `onHexMouseenter`, `onEdgeClick` (×2), `onHexUpdate`, `onWedgeUpdate`, `initWedgeElevations`, and `applyTrace` — each performed an O(n) `.find()` or `.findIndex()` scan over the full hex array on every interaction. With 2000+ hexes, that's up to 20,000 element inspections per click. The fix is a single `computed(() => new Map(hexes.map((h, i) => [h.hex, i])))` that is recomputed once when the hexes array changes and then provides O(1) lookups for all call sites.

One subtlety worth noting: `hexIndex` is a Vue computed, so it is lazily invalidated rather than eagerly recomputed. When `onHexUpdate` mutates `mapData.value.hexes[idx]`, Vue marks `hexIndex` dirty but doesn't recompute until the next access. This means the Map lookup within a single `onHexUpdate` call reads the valid pre-mutation index (correct), and the stale Map is only a concern if two handlers access `hexIndex` within the same synchronous tick with a mutation in between — which doesn't happen in this codebase. The `applyTrace` loop calls `onHexUpdate` inside a `for...of`, so `hexIndex` is re-accessed (and recomputed) on each iteration; this is O(n × k) for k trace edges but k is always small in practice.

`computedEngineExport` was converted from a Vue `computed()` to a plain `getEngineExport()` function. The computed was doing a deep recursive walk over all hexes and stripping private fields on every reactive change — a write to any hex would trigger it, even though its output is only consumed when the export overlay is open. Converting to a function means the walk only runs when the user explicitly opens the overlay (via `v-if="showExportOverlay"` which gates template rendering). The tie-break comment in `elevationDerive.js` was a one-liner, but an important one: without it, future readers might "fix" the `indexOf` to some other tie-break strategy and inadvertently change game behavior.

## 10:49 — PR: Trace Highlight Layer O(traceEdges) Render

The `traceEdgeSet` ref is retained in the component for now. It was used by the old template loop but is no longer referenced in the template after this change. ## 17:15 — PR: Fix LOS display off-by-one + map terrain digitization

This PR closes issue #114, which reported that the LOS highlight in the map editor lands one game-row off from the selected hex on odd-numbered game columns (7, 9, 11, …). The root cause turned out to be a formula in `HexMapOverlay.vue` that incorrectly subtracted 1 from the game row for even game columns: `gridRows - hex.row - (gameCol % 2 === 0 ? 1 : 0)`. The subtraction was well-intentioned — it tried to compensate for the visual stagger of alternating hex columns — but it was wrong. With honeycomb-grid's `offset: -1` (ODD_Q, flat-top), the library already encodes the column stagger into `hex.row` directly. No additional parity correction is needed, and applying one shifts the entire even-column label sequence down by one game row, making it inconsistent with `los.js` and `adjacentHexId`.

The inconsistency surfaced cleanly via a cross-system consistency check: `adjacentHexId('01.03', 'SE', {rows:3, cols:4})` returns `'02.03'`, but the old formula labeled that visual hex `'02.02'`. This meant LOS path highlights computed by `los.js` could land on visually incorrect cells when the path crossed even-numbered game columns, since the cell IDs in the overlay and the hex IDs produced by the LOS algorithm belonged to different coordinate frames. The fix — removing the erroneous subtraction — was applied in all three locations where the formula appeared: the `gridData` computed property (used for rendering cell IDs), `onSvgClick`, and `onSvgContextMenu`.

The pre-existing test that documented the wrong behavior (`even-column hex IDs display row value decremented by one`) was updated to reflect the correct expected values. A new red-phase test was committed before the fix, asserting consistency with `adjacentHexId`, which pinned the defect precisely and confirmed the fix was sufficient. Four additional round-trip tests were added to `los.test.js` to cover both odd- and even-column parities explicitly, serving as a regression baseline.

The PR also bundles two unrelated but ready pieces of work. The `.claude/settings.json` change adds `CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1` as a persistent env var, enabling the agent-teams plugin in all future sessions without needing to set it in the shell. The `map.json` commit is the more substantial addition: approximately 100 hexes in the north-west sector of the South Mountain map (columns 01–21, rows 01–10) were digitized from `sm-map.jpg`, filling in terrain types (clear/woods) and elevation levels (0–6) for the high ground around Turner's Gap, Fox's Gap, and the western ridgeline. Previously-unknown setup and VP hexes in this region were set to `clear` as a conservative baseline pending full digitization verification.

Nothing was deferred from this track. The remaining terrain digitization work — the central and eastern sectors of the map, plus verification of all `clear` baseline assignments against the map image — is tracked in `_todoHexes` within `map.json` and will be addressed in a future map-data track once the LOS engine is being actively tested against real positions.

## 18:48 — PR: Extract Composables from MapEditorView + Unify Selection State (#97)

`MapEditorView.vue` entered this PR at 1,431 lines owning eight distinct concerns in a single file: accordion panel state, hex interaction and selection, bulk map operations, drag-trace linear feature logic, map persistence (fetch/save/draft/push/pull), and more. The PR extracts five composables — `useBulkOperations`, `useLinearFeatureTrace`, `useHexInteraction`, `useEditorAccordion`, and `useMapPersistence` — leaving `MapEditorView` as a thin orchestrator at roughly 400 lines. Each composable has a co-located test file; total new tests added: 101 (570 passing after, up from 520 before Wave 1).

The most consequential design decision was the selection state unification that was the core of issue #97. Previously, `selectedHexId` and `selectedHexIds` were two independent pieces of state updated inconsistently across the file — a latent bug waiting to surface. The fix makes `selectedHexIds` (a `Set` ref) the single canonical source of truth, and converts `selectedHexId` from a writable `ref(null)` to a `computed(() => selectedHexIds.size === 1 ? [...selectedHexIds][0] : null)`. All write sites were audited and updated to operate on the Set. This eliminates the dual-state inconsistency entirely and is enforced by the type system: `selectedHexId` is now read-only at its call sites.

Two ordering constraints shaped the composable wiring in `MapEditorView`. First, `useEditorAccordion` owns `openPanel` and `editorMode` — both of which must exist before `useHexInteraction` receives them as arguments. But `useEditorAccordion`'s `onClearSelection` callback needs `selectedHexIds`, which only exists after `useHexInteraction` runs. The solution is a lazy callback: `let _clearSelection = null` is declared before the accordion composable and assigned after the hex interaction composable runs. The callback closure is only ever invoked at runtime (on user interaction), so the variable is always initialized by then — no temporal dead zone risk. Second, `useMapPersistence` registers no lifecycle hooks itself; instead it returns a `cleanup()` function that `MapEditorView` calls in its `onUnmounted`. This makes the composable usable outside a component context (important for testing without Vue warn noise).

The `useEditorAccordion` composable was the subtlest to test because its `onClearSelection` callback fires on specific state transitions, not on all panel toggles. Four behavior-specific tests cover the cases that matter: fires on closing a tool panel, does not fire on opening, does not fire on non-tool panels, and does not fire when switching between two tool panels. The last case is the non-obvious one: if the user switches from Elevation to Terrain, both have tool modes, so no selection clear should occur — the user is changing tools, not exiting tool mode.

Nothing was deferred from this track. The conductor plan's six phases (useBulkOperations, useLinearFeatureTrace, useHexInteraction, useEditorAccordion, useMapPersistence, final audit) are all complete. ## 20:58 — PR: Click / Paint Modes for Terrain & Elevation Editor

Added a Click / Paint mode toggle to both the Terrain Tool panel and the Elevation Tool panel in the map editor (issue #115). Previously, the terrain tool painted on any hex `mouseenter` event — meaning moving the cursor across the map while a terrain type was selected would silently overwrite hexes. This was the existing behavior, partly masked by the fact that `HexMapOverlay` only emitted `hex-mouseenter` unconditionally. The elevation tool had no drag mode at all. This PR addresses both problems by adding an explicit, visible mode switch.

The implementation is layered across three files in dependency order. The lowest layer is `HexMapOverlay`, which gained an `isPaintMouseDown` ref. `onSvgMouseDown` now sets this flag when `editorMode` is `'paint'` or `'elevation'`; `onSvgMouseUp` (and `@mouseleave`) clears it and emits a new `paint-stroke-done` event. The `hex-mouseenter` emission was replaced with a guard function (`onHexMouseenter`) that checks `isPaintMouseDown` before emitting — so hover-without-mousedown never fires the event in those two modes. For all other modes (select, wedge, LOS, linear feature) the old unconditional emission is preserved. The middle layer is `useHexInteraction`, which gained an optional `paintMode` ref param (`'click'` | `'paint'`, defaulting to `'paint'` for backward compat). The `onHexMouseenter` handler now returns early unless `paintMode === 'paint'`, and the elevation case was extended to call `adjustHexElevation(hexId, +1)` when dragging in elevation mode.

The top layer is the UI and wiring in `MapEditorView`. A shared `paintMode = ref('click')` is the source of truth — shared across both panels because only one is open at a time (accordion model). It is passed down as a prop to `TerrainToolPanel` and `ElevationToolPanel`, which each render a two-button segmented control and emit `paint-mode-change`. Both panels also show dynamic hint text that changes based on the active mode. `MapEditorView` handles the `paint-stroke-done` event from `HexMapOverlay` to flush `saveMapDraft` once per stroke rather than per hex. During a stroke, a `paintStrokeActive` flag suppresses the per-hex draft save inside `onMutated`.

One deliberate tradeoff: right-click-to-lower elevation was kept as click-only even in paint mode. The drag-raise path covers the primary bulk workflow; drag-lower would require tracking mouse button identity and adds complexity for an edge case. The mode toggle also does not persist across page reload — the spec explicitly excluded this, and sessionStorage would add state-management surface without meaningful benefit for the dev-tool use case. The 35 new tests (8 in HexMapOverlay, 4 in useHexInteraction, 10 in TerrainToolPanel, 13 in ElevationToolPanel) bring the total to 625.
