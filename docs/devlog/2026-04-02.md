## 01:58 — PR: Counter Auto-Detection Script Revision (#225)

This PR delivers a complete revision of `scripts/detect-counters.js` — the one-time vision
script that maps counter image files to OOB unit records via Claude's vision API. The
original script (from track counter-detect_20260325) had a 76-match rate on the first run
because it sent all 468 images against an undifferentiated list of 170 units, making it too
hard for the model to distinguish one blue infantry counter from another. The revision
attacks the problem at its root by pre-classifying every image before any API call is made.

Phase 1 added two pure-function helpers: `classifyImage(filename)` parses filename patterns
(C## → CSA front, U## → Union front, CS1-Front\_\* / CS1-Back\_\* → face from name) to
return an `{ army, face }` classification without any network call, and
`detectArmyByColor(client, imagePath)` issues a lightweight second Claude call only for the
ambiguous CS1-\* files where the filename gives no army information. This pre-classification
step cuts the candidate roster from 170 to either 75 CSA or 95 USA units, a 55–45% reduction
that materially improves prompt discrimination.

Phase 2 replaced the flat unit-list prompt with one that includes each unit's full command
chain (corps › division › brigade) so the model can use organizational context — "which
regiment in Hood's division?" is a far more tractable question than "which of 170 units is
this?". The `buildRoster` helper assembles the filtered, annotated candidate list. Phase 2
also fixed a latent JSON parse failure: Claude was responding with valid JSON followed by
explanation prose, which caused `JSON.parse()` to throw on ~293 images. The fix uses a
`/\{[^}]*\}/` regex to extract the first `{…}` object before parsing.

Phase 3 executed the revised script. The first pass matched 76 images; after the JSON fix
and a `--force` re-run, the final tally was 269 images matched out of 468 (CSA: 25 front +
27 back out of 75; USA: 31 front + 14 back out of 95; leaders: 8 of 48). The confidence
threshold was set to 0.0 to capture every best-guess match — the intent is to seed as many
counterRef values as possible for manual verification in the OOB Editor, not to filter for
certainty. The ~199 remaining null counterRefs are left for manual linkage. Twenty new unit
tests covering all helpers were added and vitest.config.js was updated to include
`scripts/**/*.test.js` in the server test project.

The devlog also records a housekeeping correction: `debt-cleanup_20260331`'s plan.md had
all task checkboxes stuck at `[ ]` because the squash merge of PR #224 collapsed the
individual mark-complete commits. All 13 debt items were confirmed implemented by code
inspection and test run (1159 tests passing), and the tracking documents were updated to
reflect the true Complete state.

## 23:16 — PR: OOB Data Model — Leader Ratings, Supply Units, and Succession Variants (#233 #234 #235)

This PR addresses three data model issues discovered during OOB editor testing and filed as
GitHub issues #233, #234, and #235. The work was planned as track `oob-data-model_20260402`
and executed in three sequential phases, each independently verifiable. The OOB editor UI
fix track (#228–#232) was deliberately held until this foundation was in place.

The first phase (issue #233) replaced the `initiativeRating` field on every leader record
with two separate fields: `commandValue` and `moraleValue`. The original field name was
simply wrong — Line of Battle counters show two distinct numbers on the left and right of the
leader counter face, and the left-hand "command value" governs command radius while the
right-hand "morale value" affects morale checks. Conflating them as a single "initiative
rating" had no basis in the rules. All 47 leader records in `leaders.json` were updated, the
Zod schema was revised, and both the detail panel UI and its tests were updated to expose two
separate number inputs.

The second phase (issue #234) added `supply` entries to the two Union corps (1 Corps, 9
Corps) that have dedicated supply units in the South Mountain scenario. The Zod `Corps`
schema gained an optional `supply` field with the same `{ id, name, counterRef }` shape as
`hq`. The `processUSACorps` function in `oobTreeTransform.js` now attaches `_supply` to the
corps node, and `OobTreeNode` already rendered supply children via the existing `_supply`
branch — so the tree display came for free. A key early error: the `SupplyNode` declaration
was placed after `Corps` in the schema file, causing a `ReferenceError` at load time; moving
it above `Corps` resolved it.

The third phase (issue #235) introduced `succession.json` as a new first-class data file
holding "promoted or variant" leader records — cases where a leader appears under a different
command or with different ratings than their base entry. Rather than polluting `leaders.json`,
variants live in a parallel file keyed by side with arrays of `SuccessionVariant` objects,
each carrying a `baseLeaderId` back-reference. A full server route (`successionEditor.js`)
mirrors the `leadersEditor` pattern with GET/PUT, rate-limiting, backup rotation, and
`_savedAt` timestamping. On the client, `useOobPersistence` was extended to load, push, and
pull succession alongside oob and leaders via a three-way fetch. The `buildDisplayTree`
function gained a fourth `succession` parameter; `buildVariantsMap` builds a `baseLeaderId →
[variant]` lookup that `withLeader` uses to attach `_variants` arrays to leader nodes.
`OobTreeNode` renders these as `leader-variant` siblings (purple 'Var' badge) directly beneath
their base leader. `SuccessionList.vue` was updated to merge base leaders and variants into its
type-ahead pool and flag variant names with a `(variant)` label.

Deferred: the anonymous 0–1 replacement brigade leader (a fallback entry for any brigade that
loses its named leader) was identified during spec gathering but pushed out of scope — it
belongs in the game engine's combat resolution logic, not the data model. Similarly, optional
HQ and detachment markers are game implementation concerns and were left for a later track.
The OOB editor UI fix issues #228–#232 are the natural next track now that this data
foundation is complete.
