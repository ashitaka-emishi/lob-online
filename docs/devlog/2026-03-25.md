# 2026-03-25

## 08:50 — PR: Design doc — OOB Editor dev tool

Added the design document for the OOB Editor, a new standalone developer tool for viewing and
editing Order of Battle data for the South Mountain scenario. The editor will sit alongside the
existing Map Editor and Scenario Editor as the third dev-only tool in the suite, guarded by
`MAP_EDITOR_ENABLED=true` and launchable via `npm run dev:oob-editor`.

The core of the design is a two-panel layout: a collapsible hierarchy tree (Corps → Division →
Brigade → Leader + Regiments/Batteries) on the left, and a scrollable detail/edit panel on the
right. Leaders are shown inline under the unit they command rather than in a separate flat list,
which keeps the command structure immediately readable. The unit hierarchy is fixed — the editor
does not support moving units between parents — keeping scope tight for this phase.

A key data model decision was where to place the succession list. Two options were considered:
on the unit's command slot (`successionIds` array on brigade/division/corps in `oob.json`) or
as a back-reference on each leader record. The unit-slot approach was chosen because it directly
models the casualty replacement flow the rules engine will implement: look up the unit's
`successionIds`, find the next living leader, promote their counter, and continue the chain
until the list is exhausted (at which point the unit receives a generic 0-initiative counter).

Counter image linkage was simplified from a sheet/number indirection scheme to direct filename
references (`{ front, back }` on all units; `{ front, back, promotedFront, promotedBack }` on
leaders). Images will be stored in `client/public/counters/` and tracked in git. A new
one-time developer script (`scripts/detect-counters.js`) will use the Claude vision API to
scan the raw counter sheet images in `docs/reference/src-counters-sm/`, match each counter to
the corresponding unit or leader by the text printed on its face, copy the matched files into
`client/public/counters/`, and pre-populate the `counterRef` fields in `oob.json` and
`leaders.json`. Unmatched counters are left null for manual resolution in the editor UI.

The `CounterImageWidget` component will appear on all node type detail panels (regiment,
battery, brigade, division, corps, and leader), providing a consistent browse/upload flow
everywhere a counter linkage exists. The home page (`StatusView.vue`) will gain a Dev Tools
nav section pointing to both the Map Editor and the new OOB Editor. One open question remains
for implementation: whether the auto-detection script should write only high-confidence matches
or flag low-confidence ones for human review.

The 308 raw counter images for South Mountain (`CS1-Front_*.jpg`, `CS1-Back_*.jpg`, and

the design doc is the primary navigation aid between the design phase and implementation tracks,

## 16:08 — PR: Counter images, counterRef schema, and auto-detection script (#197)

This PR delivers the counter image foundation for the OOB Editor: 469 counter images moved to
`client/public/counters/` where Vite and Express can serve them at runtime, nullable `counterRef`
fields added to every unit and leader record in `oob.json` and `leaders.json`, and a developer
script (`scripts/detect-counters.js`) that attempts to pre-populate those fields via the Claude
vision API.

The `counterRef` schema evolved considerably during implementation. The initial design stored just
`{ front, back }` filenames. Two changes were made mid-flight based on hands-on experience with
the counter images: first, confidence scores were added alongside each filename
(`frontConfidence`, `backConfidence`) so the OOB Editor can visually flag uncertain matches for
human review; second, the confidence threshold was dropped to 0 so all Claude guesses are
recorded rather than silently discarded — the editor UI is the right place to accept or reject
them, not the script. Leader records carry four slots (`front`, `back`, `promotedFront`,
`promotedBack`) with a confidence field for each.

The auto-detection script went through several prompt iterations informed by the actual counter
art. Key findings: side (front vs back) is reliably determined from the filename (`CS1-Front_##`
vs `CS1-Back_##`; `C##`/`U##` cut-outs are always fronts). Back faces show "COL" in red above
the regiment name and command chain; front faces show regiment name, abbreviated
division/corps chain (Confederate: `C/DH` style; Union: brigade/division/corps numbers), weapon
type and strength at bottom-left, morale at bottom-right. The roster passed to Claude was
enriched with the full organizational chain and unit stats so it could resolve ambiguities
between similarly-named regiments. Army hints (Confederate for `C##` files, Union for `U##`
files) narrow the search space further.

Despite these improvements, reliable auto-detection was not achievable — the vision API produced
too many mismatches and low-confidence guesses to be useful as a seeding mechanism. The issue is
closed as not achievable; `counterRef` linkages will be set manually in the OOB Editor UI using
the `CounterImageWidget` component. The script remains in the repository as a reference and
potential future starting point if the approach is revisited with a more capable model or a
different matching strategy (e.g. OCR + text matching rather than pure vision classification).
The counter images, schema changes, and data migration (`counterRef: null` on all 172
units/batteries and 48 leaders) are all committed and ready for the OOB Editor implementation
tracks.

## 21:59 — PR: OOB Editor Foundation — Launch Script, Store & Hierarchy Tree (#192 #196)

This PR delivers the foundational layer of the OOB Editor dev tool: the `dev:oob-editor` launch
script, a Pinia store with three-tier persistence, and a fully functional read-only hierarchy tree
that renders the complete South Mountain order of battle for both sides. The editor is accessible
at `/tools/oob-editor` when `MAP_EDITOR_ENABLED=true` and linked from the home page Dev Tools
section.

The Pinia store (`useOobStore`) handles data loading with a server → localStorage → bundled JSON
fallback chain and provides `pushToServer`/`pullFromServer` for explicit sync. Edits are tracked
with a `dirty` flag and auto-saved to localStorage after a 500ms debounce. The `updateField(path,
value)` method accepts dot-separated paths (rooted at `'oob'` or `'leaders'`) so the detail panel
can write any field without knowing the full data shape. The store tests use
`setActivePinia(createPinia())` per suite to isolate state, and mock both `fetch` and `localStorage`
via `vi.stubGlobal`.

The hierarchy tree was the most iterative part of the work. The final design builds a synthetic
display tree from the raw `oob.json` structure rather than restructuring the data file itself —
the JSON stays optimised for rules-engine lookups (keyed artillery objects, flat unit arrays) while
the view layer transforms it into the command hierarchy for display. Key transforms: USA artillery
is distributed from a corps-level keyed object down to its matching division or brigade using key
naming conventions (`arty1-1c` → `1d-1c`, `arty1-1d-9c` → `1d-9c`, `arty-1kg-9c` → `1b-kd-9c`
for Kanawha-style brigade arty); the cavalry formation is wrapped in a synthetic "Cavalry Division"
node with Pleasonton's leader entry and F/Cav as a proper brigade child underneath; the Confederate
tree is rooted at a "Longstreet's Wing" wrapper node.

Several display polish items were resolved through iteration: HQ nodes use derived names (`"1 Corps
HQ"`, `"AotP HQ"`, `"dH Div HQ"`); leader nodes show abbreviated rank prefixes (`Maj Gen`,
`Brig Gen`) drawn from `leaders.json`; regiment and battery names with a leading number receive
ordinal suffixes (`22nd NY`, `1st NH Lt`, `8th Mass Lt`); "Collapse all" folds only the second
level of the tree (corps/division rows) leaving the army/wing root expanded; brigade names in
`oob.json` had commander names stripped (they are now shown via injected leader nodes instead).

The sidebar width was set to 480px (up from 320px) to accommodate the deeper indentation of the
full hierarchy. The detail panel currently shows a placeholder; unit editing will be added in a
subsequent track. The 39 component tests and full 963-test suite all pass.

## 23:12 — PR: OOB Editor Detail Panel & Counter Image Widget (#193)

This PR implements the OOB Editor detail panel — the right-hand editing pane that renders type-dispatched form fields for the node selected in the hierarchy tree. It also delivers a model revision pass that cleaned up several legacy artifacts from the raw `oob.json` data.

The `OobDetailPanel` component receives `node`, `nodeType`, and `nodePath` as props and renders a different field set for each unit category. Regiment and cavalry nodes show id (read-only), name, type select, morale select, weapon select, and strength points, plus a `CounterImageWidget` for counter image linkage. Battery nodes show gun type, strength points, and morale (renamed from the old `ammoClass` field). Brigade and division nodes show name and wreck threshold only — deliberately minimal, since these nodes don't have their own counters or morale ratings in the Line of Battle rules. Corps nodes expose only name. HQ and supply nodes expose name plus a counter image widget.

The `CounterImageWidget` is a reusable sub-component that renders front/back thumbnail slots. When a `counterRef` filename is present, it shows an `<img>` sourced from `/counters/{filename}`; when the slot is null or the image 404s (via `@error`), it shows a grey placeholder with a Browse button. Browse triggers a hidden `<input type="file">` that POSTs to `/api/counters/upload` — the server endpoint for that route is deferred to issue #191 and will wire up when that PR merges. The widget also exposes editable filename text inputs for manual assignment without a file upload.

Selecting a node in the tree calls `store.selectNode(node, nodeType)`. The store exposes `selectedNodeType` (a ref) and `selectedNodePath` (a computed), where the path is resolved by `findNodePath` — a new utility that walks raw `oob.json` recursively by node id and returns a dot-separated path string like `union.corps.0.divisions.0.brigades.0.regiments.0`. This path is what `updateField` uses to write mutations back into the store's reactive data tree.

The model revision bundled in this PR was more substantial than expected. `stragglerBoxes` was removed from all infantry and cavalry units (it was an RSS artifact with no rules backing in Line of Battle v2.0). Brigade and division records were stripped down to name + `wreckThreshold` — removing `morale`, `wreckTrackTotal`, and counter references that had crept in during earlier design passes. Artillery `ammoClass` was renamed to `morale` throughout. Most visibly: unit names are now stored with ordinal suffixes directly in `oob.json` (`"22nd NY"`, `"1st NH Lt"`, `"8th Mass Lt"`) and the runtime ordinal-suffix transform in `OobTreeNode` was removed. A one-shot migration script (`scripts/transform-oob.js`) performed the bulk transformation; the Zod schema and all test fixtures were updated to match.
