# 2026-04-12

## 09:03 — PR: Map Test Tool (M3)

This PR delivers the Map Test Tool, the interactive rules-validation harness called for in the M3
milestone plan. It provides a read-only view of the South Mountain map with five panel-driven
tools: Movement Path, Movement Range, Hex Inspector, LOS, and Command Range. The tool was
motivated by the need to validate the M3 engine modules (hex.js, movement.js, los.js,
command-range.js) against real map data before building the game engine on top of them — having an
interactive UI that shows exactly what the engine computes for any clicked hex makes bugs
immediately visible.

The implementation followed the Conductor track (map-test-tool_20260411) through five phases: the
command-range engine module (Phase 1), five server GET endpoints at /api/tools/map-test/ (Phase 2),
the client shell and router (Phase 3), the five panel Vue components (Phase 4), and removal of the
legacy LOS prototype tab from the map editor (Phase 5). A key design decision in Phase 1 was using
movementRange() with the leader formation to compute command radius — rather than raw hex distance —
because LOB §10.6a defines command radius in movement points, not hexes. An SM §1.1a road-slope
exemption is applied by passing a shallow-copy of the scenario with roadSlopePenalty zeroed out,
avoiding any engine modification.

The panel architecture is simple and consistent: each panel SFC accepts a clickedHexId prop that
the orchestrator (MapTestView) updates on every map click, watches that prop to accumulate clicks
internally, calls its server endpoint, and emits an overlay-update event with a hexFill config that
the orchestrator merges into the HexMapOverlay's overlayConfig. Two-click panels (Movement Path and
LOS) track first/second click state internally. Switching tabs resets clickedHexId and clears the
overlay. A notable bug caught during testing: CommandRangePanel originally accessed result.value
inside the fillFn closure, which would throw when result resets to null at the start of a
re-query. The fix captures all three zone Sets as local variables at emit time so the closure is
purely data-driven.

The final UI layout mirrors the map editor — full-screen scrollable map image with the hex overlay,
accordion sidebar on the right, no separate tab bar. The map image and overlay are driven by the
same calibration.imageScale and image natural-size tracking used by the editor. A
npm run dev:map-test convenience script was added alongside the existing dev:map-editor and
dev:oob-editor scripts.

The LOS prototype that previously lived in the map editor as LosTestPanel.vue / useLosTest.js was
deleted in Phase 5 — its functionality is now fully superseded by the formal LosPanel in the Map
Test Tool. The map editor loses nothing of substance; it gains a cleaner sidebar. All five panels
have Vitest coverage (43 new tests), and the full suite ends at 1712 passing.

## 21:34 — PR: Table Test Tool — interactive validator for all LOB v2.0 game tables

This PR delivers the Table Test Tool, the second M3 dev tool and the final feature before the M3
debt-cleanup sprint. The tool provides a standalone UI for exercising all eleven LOB v2.0 game
tables without needing a map or scenario: Combat, Opening Volley, Morale, Morale Transition,
Closing Roll, Leader Loss, Command Roll, Order Delivery, Fluke Stoppage, Attack Recovery, and Zero
Rule. The motivation is the same as the Map Test Tool — having an interactive interface that
exercises the engine functions directly makes rules bugs immediately visible before they propagate
into the game loop.

The implementation followed the Conductor track (table-test-tool_20260412) through three phases.
Phase 1 created eleven POST endpoints at `/api/tools/table-test/*` in `server/src/routes/tableTest.js`,
each backed by the engine table modules already delivered in the M3 rules engine PR. Phase 2 wired
the client router entry and created `TableTestView.vue` — an orchestrator with a tab bar and
`<component :is="activePanel()">` for dynamic panel rendering. Phase 3 implemented all eleven panel
SFCs with full Vitest coverage.

A key design decision was using POST (not GET) for all table endpoints even though the operations
are pure computation with no side effects. The rationale is that table inputs include structured
modifier objects (arrays of boolean flags) that are awkward to encode as query string parameters,
and the consistent POST-with-body pattern matches how a future game loop will invoke the same
tables. Each endpoint does allowlist validation plus numeric range checks and returns a 400 with an
error message on any invalid input — the same shape the client panels display in their error slot.

The panel layer shares a single `panel-base.css` imported via `@import './panel-base.css'` in each
SFC's scoped style block. This keeps panel SFCs thin (template + script only) while providing
consistent visual language across all eleven panels. The alternative — inlining styles or using a
global class — was rejected because scoped styles prevent leakage and the import approach avoids
any Vue scoping-hash collisions. One non-obvious testing issue: three "reset clears result" tests
initially failed because the assertion keywords (e.g. "shaken", "base check") appeared in both the
form label area and the result section. The fix was to assert on terms that only appear in the
result block (e.g. "effective roll", "step 1 effective roll", "step 1 threshold").

Nothing was deferred. All eleven panels are fully implemented with loading states, error display,
modifier breakdowns, and reset. The full suite ends at 1827 passing (115 new tests — 40 server
route tests, 6 shell tests, 69 panel tests).
