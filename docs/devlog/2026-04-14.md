# 2026-04-14

## 13:36 — PR: M3 Debt Sprint — score-3 technical debt cleanup (#284 #288 #289 #300 #302 #303)

This PR closes out the M3 debt sprint conductor track, resolving six score-3 debt items across three
areas: engine correctness, Map Test Tool architecture, and the Table Test Tool from PR #305. Together
these items dropped the net open debt score from 66 to 39, clearing the way for M4 (game loop /
socket layer) without carrying known architectural compromises into real-time multiplayer code.

The most security-relevant fix is the path traversal containment guard added to `loadMap` and
`loadScenario` (#284). Both loaders now call `assertContainedPath()` — a small function using
`path.resolve` and a prefix check against `PROJECT_ROOT` and `TMPDIR` — before reading any file.
Error messages deliberately omit the resolved path to avoid leaking filesystem layout. The tmpdir
allowance was added because scenario tests write temp JSON fixtures to `os.tmpdir()` during test
runs; excluding it would have forced a much larger test fixture refactor.

The `movementPath` hexside cost fix (#288) required introducing `hexEntryCostBreakdown()` as an
internal helper that returns `{terrainCost, hexsideCost, total}` instead of a single number.
The public `hexEntryCost()` function delegates to it for backward compatibility. The path breakdown
loop now calls `getDirectionBetween()` — a new helper that walks `hexNeighbors` to recover the
direction index between consecutive path hexes — so the correct hexside entry is used per step. This
was always supposed to be correct; the bug was that the breakdown stored `hexsideCost: 0` regardless
of what the engine actually computed.

LOS terrain height bonuses (#289) were hardcoded as a module-level constant (`TERRAIN_LOS_HEIGHT_FLAG`
with woods/orchard/woodedSloping all set to 1). They are now read from `scenario.rules.terrainLosHeights`,
with a fallback to the old constant for backward compatibility. The production scenario.json was updated
to carry the explicit map. This pattern mirrors how movement costs were made data-driven in an earlier
sprint, and sets the precedent for any future per-scenario LOS customization.

The Map Test Tool phase (#300, #302, #303) added orchestration test coverage for `MapTestView`, applied
`isValidHexId()` format guards to all six hex-accepting routes (returning 400 with a descriptive message
on malformed input), and added a dedicated `GET /api/tools/map-test/data` endpoint so the view no longer
depends on the map-editor route. These three items were deferred from M2 because the map test tool was
new — it was more important to ship it than to immediately retrofit the architecture. Now that the tool
is stable and pre-M4 is the right moment to tighten its contracts.
