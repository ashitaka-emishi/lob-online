# 2026-03-19

## 14:58 — PR: Map editor design v2 — tool-based model and shared architecture

After several weeks of iterative work building out the map editor, this PR locks down the design before any further implementation. The goal was to capture everything learned from building the current editor — what worked, what was redundant, what was fragile — and produce a stable design document that won't need major revision once implementation begins in earnest.

The central design shift is moving from a global-layer-toggle model to a tool-owns-its-overlays model. Previously, a row of checkboxes in `EditorToolbar` controlled which SVG layers were visible globally. The new design eliminates that bar entirely: each tool panel declares an `overlayConfig` object, and `HexMapOverlay` becomes a pure declarative renderer with zero tool-specific logic. Adding a new tool requires no changes to the overlay component. The config-driven approach also standardizes the toggle pattern — `BaseToolPanel` auto-renders checkboxes for any layer marked `alwaysOn: false`, so tools never manage toggle state manually.

Two significant data model decisions were made. First, hex edge directions are now stored as integer face indices (0–5) rather than named `HexDir` strings. This decouples stored data from the `northOffset` calibration value — changing north offset only affects display labels, never the underlying data. The `compass.js` formula module derives compass labels at render time from the face index and `northOffset`. Second, edge data moves to canonical ownership: only face indices 0, 1, 2 are stored on the current hex; faces 3, 4, 5 are stored on the neighbour as `dir − 3`. This halves edge storage and eliminates the category of sync bugs where both copies of a shared edge could diverge. The engine lookup rule is simple and already required for movement and LOS.

A shared composable architecture was designed to eliminate the redundancy that had accumulated across tools. `useHexPaintTool` and `useEdgePaintTool` handle all paint/click/right-click interaction for their respective target types, both consuming a shared `usePaintStroke` micro-composable for stroke batching. `useClickHexside` handles ford and bridge placement. `BaseToolPanel` and `ToolChooser` provide the common UI shell. A `config/feature-types.js` registry is the single source of truth for all feature type strings and their display properties — chooser swatches and overlay colors both import from here to prevent drift. Five formula modules under `formulas/` extract all non-trivial math with JSDoc references to source material (Amit Patel's hex grid guide, LoB rulebook sections, SM scenario special rules).

The auto-detect design document (`map-editor-auto-detect-design.md`) was removed entirely. AI-assisted terrain and elevation detection is a future optimization, not needed to make the editor functional. The old milestone structure (v1.0 MVP / v2.0 Enhanced / v3.0 Extended) was replaced with a single focused milestone: "Create usable map editor." Nine GitHub issues were filed against it covering the data model migration, framework refactor, five tool implementations, ford/bridge controls, and the north offset label bug fix.

## 15:42 — PR: Map schema foundation — data model overhaul + compass label fix (#135, #143)

This PR delivers the two foundation tickets gating all further map editor work. Issue #135 overhauled the map data model to match the v2 design spec, and issue #143 fixed a long-standing off-by-one bug in the north offset compass label formula. Both were batched into a single track because they share the same face-index semantics and the schema migration had to land before anything else.

The schema changes touch three areas. First, the `features: HexFeature[]` array was replaced with `hexFeature?: { type: 'building' }` — a single optional field with a literal type constraint, since only buildings are valid hex features in the South Mountain scenario. Second, edge keys were migrated from string `HexDir` values (`'N'`, `'NE'`, etc.) to integer face index strings (`'0'`, `'1'`, `'2'`), representing only the canonical owner faces. The third face of a shared edge (indices 3, 4, 5) now lives on the neighbour hex as `faceIndex − 3`. Third, a `validateCoexistence()` function was added to the `superRefine` pass: ford requires a stream on the same edge, bridge requires a road/trail/pike, and elevation-type features (`elevation`, `slope`, `extremeSlope`, `verticalSlope`) are mutually exclusive per edge.

The `map.json` migration was the most involved part. The working file had 559 hex entries including newly digitized NW sector hexes that had already been saved with old string edge keys. The migration script converted 178 canonical edges in place, then relocated 188 non-canonical S/SW/NW edges to their canonical neighbour hexes (face 3→0 on S-neighbour, 4→1 on SW-neighbour, 5→2 on NW-neighbour). Twelve border hexes had out-of-bounds neighbours and were dropped with a warning. The merge of canonical and non-canonical data onto the same hex exposed 17 coexistence violations — cases where `elevation` had been stored alongside a more specific slope type on both sides of a shared physical edge under the old dual-storage model. These were resolved by keeping the most specific elevation type (`verticalSlope > extremeSlope > slope > elevation`) and dropping the redundant generic marker.

The compass label bug fix (#143) was straightforward: the existing inline formula in `CalibrationControls.vue` used `Math.floor` where it should have used `Math.round`, causing the N label to appear one position clockwise of the correct ring position when `northOffset ≠ 0`. The fix extracts the formula into `client/src/utils/compass.js` with 20 unit tests covering `northOffset` values 0, 3, 4, and 11, including a dedicated regression case for `northOffset=3` (the South Mountain default). The `CalibrationControls` computed now imports `compassLabel()` rather than re-implementing the formula inline.

The `HexEditPanel.vue` and `HexMapOverlay.vue` components still read the old `hex.features` and `hex.hexsides` fields — these were explicitly deferred as the UI tools are being rebuilt from scratch in follow-on tracks per the v2 design. The `useMapPersistence.js` `KNOWN_MAP_KEYS` allowlist was updated to drop `hexsideTypes` since that top-level field was removed from the schema and from `map.json`.

## 16:06 — PR: Fix schema gaps from PR #145 — hexFeature migration + EdgeFeature enum (#146, #147)

This PR closes two items deferred from the PR #145 team review that were caught after the squash merge had already landed. Both fixes were straightforward but required updating the component layer to match the schema changes that had just been made.

The `HexEditPanel` was the more involved change. The old component managed a `features[]` array (a list of `{ type }` objects) and a `hexsides` map (a legacy field removed in #135). These were replaced with a single `hexFeature` boolean toggle. Since the South Mountain scenario only recognises buildings as a valid hex feature, the new UI is a single "Building" checkbox — simpler and more accurate than a generic add/remove list. The `hexFeatureTypes` prop was removed entirely. The `toggleBuilding` handler sets `form.hexFeature` to `{ type: 'building' }` on check and `null` on uncheck. The `toggleSeed` handler was also cleaned up to stop passing the removed `features` field in `confirmedData`.

`HexMapOverlay` required a one-line change: the cell mapping computed `features: known?.features ?? []` was replaced with `hexFeature: known?.hexFeature ?? null`. No template changes were needed — the overlay doesn't currently render hex features, so this was purely a data consistency fix to ensure downstream code won't trip on the old field shape.

`EdgeFeature.type` was promoted from `z.string()` to `z.enum(EDGE_FEATURE_TYPE_VALUES)` with all 11 canonical types exported as an array constant. The full list — `road`, `trail`, `pike`, `stream`, `ford`, `bridge`, `stoneWall`, `elevation`, `slope`, `extremeSlope`, `verticalSlope` — was verified against the map editor design doc rather than inferred from the current map data alone (the data only contained 7 types, missing `trail`, `stoneWall`, `ford`, and `verticalSlope`). The old comment on `ELEVATION_TYPES` and `ROUTE_TYPES` noting they should eventually be derived from the enum was updated to reflect that the enum now exists. The tech-debt report for PR #145 was also committed on this branch since it wasn't included before the merge.
