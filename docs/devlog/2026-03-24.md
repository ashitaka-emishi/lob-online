# 2026-03-24

## 10:11 — PR: Map editor rendering and interaction polish

This PR delivers a second wave of map editor UX polish on top of the #180–#185 refactor merged
yesterday. The changes fall into four areas: road rendering correctness, glyph sizing and
orientation, terrain tool fixes, and interaction discipline.

The most significant fix is to the through-hex road rendering. The previous implementation only
iterated the three canonical edge directions (N, NE, SE) per hex when building center-to-midpoint
segments, meaning roads on a hex's S, SW, or NW edges were never drawn from that hex's center
outward. The fix extends `useEdgeLineLayer` to accept an optional `cellById` map and `gridSpec`,
then looks up mirror-edge features from adjacent hexes' canonical faces when `includeMirrors` is
true (through-hex mode only). This restores the full six-direction spoke for every routed hex.

Bridge and ford glyphs were resized from font-size 9 to 24 and given a rotation transform so
the `][` symbol aligns along the hex edge direction rather than always rendering horizontally. A
`glyphTransform` helper in `HexMapOverlay` computes `rotate(angle, cx, cy)` based on a fixed
map (N/S → 0°, NE/SW → 60°, SE/NW → −60°). Road line weights were also increased substantially
(trail 5→8 px, road 6→10 px, pike 7→12 px) so roads read clearly against the map image. The
building icon was scaled to match (font-size 10→24) and recolored to black-on-white to
distinguish it from the terrain fill beneath it.

Interaction discipline was tightened in two ways. First, `paintMode: ref('click')` is now
explicitly passed to `useHexInteraction` from `MapEditorView`, closing a bug where the composable
defaulted to `'paint'` mode and would apply terrain or elevation to every hex the cursor drifted
over. Second, hex selection is removed from elevation and paint click handlers — only the LOS
tool and wedge/select modes may set `selectedHexIds`. The LOS tool itself was also broken because
`'losTest'` was absent from `INTERACTIVE_PANELS`, preventing `HexMapOverlay` from emitting click
events when that panel was open; that is now corrected.

Two data issues were fixed: the orchard terrain color key was `orchards` (no match) while the
terrain type string is `orchard`, so orchard hexes showed no fill; the key is now corrected and
the color changed to a light green. Road groups were also removed from the base overlay config's
`ALL_EDGE_GROUPS` (renamed `BASE_EDGE_GROUPS`) so the old 20/80 along-edge road segments no
longer appear when no tool panel is active — roads are exclusively rendered via the road panel's
through-hex style.

## 10:30 — PR: Map editor team-review findings (all 10, no deferrals)

This PR resolves all ten findings from the parallel team review of the previous map editor
polish work. The findings split across medium and low severity, and all were fixed in-place
with no deferrals.

The two medium-performance findings centered on repeated `adjacentHexId` calls during reactive
invalidation. `HexMapOverlay.gridData` now builds a `neighborMap: Map<"hexId:dir", cell>` once
per calibration change (alongside the existing `cellById`). Both `useEdgeLineLayer` (which looks
up mirror-edge features for S/SW/NW directions in through-hex mode) and `roadEdgeCountMap` (which
counts road-touching edges per hex to apply the 2+ threshold) now use this pre-built map via O(1)
lookups instead of calling `adjacentHexId` per cell per computed invalidation. The change
required updating `useEdgeLineLayer`'s public API from `(cells, overlayConfig, cellById, gridSpec)`
to `(cells, overlayConfig, neighborMap)`, which is a simpler surface — callers no longer need to
pass two separate refs that the composable had to combine internally.

Two medium-architecture findings were straightforward state hygiene. `paintMode` was promoted from
an inline `ref('click')` argument in the `useHexInteraction` call to a named `const paintMode =
ref('click')` next to the other state refs, matching the naming convention of its siblings.
`activePanelOverlayConfig` was not being cleared on panel transitions, meaning the stale config
from a closed panel remained active until the newly opened panel fired its first `overlay-config`
event. A `watch(openPanel)` that resets the ref to `null` on every panel change closes this gap
cleanly without requiring the panels themselves to know about each other.

The low-performance findings were pre-computation opportunities. Bridge and ford glyph data now
carries pre-computed `{x, y, transform}` fields from `_buildGlyphEdges` instead of having the
template call `edgeMidpoint` twice and `glyphTransform` once per glyph per render. The
`cellsWithDisplayAttrs` computed was changed to a thin wrapper `{ cell, fill, fillOpacity, stroke,
strokeWidth, strokeOpacity }` rather than spreading the full cell object — the grid polygon
template was updated to access `da.cell.id` and `da.cell.points` instead of `da.id`/`da.points`.

The low-architecture findings consolidated shared logic. `EDGE_PREREQUISITES = { bridge: [...],
ford: [...] }` was extracted to `feature-types.js` and is now imported by both `RoadToolPanel`,
`StreamWallToolPanel`, and the production `onEdgeClick` dispatch in `MapEditorView`. The if/else
chain in `onEdgeClick`/`onEdgeRightClick` was replaced with an `EDGE_DISPATCH` lookup table keyed
by panel name, each entry carrying `selectedType`, `clearTypes`, and an optional `clearSingle`
flag. Finally, the `useEdgePanelWiring` JSDoc was updated to document that the caller watches
`openPanel` and resets `activePanelOverlayConfig` on close.
