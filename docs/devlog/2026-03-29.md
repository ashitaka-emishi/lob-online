## 13:55 — PR: Map Editor UI Refactor — Tool Panel Cleanup & Feature Improvements

This PR closes out conductor track `map-editor-ui-refactor_20260323`, which was scoped on 2026-03-23 to deliver six targeted improvements to the map editor tool panels: removal of click/paint mode toggles and bulk elevation ops, building-terrain coexistence in the data model, a road rendering overhaul (wider outlined lines, 2+ edge threshold, bridge glyph), and a stream tool addition (ford as edge type with glyph rendering). The track was created before the implementation but the actual code changes were completed incrementally across the `map-editor-tool-closeout_20260323` and earlier sessions. This PR closes the tracking gap by adding the four test cases that were deferred and marking the plan complete.

The core design decision worth noting is that building-terrain coexistence was implemented via the existing `hexFeature: { type: 'building' }` field rather than the originally planned `building: z.boolean()` migration. Both approaches achieve the same user-visible behavior — a hex can carry both a terrain type and a building overlay simultaneously — but the `hexFeature` path was already in place and validated by the schema. Migrating to a separate boolean would have required a data migration pass over `map.json` and offered no functional improvement at this stage.

The road rendering 2+ edge threshold (suppressing isolated single-edge roads) was implemented in `HexMapOverlay.vue` via `roadEdgeCountMap`, a computed that counts canonical edge ownership across neighbors. The threshold test added in this PR explicitly covers the edge case of exactly one road edge (not just zero), which the prior test suite left uncovered. Bridge and ford glyphs are rendered as SVG `<text>` elements in dedicated layer groups (`layer-bridge-glyphs`, `layer-ford-glyphs`), rotated to align perpendicular to their respective edge directions using pre-computed angle constants.

The four new tests added here are the only code changes in this PR. All production code was already in place. The tests were identified as gaps during the track closeout review: 1-edge road suppression, bridge glyph presence, ford glyph presence, and building-over-woods terrain preservation. Each test mounts the relevant component with a minimal fixture and asserts on rendered SVG structure or emitted values.

Deferred: no items. The full acceptance criteria for issues #180–#185 are now covered by tests and the CI gates pass cleanly.

## 16:51 — PR: OOB Editor Debt Sprint — Issues #207 #209 #210 #211

This PR resolves the four highest-priority open debt items from the OOB editor, which collectively scored 9 of the 25 net open debt points. The sprint was scoped as a single track (`oob-debt-sprint_20260329`) and implemented across three phases: store structural fixes, UX correction, and persistence safety.

The two store structural items (#209, #210) shared a root cause: the store lacked a path index and a shared `usedFiles` derivation. `selectedNodePath` was a `computed` that re-walked the full OOB tree on every reactive change to `oob` or `selectedNode` — including on every `updateField` call during editing. Converting it to a `ref` set once in `selectNode` eliminates the recurring O(n) walk. The `selectNode` signature now accepts an optional `nodePath` argument for future call sites that already hold the path; when omitted it falls back to a single `findNodePath` call at selection time. The `usedFiles` hoist (#209) moved the `collectUsed` tree walk from a per-widget-instance `computed` (up to ~200 instances each re-running on every reactive change) into a single `usedCounterFiles` computed on `useOobStore`, shared across all `CounterImageWidget` instances.

The auto-commit fix (#211) removed the `if (!current && list.length > 0) commit()` call from `activate()`. The original intent was to show a counter preview immediately on click, but `commit()` writes to the store — making an inspection click a silent, irreversible data mutation with no undo. The slot now enters an active-but-uncommitted preview state; the user must press ↑ or ↓ to persist a selection. The hint text was updated to reflect this. The existing test that asserted the auto-commit behavior was rewritten to assert the opposite, and a new companion test confirms that ↑/↓ still commits.

The push confirmation (#207 partial) follows the `useMapPersistence` pattern: `requestPush()` sets `showPushConfirm = true` (no fetch), the view renders a modal overlay, and `confirmPush()` resets the flag and calls the actual push. The Zod validation spec calls for importing server-side schemas into the client store, but Zod is not a client dependency. Instead, `_isValidOobShape` and `_isValidLeadersShape` perform the same structural guard that `useMapPersistence` uses for map data — requiring `union` and `confederate` to be non-null objects. This catches the malformed-localStorage and truncated-response cases without pulling in the full server schema graph.

Deferred: #207 remaining items (offline detection, validation on push body) and #202 (full schema validation on localStorage) remain open. The prototype pollution guard extension (#216), dual-script pattern (#215), and the three test coverage gaps (#212, #213, #214) are unaffected by this sprint and stay on the register.
