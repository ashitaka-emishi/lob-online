## 10:17 — PR: Post-M5 Debt Bundle 2 — Minor Cleanups and Test Hardening

This PR closes out the second wave of minor debt items surfaced during the PR #437 team review. Nine issues (#436, #438–#439, #442–#447) were filed as score-1 and score-2 deferred items; all nine are resolved here with no new debt accepted. The changes span architecture cleanup, test hardening, and one accessibility improvement — none touch auth, persistence, the rules engine, or API contracts.

The three architecture cleanup tasks address the same underlying pattern: redundancy introduced by sanitizeCalibration being misunderstood at the call site. GameView was spreading `DEFAULT_CALIBRATION` into the argument before calling `sanitizeCalibration`, which already fills missing fields from those defaults — so the spread was a no-op that obscured intent. That redundancy is removed. Similarly, `GridSpecSchema` was exported from `map.schema.js` but never used outside the module; removing the `export` keyword closes off an unintended public API surface. `useCalibration.js` had re-exported `DEFAULT_CALIBRATION` and `sanitizeCalibration` from the composable path, creating two valid import routes for the same symbols; only the direct `calibration.js` path is now valid.

Test hardening addressed four distinct fragility patterns. The `STUB_GRID_SPEC` fixture name was shared between server and client test files with different shapes; adding the `_WIRE` suffix makes explicit that these are pre-sanitize API wire format objects, not the post-sanitize calibration objects. The 503 startup-error test in `scenarios.test.js` relied on `vi.resetModules()` being called in a specific order relative to other test setup; moving it into its own `describe` block with scoped `beforeEach`/`afterEach` guards eliminates the ordering dependency. The `makeMultiFetch`/`makeFetchSequence` helpers now carry a comment documenting that URL patterns must be ordered most-specific first to avoid substring collisions — a latent bug waiting to happen as the test suite grows. Finally, a new `scenarios.integration.test.js` file provides a real-loadMap health-check that the mocked unit tests cannot: it mounts the actual scenarios router against the real `map.json` on disk, ensuring the integration path works end to end.

The `useOobData` composable test coverage was extended to exercise three-level OOB nesting (corps → division → brigade), confirming that `collectOobUnits` correctly recurses through arbitrary depth and that all id-bearing nodes — not just leaf brigades — appear in `oobUnitMap`. A `// TODO: oobError` comment was added to flag that the raw error string passed to `oobError.value` propagates directly to the DOM via GameView, which should eventually be sanitized or user-message-formatted. The accessibility task wrapped the three status banners (loading, error, map-config warning) in a `<section aria-label="Game status notifications">` landmark, giving screen-reader users a named navigation target for the notification area without changing any visual layout.

## 19:13 — PR: Map Editor Bugs #416 #417 #419 — Active Highlight, Contour Elevation Toggle, Map Data

This PR closes three open map editor bugs (#416, #417, #419) and adds two UX improvements requested during the ongoing South Mountain map digitization session: active-type highlighting across all edge tool panels, and an inline elevation overlay toggle on the contour tool.

Bug #416 fixed edge lines rendering in the center of the hex face rather than along the full hex side. The root cause was that `useEdgeLineLayer.js` was calling `edgeLine20_80`, which renders from 20% to 80% of the edge length — a style suited for short style indicators, not full edge coverage. Adding an `edgeLineFull` function to `hexGeometry.js` that spans the full corner-to-corner extent and wiring it into the `along-edge` style case gives clean, complete edge lines that match the visual intent of the tool.

Bug #417 was a silent coexistence failure in the contour tool: clicking a contour type over an existing different contour type did nothing instead of replacing it. The fix has two parts. First, `validateCoexistence` in `edge-model.js` was extended to include `'elevation'` in the set of mutually-exclusive contour types (it was previously only checking slope types). Second, `handleEdgePaint` in `MapEditorView` was changed from a guard-and-return pattern to a strip-then-add pattern for contour types — all existing contour features are removed before the new type is painted. This is the correct semantics for an "exclusive type" group: the user wants to overwrite, not be blocked.

Bug #419 addressed the 'unknown' terrain type leaking through the system. The stub hex used by `resolveHexOrStub` (for boundary lookups where the neighbor doesn't exist in the map) was changed from `terrain: 'unknown'` to `terrain: 'clear'`, and a load-time normalization watcher was added to `MapEditorView` that migrates any legacy 'unknown' entries in loaded map data. The 'unknown' value was also removed from the terrain tool's default type list and the terrain icon map.

The active-type highlight was straightforward in principle but revealed a subtle Vue 3 reactivity trap: `useEdgePanelWiring` returns a plain object containing a `Ref`, but Vue 3 only auto-unwraps refs in templates when the ref is a top-level property on the reactive object returned by `setup()`. When a ref is nested inside a plain object (as it is here), the raw `Ref` object is passed to the child prop rather than the unwrapped string value. Fixing it required adding explicit `.value` access on `road.selectedType.value`, `stream.selectedType.value`, and `contour.selectedType.value` in the `MapEditorView` template.

The elevation overlay checkbox replaces the prior approach of using `BaseToolPanel`'s `toggleLabel` mechanism, which placed the checkbox at the bottom of the panel where it was easy to miss. The new approach adds a direct `v-model` checkbox inside the panel content area with scoped CSS styling, and sets `hexFill.alwaysOn: true` to prevent a duplicate toggle from appearing in the BaseToolPanel footer. The `hexLabel` layer is still conditionally added to the config object only when the checkbox is checked, so label rendering tracks the checkbox state correctly.

The PR also restores `map.json` from the most complete backup — `map-2026-05-22T05:17:11` with 2261 hexes, complete terrain, road, stream, wall, and contour layers — plus incorporates live session edits made during the digitization session (road→trail corrections and new contour edges). The `countersManifest.js` was regenerated to reflect the current counter image set. A `movement.test.js` test was updated because hex 20.23, previously 'unknown' (treated as clear), is now correctly classified as 'woods' in the restored map.

## 13:38 — PR: Pre-M6 Debt Sprint — housekeeping, AT fix, store guard, CSS scoping

This PR closes six open debt items before M6 combat complexity lands. Three of the items (#431, #432, #402) were already fixed in the codebase during M5 game-map-ui work but had never had their GitHub issues closed or their debt register entries removed — the sprint first audited these and closed them as housekeeping. The remaining three (#434, #435, #441) required targeted code changes.

The most design-significant fix is the SVG button pattern change in `UnitCounterLayer.vue` (#434). The original code placed `role="button"`, `tabindex`, `aria-label`, and click/keydown handlers directly on an SVG `<image>` element. While JSDOM-based tests accepted this, `<image role="button">` is notoriously unreliable across real AT tools — NVDA, JAWS, and VoiceOver all handle it inconsistently. The fix wraps each `<image>` in a `<g role="button">` parent and moves all interactive attributes to the `<g>`. SVG `<g>` elements have reliable ARIA role support across AT implementations. The `<image>` itself becomes a presentation-only child with no ARIA attributes, which is the correct semantic split. The `:focus-visible` CSS outline stays on `.unit-counter` and applies naturally to the `<g>`.

The `loadGame` double-call guard (#441) adds a generation counter (`_loadGeneration`) inside the store definition. Each `loadGame` call increments the counter and captures its value before the first `await`. Any state write that follows an `await` checks whether the captured generation still matches the current counter; if not, a newer call has started and the write is silently dropped. This is a standard "cancel-on-supersede" pattern. It was implemented as a plain integer rather than `AbortController` because the current fetch targets are static routes with no server-side cancellation benefit, and the integer approach is simpler to test deterministically. The generation counter lives inside the store factory (not module scope) so it resets with each fresh Pinia instance — important for test isolation.

Removing `overflow:hidden` from `html` and `body` in `App.vue` (#435) required confirming that `GameView.vue`'s own layout already provides the necessary scroll-lock. The game map viewport is locked by `overflow: hidden` on `.game-body` (the flex child that contains the map area), not by the document root. Scoping overflow to the component level is the correct pattern for a multi-route SPA — document-level overflow clipping was a shortcut that happened to work for M5's single-route testing but broke lobby scroll on shorter viewports.
