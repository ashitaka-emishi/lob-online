# 2026-04-15

## 07:44 — PR: Dev Pipeline Cleanup — team-review wiring and net debt tracking

This PR closes out a small but overdue chore: tightening the dev pipeline around
code review and technical debt tracking. Three things were shipped. First, a legacy
skills audit confirmed that `pr-create` and `pr-merge` are still load-bearing — they
own the devlog entry, local CI gate, and human control point respectively, neither of
which has a conductor equivalent. Both are retained.

Second, `/tech-debt-report` is now automatically invoked at the end of every
`/team-review` session. The mechanism is a project-level `.claude/commands/team-review.md`
override that takes precedence over the plugin cache version. This is the right boundary:
the plugin file lives outside the repo and would be wiped on a plugin update, whereas the
project-level override is checked in and versioned. The override copies the plugin's five
phases verbatim and adds a Phase 6 that calls `/tech-debt-report`, passing the PR number
through when the review target is a PR. `CLAUDE.md` and `.claude/rules/sdlc.md` were
updated to remove the now-redundant manual invocation instruction.

Third, `/tech-debt-report` gained net delta tracking. "Net delta" is defined as debt
added this PR minus debt closed this PR, where "closed" means open-register items whose
linked issues are referenced as resolved in the PR. The per-PR report template now
surfaces three lines — debt added, debt closed, net delta — and the Step 7 summary
echoes the same. The project-wide `docs/tech-debt/report.md` Debt Over Time table gained
a `Net Delta` column. All 112 historical PR rows were back-filled from the existing
`Debt Added` data, which already contained both new-debt rows and resolution sub-rows;
the net for each PR is the sum of all its rows. The two most recent per-PR reports
(#305, #312) were updated inline; older files are noted as pre-net-tracking.

One non-obvious design decision: the Net Delta column is populated only on the "main"
PR row, not on resolution sub-rows (which show "—"). This keeps the table scannable —
one net number per PR at a glance — without restructuring the existing multi-row format
that records individual issue closures. The `Cumulative Added (gross)` column is
unchanged; it remains a historical gross total that never decreases, intentionally
distinct from the Executive Summary net open score.

## 09:37 — PR: M3 Debt Closeout Sprint — 9 items closed, register down to score 28

This PR closes the M3 debt closeout track, targeting nine open debt items selected for
bang-for-buck value: three score-3 OOB items that turned out to already be closed, four
score-2 engine correctness items, and two score-1 cleanup items. The net result is a
register drop from 29 items (score 47) to 20 items (score 28) — a -19 net delta with
zero new debt introduced.

Phase 1 was free. Issues #237, #245, and #247 were on the target list as score-3 OOB
items, but a quick check confirmed they were already closed on GitHub on 2026-04-09 as
part of the M2 sprint, with all three fixes verifiable in the codebase
(`processUSACavDiv`, `editorRouteFactory.js`, `useOobPersistence.test.js`). No
implementation work was needed; Phase 1 simply confirmed closure and moved on.

Phase 2 covered four engine correctness fixes, all done TDD. `parseHexId` (#285) got a
guard against null, empty string, and malformed inputs — the original code silently
propagated NaN into cube math. `combatResult` (#287) got a `RangeError` for dice rolls
outside 2–12, matching the 2d6 constraint in LOB_CHARTS §5.6. The `noEffectTerrain` Set
hoist (#296) pre-builds the Set once in `movementPath` and `movementRange` before
entering the Dijkstra hot loop, rather than constructing a new Set on every
hex-transition call; `hexEntryCostBreakdown` accepts the pre-built Set as an optional
parameter and falls back to inline construction for single-shot callers. The
`computeLOS` parameter reorder (#290) swapped `(mapData, scenario)` to `(scenario,
mapData)`, matching the canonical order established by `movementPath` and
`movementRange`; 26 call sites in `los.test.js` plus the `mapTest.js` route were updated.

Phase 3 handled the cleanup items. The `/closing-roll` handler's `mods` parameter was
renamed to `modifiers` (#308) to match the `/morale` handler, with the client
`ClosingRollPanel.vue` updated to send the new key. The succession `confirmPush` test
(#260) gained the missing `localStorage.removeItem('lob-succession-editor-v1')`
assertion — the call already existed in `useOobPersistence.js`, the test just hadn't
checked it. The tech debt register was updated and all nine GitHub issues were closed.

One deliberate deferral: `hexEntryCost` (the public one-shot wrapper) was left building
the Set inline rather than accepting a pre-built parameter. Changing its signature would
have been a broader API break for no real-world gain — the performance benefit is in the
hot Dijkstra loop, which now uses `hexEntryCostBreakdown` directly with the hoisted Set.
Single-shot callers like the path cost breakdown tool aren't in any loop, so the overhead
is negligible.
