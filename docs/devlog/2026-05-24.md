# 2026-05-24

## 08:23 — PR: Pre-M6 Debt Sprint — Issues #461–#464

This PR closes four debt items in the MapEditorView / useMapPersistence / useEdgePanelWiring
cluster that were filed during the post-PR-451 sweep. The work falls into three distinct
categories: a structural refactor of the edge-painting mutation path, a documentation and
testing commitment on the persistence migration policy, and an explicit invariant declaration
for the overlay-config composable coupling.

The most substantive change is the extraction of `mutateEdgeFeatures(hexId, faceIndex, mutateFn)`.
`handleEdgePaint` and `handleContourPaint` previously shared eight identical lines of boilerplate
— guard `mapData`, resolve `canonicalOwner`, look up the hex index, auto-create a stub with
`terrain:'unknown'` if the hex wasn't yet in the map, initialize `hex.edges` and
`hex.edges[ownerFace]`, then diverge. The extracted helper owns all of that shared setup and
calls a type-specific `mutateFn` for the divergent tail. Both handlers are now a single
`return mutateEdgeFeatures(...)` call, and three new integration tests pin the shared hex-setup
contract so any future regression surfaces immediately. The auto-create stub line is marked
`// MIGRATION-ONLY` to keep the grep audit for `terrain.*unknown` unambiguous.

Alongside the extraction, `EDGE_DISPATCH` was normalized (#462). Previously road and stream
entries had no `paintFn` (the handler had a `?? handleEdgePaint` fallback) and used disparate
`clearSingle: boolean` / `clearTypes: string[]` fields for right-click behaviour. Every entry
now carries an explicit `paintFn` and a unified `clear: { mode, types? }` shape. `onEdgeClick`
and `onEdgeRightClick` read `entry.paintFn(...)` and `entry.clear.mode` directly — no
implicit defaults, no shape ambiguity. Adding a fourth edge-tool panel in the future requires
only a single, fully explicit dispatch entry.

The persistence migration policy work (#463) replaced the existing inline comment above
`migrateUnknownTerrain` with a JSDoc block that names every authoritative producer of
`terrain:'unknown'`: the `mutateEdgeFeatures` auto-create path, the `resolveHexOrStub` /
`useEdgeToggle` legacy path, and legacy map data. A two-cycle idempotency test was added to
`useMapPersistence.test.js` — load with `terrain:'unknown'`, serialize the migrated result, load
again, assert no terrain changes on the second pass. This closes the correctness gap where the
migration contract existed only as a comment and was not verified.

The composable coupling work (#464) updated the `activePanelOverlayConfig` parameter JSDoc in
`useEdgePanelWiring.js` to name the invariant explicitly ("caller must guarantee only one panel
instance is active at a time"), describe the enforcement mechanism (the `watch(openPanel)` reset
in MapEditorView), and add an `// INVARIANT` sentinel on the write line. A new integration test
in `MapEditorView.test.js` verifies the invariant end-to-end: open Road Tool and emit a
distinctive overlay-config from it, switch to Stream Tool, emit a different overlay-config from
it, assert `HexMapOverlay`'s `overlayConfig` prop reflects only the Stream config — confirming
no stale bleed from the previous panel.

## 15:18 — PR: Map Editor Bug Fixes — #416, #418, #419

Three map editor bugs that accumulated during the M5 game-map work: slope/elevation line
visibility, a data integrity issue around playable/non-playable hex boundaries, and the
longstanding `unknown` terrain default.

The contour visibility fix (#416) was a one-line config change in `CONTOUR_GROUPS` in
`feature-types.js`, which is the single source of truth for both the overlay renderer and
the tool panel swatches. The user specified exact values: extremeSlope gets thick black
(strokeWidth 7, `#000000`), slope gets thick medium grey (strokeWidth 5, `#666666`), and
elevation contour lines get a medium-thick 35%-gray (strokeWidth 3, `#595959`). The previous
values were noticeably faint, especially the elevation lines at `#aaaaaa` / sw 2.

The unknown terrain elimination (#419) had more surface area than it first appeared. The root
producer was `resolveHexOrStub` in `hexGeometry.js`, which created stubs with `terrain: 'unknown'`
for hexes not yet in the map index. Changing that to `'clear'` was the upstream fix; the
downstream cleanup touched `HexMapOverlay.vue` (fallback `??` expression), `TerrainToolPanel.vue`
(removed from both icon and label maps), and `TERRAIN_COLORS` in `feature-types.js`.
`migrateUnknownTerrain` in `useMapPersistence.js` is intentionally left in place — it's now
a migration-only safety net that normalises any `unknown` values in old persisted map files.
Six test files needed assertions updated to expect `'clear'` instead of `'unknown'`.

The playable-boundary edge guard (#418) replaced the original bug report (which described a
rendering offset on bottom/left boundary edges) with a stronger rule: entirely disallow setting
edges where either adjacent hex has `playable: false`, and strip any such edges on save. The
implementation adds a pure `stripNonPlayableBoundaryEdges(hexes, gridSpec)` function to
`edge-model.js` — it builds a hex map, iterates canonical faces 0–2, and deletes any face
whose playability check fails. The interactive guard lives in `MapEditorView.vue` as
`isNonPlayableBoundary(hexId, dir)`, wired as an early return at the top of `onEdgeClick`
so it covers both the EDGE_DISPATCH path and the legacy `useEdgeToggle` fallback. A
`handleSave()` wrapper calls strip then save, replacing the direct `@click="save"` binding.
One tricky moment during testing: the test for the adjacent-hex strip case used `05.04` as the
N neighbor of `05.05`, but the hex ID format is `col.row` and N increments row — the actual
N neighbor is `05.06`. Fixed after the first test run.
