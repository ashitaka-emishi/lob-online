# Technical Debt Tracking

This directory contains the technical debt register for lob-online.

## Directory Structure

```
docs/tech-debt/
├── README.md          — this file; conventions and scale reference
├── report.md          — aggregated project-wide debt report (updated after every PR)
└── reports/           — per-PR debt reports
    └── pr-{number}_{YYYY-MM-DD}.md
```

## Workflow

After completing PR review resolution (following `/team-review`), run `/tech-debt-report` to:

1. Classify each deferred finding (linked issue + debt score + assessment).
2. Generate a per-PR report in `docs/tech-debt/reports/`.
3. Update the aggregated `docs/tech-debt/report.md`.

All review findings must be either **fixed in place** (resolved in the PR) or **deferred** (filed as a GitHub issue). Deferred items require a debt score and written assessment before the workflow completes.

When a deferred item's linked issue is later closed, follow the **Resolving Debt Items** process below to remove it from the register.

## Filename Convention

Per-PR reports use the following naming pattern:

```
pr-{number}_{YYYY-MM-DD}.md
```

Examples:

- `pr-95_2026-03-18.md` — report for PR #95, produced on 2026-03-18
- `pr-102_2026-04-01.md` — report for PR #102, produced on 2026-04-01

Note: If `/tech-debt-report` is accidentally run twice for the same PR, the skill will detect the existing file and ask before overwriting.

## Debt Score Scale

Scores use an ordinal 1–5 scale where each step represents a qualitatively larger category of impact. The README is the canonical reference — the skill embeds an abbreviated version.

| Score | Label       | Description                                                                                                               |
| ----- | ----------- | ------------------------------------------------------------------------------------------------------------------------- |
| 1     | Trivial     | Style, naming, minor clarity issue. No functional risk.                                                                   |
| 2     | Minor       | Duplicated logic, missing comment, small inconsistency. Low coupling risk.                                                |
| 3     | Moderate    | Missing abstraction, workaround, sub-optimal pattern. Slows future work.                                                  |
| 4     | Significant | Architectural compromise, risky coupling, missing error boundary. Can cause cascading failures or block future phases.    |
| 5     | Critical    | Security risk, correctness hazard, or structural flaw that actively blocks future work or threatens production stability. |

When scoring, err toward the higher score if uncertain — it is easier to downgrade than to discover under-reported debt later.

### Scoring Examples

| Score | Example                                                                                                                                                             |
| ----- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 1     | A function is named `doThing` instead of `applyMovementOrder`                                                                                                       |
| 2     | The same hex-neighbor lookup is written inline in two separate components instead of calling `hexGeometry.getNeighbors`                                             |
| 3     | A Vue component handles both data fetching and display logic because extracting a composable was deferred; future changes require editing one file for two concerns |
| 4     | A route handler validates user input with `if/else` chains instead of Zod, making it easy to accidentally bypass validation when the schema evolves                 |
| 5     | JWT tokens are stored in `localStorage` instead of `httpOnly` cookies, exposing them to XSS attacks                                                                 |

## Per-PR Report Template

```markdown
# Technical Debt Report — PR #{number}

**PR:** #{number}
**Date:** {YYYY-MM-DD}
**Reviewer(s):** {names}

## Fixed Findings

{N} finding(s) fixed in place during this review cycle.

## Deferred Findings

| Issue    | Title   | Score | Assessment                                          |
| -------- | ------- | ----- | --------------------------------------------------- |
| #{issue} | {title} | {1–5} | {1–3 sentence description of the debt and its risk} |

**Total debt added this PR:** {sum of scores}

---

_Generated by `/tech-debt-report`_
```

## Aggregated Report Structure

`docs/tech-debt/report.md` contains:

1. **Executive Summary** — total open items, net cumulative debt score (sum of open items), highest-risk item.
2. **Debt Over Time** — one row per PR cycle; "Cumulative Added" is a gross historical total (only increases).
3. **Risk Assessment** — prose summary of the categories of risk represented by open debt.
4. **Open Debt Items** — table ordered by score descending (ties: newest first). Resolved items are removed.

Note: The Executive Summary "Cumulative debt score" (net open) and the Debt Over Time "Cumulative Added" (gross historical) are different metrics and will diverge once items are resolved. This is intentional.

## Resolving Debt Items

When a deferred debt item's linked GitHub issue is closed:

1. Delete its row from the **Open Debt Items** table in `report.md`.
2. Append a resolution row to **Debt Over Time**:
   ```
   | {YYYY-MM-DD} | PR #{closing_pr} (resolved #issue) | -{score} | {unchanged cumulative added} |
   ```
   The "Cumulative Added" column does not change on resolution.
3. Recalculate the **Executive Summary** (open count, net score, highest-risk item).
4. Update the **Risk Assessment** prose.
5. Update the **Last updated** line.
6. Commit: `git commit -m "docs(tech-debt): resolve debt item #issue (PR #{closing_pr})"`
