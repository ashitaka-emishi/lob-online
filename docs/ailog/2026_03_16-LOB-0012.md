## AI Plan

This issue adds a Cypress E2E suite to validate the real browser→server→filesystem path for the map editor's push/pull sync and offline fallback features — the one path unit tests with mocked `fs`/`fetch` cannot cover. The implementation has three parts: (1) infrastructure — install Cypress as a root devDependency, add `cypress.config.js` with `baseUrl` and a `listBackups` task, add a `test:e2e` npm script, and create a `workflow_dispatch`-only GitHub Actions job; (2) support files — `cypress/support/commands.js` defining `cy.seedMapDraft` and `cypress/support/e2e.js` importing it; (3) four test files — `sync.cy.js` (4 tests: push happy path, push overwrite confirmation, pull happy path, pull discard confirmation), `offline.cy.js` (2 tests: offline with draft, offline without draft), and `backups.cy.js` (1 stretch test: backup file exists after push). No source files are modified — this is purely test infrastructure. The E2E suite is not a required CI gate; it runs only on `workflow_dispatch`.

## Implementation Notes

- `cy.seedMapDraft` uses `cy.window()` which is called post-visit; for the offline-with-draft
  test the draft must be set before `onMounted` runs, so `cy.visit(url, { onBeforeLoad })` is
  used directly instead of the custom command.
- Pull discard confirm test sets `unsaved = true` via Vue 3 dev-mode
  `__vueParentComponent.setupState.unsaved.value` — only works against the Vite dev server.
- ESLint config extended with a Cypress-specific block (globals + `caughtErrorsIgnorePattern`)
  to cover `cypress/**/*.js` and `cypress.config.js`.
- `backups.cy.js` is a stretch test that requires the live dev server; all other tests use
  `cy.intercept` stubs so they run without a real server.
