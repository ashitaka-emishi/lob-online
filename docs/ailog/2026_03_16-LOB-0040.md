## Implementation Notes

Single artifact: `docs/map-editor-auto-detect-design.md` (652 lines). Key design decisions:

1. **One-time Vision palette extraction** — Claude Vision API analyzes the full map image once to derive the elevation color gradient (blueish-green → green → tan → brown); stored in `scenario.json` as `autoDetectConfig.elevationPalette`. This avoids per-hex API costs and gives a ground-truth palette.
2. **Local color analysis as primary path** — hexagonal mask + k-means clustering + palette nearest-neighbor match handles the bulk of hexes without API calls. Confidence scoring triggers Vision fallback only for ambiguous hexes (threshold: 0.60).
3. **Seed hexes as few-shot ground truth** — ~25 confirmed hexes spanning all terrain types × elevation bands serve as few-shot examples fed to the Vision API fallback. Persisted as `autoDetectConfig.seedHexes` with base64 crop images. Editing workflow requires ≥10 seed hexes before auto-detect can be enabled.
4. **`playable` flag on `HexEntry`** — new field for off-map edge hexes; hexside playability derived at runtime from adjacent hex states (no new field on `EdgeFeature`); vertex playability deferred to game engine phase (DL-07).
5. **Realistic v1 accuracy** — terrain 60–70%, elevation 70–80%, hex-centre features 40–60%; hexside detection deferred as not feasible with color analysis alone.
6. **7 open decisions** — hexside detection method (DL-01), Vision model selection (DL-02), seed hex minimum count (DL-03), non-playable marking UX (DL-04), batch crop size (DL-05), overwrite confirmation scope (DL-06), vertex playability (DL-07).
