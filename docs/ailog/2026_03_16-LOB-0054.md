## AI Plan

**Server:** `map-autodetect-config.schema.js` defines three Zod schemas (`ElevationBandSchema`, `SeedHexSchema`, `AutoDetectConfigSchema`) matching design doc §9. Initial data file `map-autodetect-config.json` contains `{ "seedHexes": [] }`. A new Express router `autoDetectConfig.js` provides GET/PUT `/data` endpoints with versioned backup logic identical to `mapEditor.js`; registered in `server.js` under the `MAP_EDITOR_ENABLED` guard at `/api/tools/map-autodetect-config`. Route tests follow the `mapEditor.test.js` mock-fs pattern.

**Client:** `HexEditPanel.vue` gains a `isSeedHex` boolean prop and a "Mark as Seed Hex" toggle button (enabled when terrain ≠ 'unknown' and elevation is set); clicking emits `seed-toggle` with the hex's `confirmedData`. `ElevationSeedEditor.vue` accepts a `config` prop (the full `AutoDetectConfig` object) and renders: a coverage grid showing which elevation bands have ≥1 seed hex, a list of current elevation seed hexes each with a delete button emitting `seed-remove`. `TerrainSeedEditor.vue` mirrors this for terrain types (open, woods, town, orchard, rough). Both editors update reactively via prop. Component tests use `@vue/test-utils` mount pattern. `cropBase64` is stored as an empty string placeholder — crop extraction is deferred to Issue C.
