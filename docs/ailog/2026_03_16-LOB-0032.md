readline gate handler, loop-back routing, and `WorkflowInstance` JSON persistence to
`docs/ailog/`). Companion test files (`schemas.test.js`, `runtime.test.js`, `registry.test.js`)
target ≥ 70% line coverage. Three workflow definitions under `docs/workflows/` formalise the
`sdlc-feature`, `issue-intake`, and `issue-implement` pipelines, each with a `.workflow.json`
and a `.states.md` containing prose, a Mermaid `stateDiagram-v2` diagram, and a gate checkpoint
table. `docs/architecture.md` and `CLAUDE.md` are updated to reference the new subsystem.

## Implementation Notes

- Zod validation is applied to the full `WorkflowDefinition` at `runWorkflow` entry; crucially
  `defResult.data` (with defaults filled in) is used for step execution so that `inputMap: {}`
  defaults are always present — raw input objects lack these and would crash `Object.entries`.
- JSONPath resolver is a bespoke subset (`$.stepId.field.subfield`); no external library needed
  for the current use cases. Full `jsonpath` library was evaluated and deferred (Out of Scope v1).
- `presentGate` accepts an injectable `io` readline interface for testability; all `runWorkflow`
  tests pass a `gateHandler` function so the readline path is only exercised by direct unit tests
  of `presentGate` itself (not added — readline testing would require mocking Node streams).
- Registry module has a module-level cache keyed to the default `REGISTRY_PATH`; a `_resetCache`
  export allows test isolation. Cache is bypassed for custom paths so tests can pass temp files.
- `issue-implement.workflow.json` has a `gate-merged` step after `pr-merge` to capture HCP 3;
  this means the workflow ends with a gate rather than an agent step — intentional to model the
  human-approved squash merge pattern accurately.
