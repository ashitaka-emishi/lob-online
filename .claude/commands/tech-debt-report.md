description: Score deferred PR review findings, generate a per-PR technical debt report, and update the project-wide debt register
allowed-tools: Read, Edit, Write, Glob, Bash
---

You are running the technical debt report workflow. Work through the steps below in order.

## Debt Score Scale (reference)

| Score | Label       | Description                                                                      |
| ----- | ----------- | -------------------------------------------------------------------------------- |
| 1     | Trivial     | Style, naming, minor clarity. No functional risk.                                |
| 2     | Minor       | Duplicated logic, small inconsistency. Low coupling risk.                        |
| 3     | Moderate    | Missing abstraction, workaround, sub-optimal pattern. Slows future work.         |
| 4     | Significant | Architectural compromise, risky coupling. Can cause cascading failures.          |
| 5     | Critical    | Security risk, correctness hazard, or blocks future work / production stability. |

When in doubt, score higher — it's easier to downgrade than discover under-reported debt later.

---

## Step 1 — Identify the PR

If a PR number was provided as an argument, use it. Otherwise ask:

```
What PR number does this debt report cover?
```

Get today's date by running `date +%Y-%m-%d`. The per-PR report filename will be:
`docs/tech-debt/reports/pr-{number}_{YYYY-MM-DD}.md`

Also ask (or confirm from context):

```
Who performed the review? (reviewer name(s))
```

---

## Step 2 — Inventory the Review Findings

Ask the user to provide the list of all findings from the review (they may paste from the team-review output or summarize):

```
Please list the findings from this PR review. For each finding, indicate:
- Was it FIXED in place (resolved in this PR)?
- Or DEFERRED (filed as a GitHub issue)?

Format:
1. [FIXED] Brief description
2. [DEFERRED] Brief description — Issue #NNN (or "no issue yet")
```

Count the fixed findings. For any deferred item without a linked issue, require one before proceeding:

```
Deferred item "{title}" requires a linked GitHub issue before this workflow can complete.
Please file the issue and provide the issue number.
```

---

## Step 3 — Score Each Deferred Finding

For each deferred finding, collect three pieces of information one finding at a time:

```
Deferred finding: "{title}" (Issue #{number})

1. Debt score (1–5):
   1 = Trivial  |  2 = Minor  |  3 = Moderate  |  4 = Significant  |  5 = Critical

2. Assessment (1–3 sentences describing the debt and its risk to the codebase):
```

Collect all responses before proceeding to report generation.

---

## Step 4 — Generate the Per-PR Report

Create the file `docs/tech-debt/reports/pr-{number}_{YYYY-MM-DD}.md` with the following content:

```markdown
# Technical Debt Report — PR #{number}

**PR:** #{number}
**Date:** {YYYY-MM-DD}
**Reviewer(s):** {names}

## Fixed Findings

{N} finding(s) fixed in place during this review cycle.

## Deferred Findings

| Issue    | Title   | Score   | Assessment   |
| -------- | ------- | ------- | ------------ |
| #{issue} | {title} | {score} | {assessment} |

**Total debt added this PR:** {sum of all deferred scores}

---

_Generated by `/tech-debt-report`_
```

If there are no deferred findings, write "None — all findings fixed in place." in the Deferred Findings section and set total debt to 0.

---

## Step 5 — Update the Aggregated Report

Read `docs/tech-debt/report.md`. Update it in place with the following changes:

### Open Debt Items table

Append one row per deferred finding:

```
| {score} | #{issue} | {title} | PR #{pr_number} | {assessment} |
```

Then re-sort all rows by score descending. For ties, put the newer PR first (higher PR number first).

Remove the placeholder `| — | — | — | — | — |` row if it is still present.

### Debt Over Time table

Append one row:

```
| {YYYY-MM-DD} | PR #{number} | {debt added this PR} | {new cumulative score} |
```

Calculate the new cumulative score by summing the "Debt Added" column across all rows (including the new one). Remove the placeholder `| — | — | — | 0 |` row if still present.

### Executive Summary table

Recalculate and update:

- **Open debt items** — count rows in the Open Debt Items table (excluding header and placeholder).
- **Cumulative debt score** — sum of all scores in the Open Debt Items table.
- **Highest-risk item** — title of the row with the highest score (first if tied).
- **PRs tracked** — count rows in the Debt Over Time table (excluding header and placeholder).

### Risk Assessment section

Replace the current Risk Assessment prose with an updated paragraph. Base it on the current open items:

- If cumulative score is 0: "No debt items recorded. Risk level: **None**."
- If cumulative score 1–5: "Low risk. Minor deferred items with no functional impact."
- If cumulative score 6–15: "Moderate risk. Some deferred workarounds and sub-optimal patterns that will slow future phases if not addressed."
- If cumulative score 16–30: "Elevated risk. Several significant deferred items that introduce coupling or architectural compromise. Recommend a debt reduction sprint before the next major phase."
- If cumulative score > 30: "High risk. Critical or significant deferred items pose a threat to production stability or block future work. Immediate attention recommended."

Add a brief prose summary of the categories of risk present (e.g., "Current debt is concentrated in missing abstractions and test coverage gaps.") based on the assessment text of the open items.

### Last updated line

Update the `_Last updated:_` line at the top of the file to today's date and the PR number:

```
_Last updated: {YYYY-MM-DD} after PR #{number}._
```

## Step 6 — Commit

Stage and commit both files:

```bash
git add docs/tech-debt/reports/pr-{number}_{YYYY-MM-DD}.md docs/tech-debt/report.md
git commit -m "docs(tech-debt): add debt report for PR #{number} ({YYYY-MM-DD})"
```
