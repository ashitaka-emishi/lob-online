# /ecosystem-docs-generate

Regenerate all ten files in `docs/claude-ecosystem/` and `docs/designs/ecosystem-design.md` from the
ecosystem's source-of-truth inputs.

**Six reference files** are fully rebuilt from sources on every run:
`README.md`, `agents.md`, `skills.md`, `network-diagram.md`, `guardrails-and-logging.md`,
`orchestration.md`

**Four narrative files** are updated in place — preserve voice and prose; only correct
factual content (agent counts, file paths, skill names, step sequences) that has drifted:
`blog.md`, `tutorial-new-agent.md`, `tutorial-modify-agent.md`, `tutorial-orchestration.md`

The three-layer relationship must appear in every reference file:
**agents** (scope — what a subprocess is allowed to do) →
**skills** (procedure — steps in what order) →
**orchestration** (sequencing — when and in what order, with gate checkpoints).

Every reference file begins with this header immediately after the `# Title` line:

```
> Auto-generated by /ecosystem-docs-generate — do not edit by hand.
> Source of truth: docs/agents/*/design.md, .claude/agents/registry.json,
> .claude/commands/*.md, docs/workflows/*/
```

---

## Step 1 — Read all source-of-truth inputs

Read these files in full before generating any output:

**Agent design docs** — for each directory under `docs/agents/`, read `design.md`:

```bash
ls docs/agents/
```

**Registry** — canonical index of every agent and skill:

```
.claude/agents/registry.json
```

**Skill command files** — read every file:

```bash
ls .claude/commands/
```

**Workflow definitions** — for each directory under `docs/workflows/`, read both files:

```bash
ls docs/workflows/
```

- `docs/workflows/<name>/<name>.workflow.json`
- `docs/workflows/<name>/<name>.states.md`

**Existing docs** (for diffing and narrative updates) — read all ten files in
`docs/claude-ecosystem/` and `docs/designs/ecosystem-design.md`.

---

## Step 2 — Build the six reference documents

Derive each document fresh from the source-of-truth inputs. Do not copy the existing file.

### 2a — `README.md`

The hub index and entry point. Must contain:

1. **Project intro paragraph** — what the ecosystem is and why it exists (one paragraph,
   existing tone).
2. **SDLC overview** — two paragraphs explaining the full lifecycle from raw idea to merged
   PR: intake (idea → filed issue), implementation (plan → code → build/test → push),
   review and merge (PR → review → squash merge). Name the agents and skills that handle
   each phase. This replaces the need for a separate overview document.
3. **Contents table** — one row per file in `docs/claude-ecosystem/`, all ten files.
   Update "What it covers" to match current content.
4. **Quick-Start section** — `/issue-intake`, `/issue-implement`, `/dev-build`, `/dev-test`,
   `/ecosystem-docs-generate`. Claude Code invocation + brief manual equivalent for each.

### 2b — `agents.md`

Agent reference only — no orchestration content (that lives in `orchestration.md`). Must
contain:

1. **Intro paragraph** — agent count (from registry `type: "agent"` entries), what agents
   are, where they are defined, how `design.md §4` is the source of truth. One-line pointer:
   `> For the workflow engine and workflow definitions, see [orchestration.md](orchestration.md).`
2. **One section per agent** (registry `type: "agent"` entries). Each section:
   - **Purpose** — from `design.md §1` or `§4` description
   - **Allowed tools** — from `design.md §4` frontmatter
   - **Collaborators** — agents it delegates to or consults
   - **Owned skills** — table with one-line description per skill
   - **Design spec** — link to `docs/agents/<name>/design.md`

### 2c — `skills.md`

Skill reference. Must contain:

1. **Intro paragraph** — what skills are, where they live, composability principle.
2. **One section per skill**, grouped by owning agent. Derive from registry `type: "skill"`
   entries and command files. Each section:
   - **Purpose** — one sentence from the skill's `description` frontmatter or first paragraph
   - **Owning agent** — from registry or design docs; "unowned" if none
   - **Related skills** — any skills it explicitly calls
   - **Claude Code** — `/skill-name [args]`
   - **Manual equivalent** — bash steps a developer would run without Claude Code
3. If a skill exists in `.claude/commands/` but not in `registry.json`, list it as
   "unregistered" rather than omitting it.

### 2d — `network-diagram.md`

Two static structural diagrams only. Must contain:

1. **Agent and skill ownership** (`graph TD`) — one node per agent; edges to each owned
   skill; unowned skills in a standalone subgraph. Derive from registry.
2. **Skill dependency graph** (`graph LR`) — directed edges showing which skills call other
   skills. Derive by scanning each `.claude/commands/*.md` for `/skill-name` invocations.

### 2e — `guardrails-and-logging.md`

Must contain:

1. **Human Control Points** — derive each HCP from the `gateConfig` entries in every
   `.workflow.json`. Table: HCP name, workflow, gate prompt summary, choices, default.
2. **CI gates** — the three checks, where enforced (local `/dev-build`, remote CI).
3. **Rules-lawyer gate** — trigger conditions, how rulings are recorded in AC.
4. **AI log (ailog)** — naming convention, sections written at each checkpoint, commit
   convention.
5. **Devlog** — naming convention, section format, index file.
6. **Summary table** — mechanism, scope, who it protects.

### 2f — `orchestration.md`

New dedicated workflow engine reference. Must contain:

1. **Overview** — what the orchestration runtime is (`server/src/orchestrator/`), why it
   exists, and how it relates to agents and skills (the third layer).
2. **`WorkflowDefinition` schema** — explain each field: `id`, `name`, `description`,
   `steps[]`. For each step, explain: `id`, `name`, `agentId` (null for gates), `inputMap`,
   `onError`. For gate steps, explain `gateConfig`: `prompt`, `choices[]` (label, value,
   onChoice.nextStep), `defaultChoice`.
3. **Runtime capabilities** — three subsections:
   - **JSONPath `inputMap`** — `$.stepId.field` syntax, how prior step outputs are passed as
     inputs
   - **Loop-back routing** — how gate choices can name an earlier step as `nextStep` to
     re-run part of the workflow
   - **`WorkflowInstance` persistence** — how execution state is written to `docs/ailog/`
     as JSON alongside the human-readable ailog Markdown files
4. **Workflow definitions** — one subsection per workflow in `docs/workflows/`. Derive step
   count, gate count, and purpose from the `.workflow.json`. Include the Mermaid state
   diagram from the corresponding `.states.md`. Also include the gate checkpoint table.
5. **SDLC sequence diagram** — the full `issue-implement` flow as a `sequenceDiagram`
   showing Engineer, agents (PM, DevOps, CodeReview), and skills annotated at each step.
   Derive gate positions from `issue-implement.workflow.json`. (Moved from
   `network-diagram.md`.)
6. **Intake flowchart** — the `issue-intake` flow as a `flowchart TD` from branch creation
   through merge. Derive from `issue-intake.workflow.json`. (Moved from `network-diagram.md`.)
7. **How to add a new workflow** — concise step list: create directory, write
   `.workflow.json`, write `.states.md`, add registry entries, validate with Zod schemas,
   run `/dev-build`.

### 2g — `docs/designs/ecosystem-design.md`

The top-level architecture and design reference for the entire ecosystem (renamed from
`docs/architecture.md`). Must contain:

1. **Overview paragraph** — what agents, skills, and orchestration are, and how they compose.
2. **Agent ownership diagram** (`graph TD` Mermaid) — identical in structure to
   `network-diagram.md §1`; derive from registry. This file is the standalone reference for
   developers who do not navigate to `docs/claude-ecosystem/`.
3. **Skill dependency graph** (`graph LR` Mermaid) — identical structure to
   `network-diagram.md §2`; derive from skill command files.
4. **SDLC sequence diagram** — full `issue-implement` flow; derive from
   `issue-implement.workflow.json`. Identical in structure to `orchestration.md §SDLC
sequence`.
5. **Key file index** — table of the most important files a developer needs to navigate the
   ecosystem: agent definitions, registry, workflow JSON files, skill command files,
   `docs/claude-ecosystem/` hub, this file.

Add the auto-generated header immediately after the `# Title` line.

## Step 3 — Update the four narrative files

For each narrative file, read the current content and the source-of-truth inputs. Identify
only the factual content that has drifted: agent names, skill names, file paths, step
counts, workflow names, HCP numbers. Make targeted edits to correct those facts. Preserve
all prose, voice, structure, and examples.

**Do not rewrite paragraphs wholesale.** If a skill was renamed, update the name. If a new
agent was added, add it to any agent list. If a workflow step count changed, update the
number. Nothing more.

### 3a — `blog.md`

Check for: correct agent count and names, correct skill names and invocations, accurate
description of the three-layer model, accurate HCP count.

### 3b — `tutorial-new-agent.md`

Check for: correct file paths, correct slash-command names, accurate step count, valid
worked example (does the example agent still exist in registry?).

### 3c — `tutorial-modify-agent.md`

Check for: correct reference to `design.md §4` as source of truth, correct skill names
(`/agent-regenerate`, `/agent-sync`), valid worked example.

### 3d — `tutorial-orchestration.md`

Check for: correct workflow JSON field names (match current Zod schema in
`server/src/orchestrator/schemas.js`), correct file paths for workflow definitions,
correct registry format, accurate step count in how-to procedure.

## Step 4 — Diff each file before writing

For each of the ten target files, compare the newly generated (or updated) content against
the existing file. Report a structured delta summary:

```
### README.md
  Lines added:   N
  Lines removed: N
  Key changes:   [brief description]

### orchestration.md  (NEW)
  Key changes:   new file

### blog.md
  Key changes:   [factual corrections only, e.g. "updated agent count from 4 to 5"]
```

If a file would be unchanged, report `No changes` and skip writing it.

Ask: **"Write all changed files? (yes / show diff for \<file\> / skip)"**

- **yes** — write all changed files immediately
- **show diff for \<file\>** — display the full unified diff, then re-ask
- **skip** — stop without writing

## Step 5 — Write the files

For each changed file approved by the engineer:

- Reference files (`README.md`, `agents.md`, `skills.md`, `network-diagram.md`,
  `guardrails-and-logging.md`, `orchestration.md`): write the full new content with the
  `> Auto-generated by...` header block.
- Narrative files (`blog.md`, tutorial files): apply only the targeted edits identified in
  Step 3; do not add the auto-generated header.

After writing, run Prettier on all modified files:

```bash
npx prettier --write \
  docs/claude-ecosystem/README.md \
  docs/claude-ecosystem/agents.md \
  docs/claude-ecosystem/skills.md \
  docs/claude-ecosystem/network-diagram.md \
  docs/claude-ecosystem/guardrails-and-logging.md \
  docs/claude-ecosystem/orchestration.md \
  docs/claude-ecosystem/blog.md \
  docs/claude-ecosystem/tutorial-new-agent.md \
  docs/claude-ecosystem/tutorial-modify-agent.md \
  docs/claude-ecosystem/tutorial-orchestration.md \
  docs/designs/ecosystem-design.md
```

Report which files were written and how many lines changed in each.

## Constraints

- **Never modify:** any `.js`, `.vue`, `.json` data file, or `.claude/agents/*.md` agent
  definitions
- **Reference files:** build entirely from source-of-truth inputs; do not preserve stale
  content from the existing file
- **Narrative files:** targeted factual corrections only; preserve all prose and voice
- **Registry-driven skill list:** derive skill count and names from `registry.json`; note
  unregistered skills rather than silently omitting them
- **Agent count must be accurate:** derive from registry `type: "agent"` entries
- **Diagram fidelity:** every Mermaid node and edge must be derivable from a source file;
  do not invent relationships
